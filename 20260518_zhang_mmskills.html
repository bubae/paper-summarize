TL;DR

  • 문제: 기존 agent skill library는 ##재사용 가능한 지식을 텍스트(prompt, code, routine)로만## 표현한다. 그러나 visual agent는 desktop GUI/게임 같은 ##시각 환경##에서 동작하며, "언제 그 절차를 써야 하는가"는 ##관찰 가능한 시각 단서##에 의존한다. 텍스트만으로는 "다이얼로그가 준비되지 않았다", "차트가 잘못된 시트에 만들어졌다" 같은 결정이 불가능.
  • 해결: MMSkill = ##(D, P, S, K)## — descriptor + textual procedure + runtime state cards + multi-view keyframes. State card는 when-to-use/when-not-to-use/visible-cues/verification-cue/available-views를 ##agent-facing schema##로 명시. Keyframe은 full_frame, focus_crop, before, after의 multi-view bundle.
  • 왜 작동하는가: (1) public non-evaluation trajectory에서 ##Generator##가 5-phase pipeline으로 skill 추출, (2) 추론 시 ##branch loading##이 보조 trajectory에서 state card와 view를 align 후 ##distilled guidance##만 main agent에 전달, (3) reference image가 live observation을 over-anchor하는 것을 방지.
  • 비유: 텍스트 매뉴얼만 든 신입사원과, ##사진 첨부 매뉴얼 + 체크포인트 + 비포어/애프터 예시##를 든 신입사원의 차이. 후자는 "지금 화면이 이 단계 맞나?"를 ##스스로 시각적으로 확인##한 뒤 행동한다.

핵심 기여 (Key Contributions)

  1. Multimodal Procedural Knowledge 정식화: 재사용 가능한 시각 agent skill을 처음으로 ##state-conditioned multimodal package##로 정의. text 절차 + visual evidence + 사용 조건을 하나의 reusable unit으로 묶음.
  2. Agentic Trajectory-to-Skill Generator: ##5-phase pipeline## — (Phase 0) task embedding & clustering, (1) cluster-level skill planning, (2) skill merging, (3) text-first drafting, (4) image grounding & audit. Meta-skill-guided LLM agent가 control. ##Evaluation task와 분리된 public trajectory##만 사용.
  3. Branch-Loaded Multimodal Skill Agent: 메인 trajectory에 전체 skill package를 주입하지 않고, ##temporary branch##에서 state card/view selection을 수행해 structured guidance tuple $G_t$만 반환. Stage 1 (gated view selection) + Stage 2 (branch planning)의 2단계.
  4. 4개 벤치마크에서 검증: OSWorld (desktop GUI), macOSWorld, VAB-Minecraft (open-world game), Super Mario (LMGame-Bench). ##frontier model과 smaller model 모두에서 일관된 향상##. 특히 약한 모델 (Qwen3-VL-8B-Instruct)에서 OSWorld 10.78% → 25.40%, VAB-Minecraft 23.28% → 38.79%.
  5. Behavioral Shift 정량화: skill 사용 시 agent의 click 비율 75.8% → 63.7% (Qwen3-VL-235B), ##exact repeated actions 21.8% → 6.2%##. ##exploratory trial-and-error를 grounded, state-aware execution으로 전환##함을 입증.

배경 (Background)

Skill 추상화의 발전

Agent를 위한 skill 개념은 robotics의 motor primitive (Ijspeert et al., 2013)에서 출발해 LLM agent로 옮겨오면서:

  • Voyager, AutoGen 계열: skill = ##실행 가능 코드##.
  • ReAct, Plan-and-Solve 계열: skill = ##자연어 절차 (planning template)##.
  • AgentS, Mind2Web 계열: skill = ##학습된 routine library##.

이들 모두 ##"관련 state는 텍스트로 충분히 추상화 가능"##이라는 암묵적 가정. 그러나 시각 agent는 다음 같은 실패 모드를 보인다:

  • Desktop agent가 ##다이얼로그가 아직 안 떴는데 다음 단계로 진행##.
  • Game agent가 ##목표는 알지만 progress vs completion 구분 실패##.
  • GUI agent가 ##wrong sheet에 chart 생성##.

핵심 통찰: Visual Procedural Knowledge의 필요성

Mayer (2009)의 multimedia learning 연구에 따르면 ##시각 정보는 절차 학습에서 verbal 설명을 보완##한다. 텍스트 skill은 너무 verbose하면서도 underspecified, demonstration은 visual context를 보존하지만 lengthy하고 instance-specific이며 adapt하기 어렵다. MMSkills는 이 양극단 사이의 ##중간 추상화##를 목표.

Reusable Multimodal Knowledge가 만족해야 할 3가지 challenge

  • Representation: skill package가 어떤 내용을 담아야 하고, 절차/시각/검증 단서를 어떻게 binding할 것인가?
  • Generation: hand-written이나 raw demonstration replay가 아닌 ##public non-evaluation interaction##에서 어떻게 자동 derive할 것인가?
  • Utilization: 추론 시 ##과도한 image context, distracting state description, reference screenshot over-anchoring## 없이 어떻게 consult할 것인가?

방법 (Method)

전체 시스템 개요

[Skill Library M] [Generator F] [Branch-Loaded Agent] Public trajectories → 5-phase pipeline → Multimodal Skill Library ↓ Pre-recall: I → candidate set C_I ⊂ M ↓ At step t: Observe O_t, history H_t Direct: A_t = π_main(O_t, H_t, C_I) Branch: G_t = Branch(O_t, H_t, M_t) A_t = π_main(O_t, H_t, C_I, G_t)

MMSkill 데이터 구조 (Equations 3–6)

각 MMSkill은 state-conditioned procedure package:

$$M = (D, P, S, K) \quad (3)$$

여기서:

  • $D$: compact descriptor (when_to_use 요약).
  • $P$: reusable textual procedure.
  • $S = \{S_j\}_{j=1}^{m}$: runtime state cards.
  • $K = \{K_j\}_{j=1}^{m}$: $S_j$와 aligned된 keyframe bundle.

Runtime State Card $S_j$ (Equation 4)

$$S_j = (\text{when\_to\_use}_j, \text{when\_not\_to\_use}_j, \text{visible\_cues}_j, \text{verification\_cue}_j, \mathcal{V}_j)$$

각 필드의 역할:

  • when_to_use: 이 단계 절차가 follow되어야 하는 조건.
  • when_not_to_use: skip되어야 하는 조건 (## 매우 중요한 negative constraint##).
  • visible_cues: 현재 state임을 판정할 시각 단서.
  • verification_cue: 진행/완료 확인 단서.
  • available_views: 어떤 view를 load할 수 있는가.

Multi-View Keyframes (Equations 5–6)

View 집합:

$$\mathcal{V} = \{\text{full\_frame}, \text{focus\_crop}, \text{before}, \text{after}\} \quad (5)$$

각 state의 keyframe bundle:

$$K_j = \{K_j^v : v \in \mathcal{V}_j, v \in \mathcal{V}\} \quad (6)$$

왜 4종류 view인가:

  • ##full_frame##: global context — 어떤 application인지, layout이 어떤지.
  • ##focus_crop##: visual cue localization — 작은 button, status indicator.
  • ##before/after##: transition evidence — "이 단계가 완료되면 이렇게 변한다"를 보여줌.

이 design은 reference image가 ##coordinate를 복사하기 위한 것이 아니라 평가 evidence##임을 명시. Text-only skill은 degenerate case $(D, P, \emptyset, \emptyset)$.

Skill Generator from Public Trajectories

Trajectory $\tau_i = (I_i, O_{i,1:T_i}, A_{i,1:T_i})$ (instruction, observations, actions). Generator는 multimodal-skill-factory meta-skill $\mathcal{F}$로 control:

$$\mathcal{G}_\mathcal{F}: \mathcal{T}_d \mapsto \mathcal{M}_d \quad (8)$$

$$\mathcal{T}_d \xrightarrow{\text{Phase 0: embed+cluster}} \mathcal{C}_d \xrightarrow{\text{Phase 1: cluster plan}} \mathcal{A}_d \xrightarrow{\text{Phase 2: merge}} \mathcal{R}_d \xrightarrow{\text{Phase 3: text draft}} \widehat{\mathcal{M}}_d \xrightarrow{\text{Phase 4: ground+audit}} \mathcal{M}_d \quad (9)$$

5-Phase Pipeline 상세

  1. Phase 0: Task Embedding & Clustering: instruction과 trajectory metadata를 embed → semantic cluster $\mathcal{C}_d$.
  2. Phase 1: Cluster-Level Skill Planning: 각 cluster마다 LLM-based agent가 atomic skill 후보 제안 (workflow boundary, completion condition, covered task ids).
  3. Phase 2: Skill Merging: cluster-level plan을 deduplicate, merge, generalize → $\mathcal{R}_d$. 너무 broad한 umbrella skill은 reject.
  4. Phase 3: Text-First Drafting: ##image 보지 않고## reference task 선택하여 $(D, P, \widehat{S}_r)$ 초안 작성. ##text rationale을 먼저 고정##해 이후 visual grounding이 그 rationale에 anchor되도록.
  5. Phase 4: Image Grounding & Audit: keyframe 읽어 focus region 결정, multi-view bundle 구성, ##final package audit##: $$\widehat{M}_r = (D_r, P_r, \widehat{S}_r, \widehat{K}_r) \xrightarrow{\text{ground+audit}} M_r = (D_r, P_r, S_r, K_r) \quad (10)$$

핵심 design 결정: visual grounding policy가 ##conservative##. View는 state 인식 / transition 비교 / completion 검증 같은 ##진단적 목적##에만 추가됨. demonstration replay는 저장하지 않음. 즉 ##skill = diagnostic state library##.

Branch-Loaded Multimodal Skills Agent

Direct loading의 문제: state card + multi-view keyframe + transition example을 main context에 넣으면 ##(i) substantial context pressure, (ii) reference image가 live observation과 경쟁, (iii) main agent가 visually anchored##되어 reference scenario를 따라하려 함.

Stage 1: Gated View Selection (Equation 11)

Main agent가 $M_t = (D_t, P_t, S_t, K_t) \in C_I$를 호출하면:

$$(J_t, R_t) = \text{SelectViews}(O_t, H_{t-1}, P_t, S_t), \quad V_t = \{K_j^v : j \in J_t, v \in R_{t,j}\} \quad (11)$$

$J_t$는 선택된 state card index, $R_{t,j} \subseteq \mathcal{V}_j$는 state $j$에 대해 load할 view 집합. ##Text와 state card만으로 충분하면## $R_{t,j}$가 empty.

Stage 2: Branch Planning (Equation 12)

$$G_t = \text{PlanBranch}(O_t, H_{t-1}, P_t, \{S_j : j \in J_t\}, V_t) \quad (12)$$

Branch는 selected evidence를 live state와 align하여 ##structured guidance tuple##:

$$G_t = (\text{applicable}_t, \text{subgoal}_t, \text{plan}_t, \text{do\_not\_do}_t, \text{verify}_t) \quad (2)$$

  • applicable: 이 skill이 현재 state에 적용 가능한가?
  • subgoal: local target.
  • plan: skill-conditioned action plan.
  • do_not_do: ##negative constraint## (가장 중요!).
  • verify: completion check.

핵심: $G_t$는 ##planning signal##이고, main agent는 이를 ##mechanically execute하지 않고 grounded action을 현재 screenshot에서 선택##. Reference image가 live observation을 override하지 못하게 보장.

비판적 분석 (Critical Analysis)

  • Skill generation cost가 측정되지 않음: 5-phase Generator는 LLM agent를 반복 호출하므로 ##비용이 trajectory 개수에 비례##. Per-skill generation cost (API tokens, compute hours)가 보고되지 않아 ##scale up 가능성을 정량 평가 어려움##.
  • Branch loading의 inference overhead: 매 step skill을 invoke할 때마다 ##추가 LLM/VLM call이 branch에서 발생##. Table 3에서 step 수는 감소했지만 ##wall-clock latency 또는 token cost##는 보고되지 않음. Production에서 step 감소가 token 증가를 상쇄할지 불명.
  • weak model에서의 Qwen-VL-8B 한계: OSWorld는 10.78 → 25.40 (+14.6)이지만 ##GIMP/Calc/Writer 같은 일부 도메인에서는 marginal##. 일부 도메인에서 text-only가 MMSkills보다 더 좋은 경우 (OS: 13.43 vs text 16.67) — ##weak VLM이 multimodal cue를 잘 활용하지 못하는 영역##도 존재.
  • State card의 generation quality 의존성: when_to_use / when_not_to_use가 잘못 generate되면 ##skill 자체가 wrong-state에서 invoked##. Generator의 audit phase가 이를 방지한다고 주장하지만, audit 자체도 LLM 기반이라 ##circular validation##.
  • Public trajectory의 quality 가정: trajectory가 ##non-evaluation##임을 가정하지만, public dataset이 모든 task variant를 cover하지 못하면 skill library가 incomplete. domain transfer (다른 OS, 다른 game)에서의 skill reusability는 미검증.
  • Reference image의 over-anchoring 회피 메커니즘 모호: branch loading이 reference image를 main context에서 분리한다고 주장하지만, ##branch agent가 결정한 plan이 본질적으로 그 reference에 anchored##되어 있을 수 있음. plan의 visual specificity를 정량 측정하지 않음.
  • Focus crop의 위치 결정 자동화: skill package가 focus_crop region을 미리 정의하지만, ##새로운 UI layout이나 다른 해상도##에서는 그 region이 적용 불가. adaptive crop이 필요한 시나리오가 미고려.
  • Safety/Error mode 분석 부재: agent가 ##misapplied skill로 destructive action## (예: 잘못된 파일 삭제)을 할 수 있는 시나리오가 GUI/desktop 환경에 잠재. error recovery나 verification fail 시 행동은 미정의.

Key Takeaways

  • "Text-only skill은 일관성 없는 향상을 준다": 일부 case (Calc 38.30 vs 57.45)에서 ##오히려 성능 저하##. visual state matching이 결정적인 도메인에서는 procedure만으론 부족. ##multimodal skill이 standard가 되어야 한다##.
  • "State card가 visible cue보다 더 중요한 경우가 있다": when_not_to_use 같은 ##negative constraint##가 agent의 exploration을 줄임. text 절차에서 흔히 누락되는 정보.
  • "Branch loading은 distillation의 새 모드": 전체 skill package를 inject하지 않고 ##distilled guidance tuple##만 전달. context pressure를 줄이면서 reference image over-anchoring을 동시에 해결.
  • "약한 VLM이 가장 큰 이득을 본다": Qwen3-VL-8B-Instruct에서 OSWorld 2.4× 향상. external procedural knowledge가 ##internal prior가 약한 모델의 cognitive aid##로 작동.
  • "Action mode distribution이 agent 품질의 진짜 지표": click-heavy → structured input의 전환은 success rate보다 더 근본적인 quality signal. ##exploratory trial-and-error를 grounded execution으로 바꾸는 것##이 multimodal skill의 본질적 기여.
  • "Focus crop이 가장 자주 사용된다": 4종 view 중 focus crop이 압도적. visual agent 시스템 설계 시 ##local crop 자동 생성 메커니즘에 우선 투자##.
