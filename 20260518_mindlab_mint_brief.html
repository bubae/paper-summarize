"Base 모델은 한 번만 띄워두고, 그 위에서 수만 개의 LoRA adapter가 시간을 나눠 살아 움직인다. 학습-서빙-롤백-평가가 모두 'adapter revision'이라는 단일 단위로 흘러간다."

기존 인프라의 한계

기존 시스템은 각 변종마다 full checkpoint를 복사해 서빙. 30B 모델이면 변종 하나가 60+ GB를 차지. 100개 변종 → 6 TB 이상의 storage + 100개 inference server. 학원이나 중소 기업은 이를 못 견딘다.

▲ Full-checkpoint 시대 vs MinT 시대: artifact 크기가 100× 이상 감소.

배경 지식: 알아두면 좋은 6가지 개념

1) Post-training (후처리 학습)

Pretrain된 LLM에 추가 학습을 입혀 특정 도메인/스타일/능력을 발현시키는 단계. SFT (감독학습), RLHF (인간 피드백 RL), GRPO 같은 reasoning RL이 대표.

2) LoRA (Low-Rank Adaptation)

Base 모델의 weight를 그대로 두고 low-rank 행렬 두 개를 곁붙여 학습. Base 메모리는 그대로, 학습 파라미터는 1% 미만. Base + LoRA = adapted 행동.

3) Multi-tenant LLM

한 인프라가 여러 고객/팀/제품의 LLM 변종을 동시에 서빙. 클라우드 서비스의 자연스러운 형태이지만 GPU 자원이 곱셈으로 늘면 비용 폭발.

4) GRPO (Group Relative Policy Optimization)

한 prompt에 여러 답안을 만들고 group 내 상대 우수성을 reward로 사용하는 RL. DeepSeek-R1 이후 reasoning 모델의 표준.

5) MoE (Mixture-of-Experts)

전체 파라미터 일부 expert만 활성화하는 효율 아키텍처. Qwen3-30B-A3B = 30B 중 3B 활성, Kimi K2 = 1.04T 중 32.6B 활성. LoRA를 expert별로 붙이면 tiny tensor가 수만 개 생긴다.

6) vLLM

현대 LLM serving 엔진 표준. PagedAttention과 KV cache 관리, batching, LoRA dynamic loading 등을 제공. MinT는 vLLM 위에 정책 lifecycle 추상화를 얹는다.

MinT의 핵심 추상화: Adapter Revision vs Policy Record

MinT의 가장 중요한 conceptual contribution은 두 개체를 명시적으로 분리한 것이다:

📦 Adapter Revision

특정 학습 step에서 frozen된 LoRA의 snapshot. Rollout/평가/서빙/롤백이 소비하는 실행 가능한 payload. 한 번 export하면 변하지 않는다.

📋 Policy Record

그 payload를 schedulable/durable하게 만드는 service state. compatible base 버전, LoRA rank, target modules, latest checkpoint, rollout records, exported revisions의 목록 등을 보관.

🎬 비유: 영화 필름 vs 영화관 운영 정보. 필름(adapter revision)은 한 번 찍히면 그 자체는 고정 — 재생만 하면 같은 영화가 나온다. 영화관 운영 정보(policy record)는 어느 상영관에서 언제 상영 중인지, 다음 상영은 언제인지, 백업 필름은 어디에 있는지를 추적하는 lifecycle metadata. 둘을 섞어두면 필름이 망가지거나 운영이 멈춘다.

왜 이 분리가 중요한가

  • 한 policy record가 여러 cache tier에 동시 존재 가능 (durable storage, CPU cache, GPU batch).
  • 학습 worker가 사라져도 나중 요청이 같은 adapter revision을 selectable.
  • 학습 도중 export된 revision이 서빙되는 동안 trainer는 다음 update를 계속.

세 가지 확장 축

MinT는 같은 adapter lifecycle을 3가지 방향으로 scale한다:

▲ MinT의 3-axis scaling: 같은 lifecycle이 모델 크기·artifact 크기·정책 개수 세 방향으로 확장.

🚀 Scale Up — 1T 파라미터까지

Megatron-style tensor parallel + expert parallel로 base를 sharding. LoRA tensor가 그 위에 compatibility 있게 매핑: dense LoRA는 TP shard 따라가고, expert LoRA는 EP shard가 own한다. Kimi K2 1.04T 모델 LoRA RL까지 검증.

📉 Scale Down — Full checkpoint 제거

학습-서빙 사이로 옮기는 artifact가 full model이 아닌 LoRA만. Qwen3-4B에서 252 MiB vs 8 GB = 3.3%. Rank-1 설정이면 0.10% 미만. 매 update의 wall-time이 18.3× 빠름.

🌐 Scale Out — 백만 개 정책

Addressable catalog (10^6)와 engine-local working set (수백)을 분리. 정책 이름은 카탈로그에 있고, 실제 실행은 cache가 결정. Cold load는 scheduled service work로 격상 (deduplication + bounded backpressure).

실용 기법: Time-Sliced Multi-LoRA Training

한 base 위에서 여러 정책을 시간 분할로 학습. mLoRA처럼 한 batch에 여러 adapter를 fusion하는 게 아니라, gradient step 단위로 switch.

▲ Base 모델은 상주, switch 때 옮기는 건 LoRA-specific state뿐.
🎓 비유: 한 강의실에서 수업 시간표대로 다른 강사가 들어와 강의한다. 강의실(base)은 그대로, 강사의 강의 자료와 노트(adapter)만 교체. 모든 강사를 동시에 한 강의실에 넣으려는(mLoRA 방식) 대신, 시간을 나눠 쓰는 게 RL 환경에선 더 효율적이다 — 각 강사가 가져오는 학생(rollout token)이 이미 충분히 많기 때문.

흥미로운 시스템 트릭: Packed MoE LoRA

MoE base에 LoRA를 붙이면 expert별로 tiny tensor가 분리되어 한 adapter가 37,248개의 작은 객체로 쪼개진다. 파일 크기는 110 MB 정도지만 Python 객체 등록 비용이 N배.

▲ Packed representation은 byte는 같고 loader-side object 수만 줄여 inference 시작 latency를 단축한다.
📦 비유: 100개 작은 박스에 똑같이 든 물건과 1개 큰 박스에 한꺼번에 든 물건. 내용물 무게는 같지만 박스를 열어서 등록하는 시간은 100배 차이. cold-load 경로에선 이 차이가 transient latency를 결정한다.

Cold-load의 비용 측정

Cache miss는 단순 파일 읽기가 아니다: tensor fetch → loader object materialization → engine registration → adapter activation. 16개 distinct cache-miss adapter가 도착하면 1.375s → 23.267s의 staircase (약 1.36 s/adapter)로 직렬화.

따라서 cold load는 scheduled service work로 다뤄야 한다: deduplication(같은 missing policy는 한 번 로딩 후 공유), bounded backpressure(너무 많이 동시에 들어오면 throttle).

실용 케이스: AutoResearch on LawBench

법률 reasoning 도메인에서 28개 실험 candidate를 proxy로 screening, 6개 kept, v23이 winner로 maintain되는 과정을 같은 adapter lifecycle이 자연스럽게 carry한다.

▲ AutoResearch는 single high point가 아니라 running-best line + kept-candidate filter로 진행. v11의 proxy-full divergence가 distribution shift를 자동 탐지.

한계점

① 저자/소속 익명성

"Mind Lab"만 명시. 기업 기술 백서 형태이므로 peer review를 통한 method scrutiny가 어렵다. 코드는 mint-cookbook으로 일부 공개되지만 management plane 핵심은 closed.

② Warm inference에서 Merge가 약간 빠름

Adapter handoff는 transfer가 18.3× 빠르지만 warm sample 속도는 merge path가 약간 우월(20.6 vs 15.6 tok/s). Long-lived deployment에서 이 차이가 transfer saving을 잠식하는 break-even point가 명시되지 않음.

③ 10⁶ 카탈로그는 extrapolation

실측은 100k까지. 10⁶은 sizing model에 의한 추정값이며 실제 measured 값이 아님. abstract는 다소 단정적.

④ Time-slicing vs mLoRA의 정량 비교 부재

RL의 큰 rollout token이라는 가정 하의 design choice. 짧은 rollout 환경에서 mLoRA fusion이 유리한 regime을 정량적으로 비교하지 않음.

⑤ DSA correction의 정확도 미검증

IcePop-style mask가 unsafe scoring term을 0으로 만들지만, 그것이 학습 신호를 얼마나 손상시키는지 ablation이 없음. R3 explicit replay와의 quality 비교가 누락.

한 줄 결론

Post-training은 OS가 필요한 워크로드다. MinT는 그 OS의 첫 reference implementation.
학습-서빙 경계가 가로지르는 artifact를 "GB짜리 base"에서 "MB짜리 adapter"로 바꾸면,
trillion-parameter LoRA RL이 학원 규모 클러스터에서도 가능해진다.

가져갈 인사이트 3가지

  • "Payload vs Service State 분리": adapter revision (실행 가능 payload)과 policy record (서비스 lifecycle state)의 명시적 분리는 다른 분산 시스템에도 transferable한 디자인 패턴.
  • "같은 Byte, 다른 Object Layout": Packed MoE LoRA의 8.5× live-load 가속은 사이즈가 아니라 객체 fanout이 cache-miss latency를 지배한다는 system-level 교훈. sparse model serving 일반에 적용 가능.
  • "3-Tier Cache는 측정 가능한 SLA": 추상적인 "cache" 대신 catalog/CPU/GPU 3계층을 각자 latency·promotion·eviction이 다른 표면으로 노출하면 cold-load가 scheduled work가 되어 throttle/dedup 가능해진다.
