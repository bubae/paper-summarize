"학생에게 두꺼운 교과서 음독을 시키는 대신 아무 곳에서나 질문을 던지는 게 진짜 '두꺼운 책을 다루는 능력'을 길러준다. Vision LLM 학습도 마찬가지였다."

왜 이 문제가 중요한가

"긴 문서를 다루는 능력"이 차세대 AI의 핵심

2026년 현재 AI가 풀어야 하는 task가 점점 길어진다:

📚 100쪽 보고서 QA

"이 의학 논문 47쪽의 통계와 12쪽의 정의를 합쳐 결론을 내려라"

📹 1시간 비디오

회의 영상에서 특정 의사결정 순간을 찾아내기

🌐 웹 페이지 다수

여러 웹페이지를 한꺼번에 읽고 핵심 정보 retrieval

🤖 Agentic workflow

여러 turn의 대화 history 일관성 유지

그런데 어떻게 가르치나? — 공개된 레시피가 없다

Gemini 3 Pro, GPT-5.4, Qwen3-VL, GLM-4.5V 같은 frontier LVLM이 128K+ context를 지원한다고 발표하지만, 정작 어떻게 그렇게 학습했는지 detail은 거의 공개되지 않는다. 학원 / open-source 진영에서는 다음 질문들이 missing:

❓ 미해결 질문들

  • 어떤 종류의 데이터를 써야 long-context 능력이 늘까? (긴 문서 음독? VQA?)
  • 데이터 길이 분포는 어떻게? (전부 128K로? 아니면 다양하게?)
  • 여러 task를 어떤 비율로 섞을까?
  • 짧은 데이터를 같이 학습해야 short-context 능력이 보존되나?
  • RoPE base frequency는 얼마로 scale?

이 paper는 이 5가지 질문에 대해 controlled ablation 5B token budget으로 답을 제시.

배경 지식: 알아두면 좋은 6가지 개념

1) LVLM (Large Vision-Language Model)

이미지와 텍스트를 동시에 다루는 LLM. GPT-5, Gemini 3, Claude, Qwen-VL 등. 이미지 한 장당 수백~수천 토큰을 소비.

2) Context Window

모델이 한 번에 보고 추론할 수 있는 토큰 총량. 32K = 약 24,000 단어. 128K = 약 96,000 단어. 길수록 더 긴 문서를 처리.

3) Long-Context Continued Pre-training (LongPT)

이미 학습된 모델을 추가로 긴 시퀀스 데이터로 더 학습시켜 context window를 확장. 본 논문의 핵심 기법.

4) Long-Document VQA

긴 문서 (수십~수백 페이지)를 보고 특정 위치의 정보에 대해 묻고 답하는 task. "47쪽 표 3에서 보고된 매출액은?" 같은.

5) OCR Transcription

문서 이미지를 페이지별로 텍스트화하는 task. 본 paper는 long-doc VQA와 OCR을 비교.

6) RoPE (Rotary Position Embedding)

토큰의 위치 정보를 인코딩하는 방법. base frequency를 scale하면 더 긴 context를 표현 가능 (Dynamic-NTK heuristic).

가장 큰 발견: Long-Doc VQA >> OCR

저자들이 5가지 후보 task로 같은 5B token budget으로 학습시켜 비교:

▲ VQA 3종은 모두 +5~6%, OCR은 -6.8~-17.4. SFT 보정해도 VQA에 못 미친다.
📚 비유: 학생에게 두꺼운 교과서를 처음부터 끝까지 음독하라고 시키면 (OCR) 목소리 훈련은 되지만 내용 이해는 안 됨. 아무 페이지에서나 질문을 던지는 것(VQA)이 진짜 '교과서를 다루는 능력'을 만든다.

왜?

  • OCR은 sequential dense output만 만들어 instruction-following style과 mismatch.
  • VQA는 다양한 task diversity (extraction~reasoning)를 제공.
  • VQA는 다양한 위치에서 retrieve하는 연습 → generalize.

Long-Doc VQA 데이터 만드는 법

긴 문서 (32–50 페이지)를 그대로 QA 생성기에 넣으면 비용이 폭증한다. 저자들의 영리한 trick:

▲ short segment에서 QA를 만들고 full document context와 결합해 long-context training instance 완성.

3가지 VQA 난이도

난이도설명예시
extract-single단일 페이지에서 사실 추출"39쪽 레시피에서 herbs는 vodka에 얼마나 담그라고 하나?"
extract-multi여러 페이지에서 사실 통합"6쪽, 13쪽, 19쪽에 언급된 모든 위험 요인을 list로"
reasoning추출 후 numerical/logical 추론"2020년 총 소비량과 총 수입량의 차이는?"

핵심 디자인: false positive 방지

같은 질문이 sampled segment에선 명확하지만 full document에선 ambiguous할 수 있다 (예: "보고된 매출액?" → 여러 부서/년도 매출 존재). 해결: segment anchor 추가 ("in the Introduction section", "on pages 20–25").

Counter-intuitive 발견: 다양한 길이가 더 좋다

"128K context에서 평가받을 거니까 학습 데이터도 100K+에 집중시켜야지" — 직관적이지만 틀렸다.

55.7 (+1.3) extract-multi avg: 56.8 reasoning avg: 56.5 (+1.7)
▲ 다양한 길이의 자연스러운 분포가 일관되게 우월. 더 적은 near-128K 데이터를 쓰면서도.
🏃 비유: 마라톤(42km)을 위해 42km만 반복 연습하는 게 좋을까? 아니다. 5km, 10km, 21km, 30km 다양한 거리를 mix해서 훈련하는 게 더 좋다. 다양한 페이스와 위치에서의 retrieval 능력이 특정 거리 specialization보다 generalize한다.

해석: long-context 능력은 specific length가 아니라 generalizable retrieval skill. 모델은 다양한 위치와 거리에서 key information을 찾는 법을 배워야 한다.

Optimal Mixture: Extraction 8, Reasoning 2

추출(extraction)과 추론(reasoning) data를 어떤 비율로 섞을까? 저자들이 grid search:

▲ Extraction-heavy mixture가 우월. 8:2가 best. 단일 데이터(10:0)나 reasoning-heavy(0:10)보다 좋다.
🥗 비유: 좋은 다이어트는 주식 80% + 보조 영양소 20%이다. 한 종류만 먹으면 결핍이 생긴다. Long-context 학습도 정보 추출 80% + 추론 20%가 balanced.

해석: long-context의 진짜 병목은 retrieval (key info 찾기). reasoning은 task diversity 보존을 위한 minimal supplement.

또 다른 surprise: Short-context data 불필요

LLM에서는 long-context training이 short-context 능력을 손상시킨다고 알려져 있다 (Fu et al. 2024). 그래서 short data를 일정 비율 섞어야 한다.

그러나 LVLM에서는 다르다:

Pure long training (0% short data)

  • Long-doc VQA: 57.70 (최고)
  • Short-context capability: 65.48 (vs original 66.47, 단 -0.99)
  • → Long 성능 최고 + short 거의 그대로

20% short data 추가

  • Long: 55.57 (-2.13 손실)
  • Short: 66.53 (+1.05)
  • → Short 회복하지만 long 더 큰 손실

왜? Long-document VQA가 이미 instruction-formatted이라 모델이 short-context의 instruction-following style을 자연 보존한다. LLM과 LVLM의 본질적 차이.

🧑‍🎓 비유: 대학생이 두꺼운 전공 책으로 공부해도 단답형 시험을 보는 능력이 거의 그대로 유지된다. 두꺼운 책의 문제 풀기 연습 자체가 단답형 시험의 'instruction-따라가기'와 본질적으로 같기 때문.

한계점

① Closed-source와 큰 격차

GPT-5.5 90.77 vs MMProLong 57.70. 7B vs frontier의 절대 차이가 크다. "open-source 중 SOTA"이지 overall SOTA가 아니다.

② 5B token budget 임의성

모든 ablation이 5B 토큰. 더 큰 budget(20B, 50B)에서는 long-biased가 우월해질 가능성이 있음.

③ Document domain 편향

학습 데이터가 학술 논문, 책, 기술 매뉴얼 위주. 대화, 코드, 법률같은 다른 long-context 도메인 transfer는 미검증.

④ OCR과 fair comparison 의문

OCR이 본질적으로 instruction-formatted가 아니라 평가에서 불리할 수 있다. instruction wrapper로 OCR을 dress up한 fair comparison이 없음.

⑤ Mechanistic explanation 부족

"왜 LVLM은 short capability를 유지하는가"에 대한 mechanistic analysis가 부족. instruction-formatted라는 설명은 plausible하지만 정량 검증 미흡.

활용 분야

▲ Long-context VLM의 응용 영역.

한 줄 결론

Long-context는 특정 길이의 specialization이 아니라 generalizable retrieval이다.
5B 토큰으로 32K → 128K, 그리고 256K/512K까지 추가 학습 없이 generalize.
Open-source 진영에 frontier 수준의 long-context 능력을 만드는 첫 공개 레시피.

가져갈 인사이트 3가지

  • "Supervision의 task structure가 raw scale보다 중요": 같은 token budget으로 VQA가 OCR보다 5–23점 우월. instruction-formatted supervision이 모델에게 어떻게 사용될지를 가르친다. 다른 long-context training (audio, video)에도 transferable.
  • "Diversity > Specialization": pool-native 분포 우월, extraction 8 + reasoning 2 mix 우월. 모델은 generic retrieval skill을 배워야지 specific length나 single task type을 외워선 안 된다.
  • "LVLM은 LLM과 다른 short-context 보존 behavior": pure long-context training이 short capability를 단 -0.99만 감소시킨다. instruction-formatted data로 train되는 LVLM의 새로운 advantage.
