OneComp

AI 모델을 "한 줄 명령"으로 자동 압축하는
올인원 오픈소스 프레임워크

"수십 억 개의 AI 뇌세포를 자동으로 다이어트 시켜서
절반의 메모리로도 거의 같은 지능을 유지하게 한다"

먼저 알아야 할 것: AI 모델은 왜 이렇게 큰가?

OneComp를 이해하려면, AI 모델이 왜 거대하고 그것이 왜 문제인지를 먼저 알아야 합니다. 세 가지 핵심 개념을 그림과 함께 설명합니다.

개념 1: AI 모델 = 수십억 개의 "숫자 뭉치"

ChatGPT 같은 대형 AI 모델(LLM)은 내부에 수십억~수천억 개의 숫자(parameter)를 가지고 있습니다. 이 숫자들이 곧 AI의 "지식"이며, 모두 GPU 메모리에 올라가야 AI가 작동합니다.

최신 AI 모델은 GPU 한 장의 메모리를 크게 초과하며, 여러 장을 묶어야 작동한다

개념 2: 메모리가 부족하면 AI를 쓸 수 없다

AI 모델의 모든 숫자는 GPU 메모리에 올라가야 합니다. 메모리가 부족하면 모델을 실행할 수 없습니다. GPU는 매우 비싸기 때문에 (A100 한 장에 약 2,000만 원), 메모리를 절약하는 것이 곧 비용을 절약하는 것입니다.

모델 압축은 AI 서비스 비용을 직접적으로 절감하는 핵심 기술
이삿짐의 비유

AI 모델은 거대한 이삿짐 트럭 가득한 짐과 같습니다. 원래는 대형 트럭(비싼 GPU) 여러 대가 필요합니다.


모델 압축은 짐을 진공 압축팩으로 줄이는 것과 같습니다. 옷의 질감은 그대로인데 부피만 1/4로 줄어들어, 소형 트럭 하나로도 충분히 옮길 수 있습니다.

개념 3: 양자화(Quantization) — 숫자의 정밀도를 낮추기

모델 압축에서 가장 많이 쓰이는 방법이 양자화(Quantization)입니다. AI 모델의 각 숫자는 보통 16비트(소수점 이하 매우 정밀)로 저장되어 있는데, 이를 4비트나 3비트 같은 낮은 정밀도로 "반올림"하여 메모리를 절약합니다.

16-bit를 4-bit로 줄이면 메모리 1/4, 대신 약간의 정밀도 손실이 발생
가격표 반올림의 비유

16-bit는 모든 상품 가격을 "12,847원"처럼 원 단위까지 정확히 기록하는 것입니다.


4-bit 양자화는 "약 13,000원"으로 천 원 단위로 반올림하는 것과 같습니다. 장부가 1/4로 얇아지지만, 대부분의 거래에서 큰 문제가 없습니다. 다만 "12,847원과 12,851원"처럼 미세한 차이가 중요한 경우에는 정보가 손실됩니다.

개념 4: AI 모델 내부 = 층(Layer)이 반복된 구조

ChatGPT 같은 AI 모델(Transformer)은 같은 구조의 "블록"이 수십 개 겹겹이 쌓인 형태입니다. 각 블록 안에는 7개의 큰 숫자 행렬(weight matrix)이 들어있고, 이것들이 메모리의 대부분을 차지합니다.

Llama-3-8B은 32개 Transformer 블록으로 구성되며, 각 블록에 7개의 weight matrix가 있다. 양자화는 이 행렬들의 숫자 정밀도를 낮추는 것

그런데, 양자화는 왜 어려운가?

양자화의 핵심 난관은 크게 두 가지입니다: (1) AI 모델의 각 층(layer)마다 숫자의 분포가 달라서 "하나의 반올림 규칙"이 모두에게 맞지 않고, (2) 한 층에서 생긴 반올림 오차가 다음 층으로 전파되어 점점 커집니다.

양자화의 두 핵심 문제: 층간 민감도 차이와 오차 전파. OneComp는 둘 다 자동으로 해결한다

기존 양자화 도구의 문제점

양자화 기술 자체는 오래 전부터 있었습니다. 문제는 도구가 너무 파편화되어 있다는 것입니다. 마치 수십 가지 조리 도구를 일일이 골라서 써야 하는데, 어떤 도구를 어떤 순서로 쓸지 전문가만 안다는 상황입니다.

OneComp는 파편화된 양자화 도구 생태계를 하나의 자동화 파이프라인으로 통합
세탁기의 비유

기존 양자화는 빨래를 할 때 비누 종류, 물 온도, 세탁 시간, 탈수 강도를 매번 직접 선택해야 하는 것과 같습니다. 섬유 종류에 따라 최적의 조합이 다르고, 잘못 선택하면 옷이 망가집니다.


OneComp는 "의류 자동인식 세탁기"와 같습니다. 옷을 넣고 버튼 하나만 누르면, 기계가 자동으로 섬유를 분석해서 최적의 세탁 코스를 적용합니다. 결과물은 전문 세탁사와 비슷하거나 더 좋습니다.

OneComp의 핵심 아이디어

OneComp의 핵심은 "점진적 다듬기"(Progressive Refinement)입니다. 한 번에 완벽한 압축을 하려는 게 아니라, 3단계에 걸쳐 조금씩 품질을 개선합니다. GPU가 충분하면 더 깊은 단계까지, 부족하면 중간에 멈춰도 괜찮습니다.

OneComp의 3단계는 글쓰기 과정과 똑같습니다:


1단계 (층별): 초고 작성 - 각 문단을 개별적으로 다듬습니다. 빠르고 적은 자원으로 가능하지만, 문단 간 연결이 매끄럽지 않을 수 있습니다.

2단계 (블록별): 챕터 교정 - 인접한 2-3개 문단을 묶어서 흐름을 조정합니다. 1단계보다 더 좋은 결과물이 나옵니다.

3단계 (전체): 최종 편집 - 책 전체를 통독하며 전체 흐름과 일관성을 다듬습니다. 가장 좋은 품질이지만 시간이 더 걸립니다.


핵심은? 1단계만으로도 출판 가능한 수준이고, 시간이 있으면 2-3단계로 더 완벽하게 만들 수 있다는 것입니다.

OneComp는 모델 이름 하나만 입력하면 전체 파이프라인이 자동 실행된다. Stage 1 완료만으로도 배포 가능

핵심 기술 1: AutoBit — 자동 맞춤형 bit 할당

AI 모델의 32개 층이 모두 같은 중요도를 갖진 않습니다. 어떤 층은 1비트만 줄여도 성능이 폭락하고, 어떤 층은 절반으로 줄여도 괜찮습니다. AutoBit는 각 층의 "민감도"를 자동 측정하여, 주어진 메모리 예산 안에서 최적의 bit를 배분합니다.

AutoBit는 각 층의 민감도를 빠르게 측정한 뒤, ILP 솔버로 수학적 최적 할당을 자동 계산
Llama-3-8B에서 평균 4.0 bit: 균등 할당은 PPL 383으로 실패하지만, AutoBit은 PPL 7.04로 원본에 근접
시험 공부 시간 배분의 비유

시험 5과목을 준비하는데 총 10시간이 있다고 합시다.


균등 할당: 각 과목에 2시간씩. 하지만 이미 잘하는 수학에 2시간을 쓰고, 약한 역사에도 2시간만 쓰면 전체 점수가 나빠집니다.


AutoBit: 수학(쉬운 과목)은 1시간만, 역사(약한 과목)에 4시간을 배분합니다. 총 공부 시간은 같지만, 전체 점수는 훨씬 높아집니다. AutoBit은 이 "최적 배분"을 수학적으로 자동 계산합니다.

핵심 기술 2: QEP — 오차 전파를 자동 보정

기존 양자화 도구(GPTQ)는 각 층을 압축할 때 "원래 숫자에 최대한 가깝게" 반올림합니다. 하지만 이전 층에서 이미 오차가 생겼다면, "원래 숫자"를 목표로 삼는 것 자체가 잘못된 전략입니다. QEP는 이전 층의 오차를 고려하여 목표 자체를 수정합니다.

QEP는 단순히 "원래 값을 맞추는" 대신 "올바른 출력을 내는 값"을 목표로 설정하여 오차 전파를 차단
내비게이션의 비유

기존 GPTQ: 경로를 이탈했을 때 "원래 목적지 방향"으로만 갑니다. 하지만 이미 엉뚱한 도로에 있으면, 원래 방향으로 가는 것이 오히려 더 멀리 돌아가게 만들 수 있습니다.


QEP: 경로를 이탈했을 때 "현재 위치에서 목적지에 가장 빨리 도착하는 새 경로"를 계산합니다. 이전의 실수를 인정하고, 현재 상태에서 최선을 찾습니다.

핵심 기술 3: JointQ와 MDBF

JointQ: Scale과 Weight를 함께 최적화

기존 GPTQ는 "반올림 규칙(scale)"을 먼저 정하고, 그 규칙에 따라 숫자를 반올림합니다. 하지만 JointQ는 반올림 규칙과 반올림된 숫자를 동시에 최적화합니다. 마치 자물쇠와 열쇠를 함께 맞추는 것처럼, 둘이 서로에게 최적인 조합을 찾습니다.

JointQ는 GPTQ와 동일한 파일 형식을 출력하므로, 기존 시스템에 바로 적용 가능

MDBF: 1-bit 극한 압축의 혁신

보통의 양자화는 3-4 bit까지가 한계입니다. MDBF (Multi-envelope Double Binary Factorization)는 완전히 다른 접근법으로 1-bit 양자화를 가능하게 합니다. 숫자를 반올림하는 대신, 행렬 전체를 "+1과 -1로만 구성된 두 행렬의 곱"으로 분해합니다.

MDBF는 가중치 행렬을 이진 행렬로 분해하여 1-bit 압축을 가능하게 함. 메모리 16배 절감

성과 1: AutoBit으로 catastrophic failure 방지

Llama-3-8B에서 평균 4-bit 양자화 시, 균등 할당은 PPL 383으로 완전히 망가지지만, AutoBit의 activation-aware 할당은 PPL 7.04로 원본(6.14)에 근접합니다. 소수의 매우 민감한 모듈에 높은 bit를 자동 배정하여 전체 모델의 붕괴를 방지합니다.

OneComp의 핵심 기술(QEP, AutoBit)은 기존 GPTQ 대비 극적인 성능 개선을 달성

성과 2: JointQ로 GPTQ를 모든 설정에서 능가

Llama-3-8B에서 JointQ는 4-bit/128그룹 설정에서 PPL 12.66 → 6.67, 3-bit per-channel에서 PPL 1721 → 21.15로 개선했습니다. 특히 aggressive한 압축 설정일수록 개선 폭이 더 큽니다.

성과 3: MDBF로 1-bit 극한 압축 가능

MDBF는 모든 모델과 bit 수준에서 기존 DBF를 일관되게 능가합니다. Llama-3-8B에서 1-bit: PPL 14.76 → 13.25, 1.5-bit: PPL 10.61 → 10.03. 모델이 클수록 개선 효과가 더 큽니다.

성과 4: 단계를 더할수록 성능이 단조 향상

Layer-wise → Block-wise → Global 각 추가 단계가 항상 성능을 개선합니다. GPU 자원이 허용하는 만큼 더 깊은 단계를 적용하면 더 좋은 결과를 보장합니다.

Llama-3.2-1B에서 DBF 양자화: 모든 bit-width에서 각 추가 단계가 성능을 단조 개선

실험 규모가 아직 제한적

현재 실험은 8B, 14B 모델에서만 진행되었습니다. 실제 프로덕션 환경에서 흔히 사용되는 70B 이상 모델이나 MoE(Mixture-of-Experts) 모델에서의 효과는 아직 검증되지 않았습니다. 모델이 커질수록 양자화의 특성이 달라질 수 있으므로, 대규모 모델에서의 추가 검증이 필요합니다.

평가 벤치마크가 제한적

성능 평가가 WikiText-2 Perplexity와 4개 zero-shot 태스크(ARC, PIQA, WinoGrande)에 한정됩니다. MMLU(지식 이해), HumanEval(코드 생성), MT-Bench(대화 품질) 등 현대적 벤치마크에서의 평가가 없어, 실제 서비스 품질에 미치는 영향을 판단하기 어렵습니다.

일부 핵심 기능이 아직 미완성

논문에서 가장 혁신적인 두 기능이 "현재 개발 중"입니다:

  • Global PTQ (Stage 3): 전체 모델 최적화 단계가 향후 릴리스 예정
  • MDBF: 극한 1-bit 압축 기능도 향후 릴리스 예정

즉, 현재 공개된 오픈소스 코드에서는 논문에 설명된 전체 파이프라인을 아직 사용할 수 없습니다.

실행 시간과 GPU 사용량 미보고

"Resource-adaptive"를 강조하지만, 각 단계의 실제 실행 시간과 GPU 메모리 사용량에 대한 구체적인 데이터가 없습니다. 따라서 "한 줄 명령"이 실제로 얼마나 빠른지, GPU 한 장으로 어디까지 가능한지 정량적으로 판단하기 어렵습니다.

기존 통합 프레임워크와의 비교 없음

AutoGPTQ, llama.cpp, vLLM 등 이미 널리 사용되는 양자화 도구와의 직접 비교가 없습니다. 개별 알고리즘(GPTQ vs JointQ)의 비교만 있으므로, "OneComp를 써야 하는 이유"가 기존 도구 대비 얼마나 강한지 판단하기 어렵습니다.

균형 잡힌 시각

OneComp는 파편화된 양자화 생태계를 통합하는 의미 있는 시도이며, QEP의 오차 보정 효과(PPL 87배 개선)는 인상적입니다. 하지만:

  1. Global PTQ와 MDBF가 공개되기 전까지는 논문의 full pipeline을 재현할 수 없습니다
  2. 70B+ 모델과 실제 서비스 환경에서의 안정성은 아직 미지수입니다
  3. 도입 검토 시 기존 도구(AutoGPTQ 등)와의 실용적 비교를 먼저 확인해야 합니다
