먼저 알아야 할 것: 추론 AI의 한계

2025년 화제가 된 DeepSeek R1, OpenAI o1 같은 ##추론 모델##은 문제를 풀 때 ##길게 생각하는(long chain-of-thought) 능력##이 뛰어납니다. 사람처럼 "잠깐, 다시 생각해보자"라며 자기 답을 점검하기도 하죠.

하지만 이런 모델도 ##다음 영역에서는 약합니다##:

  • 정밀 산술: 큰 수의 모듈러 연산("1000 mod 7 = ?")
  • 기하 추론: 정확한 좌표 계산
  • 방정식 풀이: 복잡한 다변수 방정식
  • 경우의 수 탐색: 1부터 1000까지 일일이 확인

이런 일은 ##Python 코드를 짜서 실행##하면 100% 정확히 풀립니다. 그래서 자연스러운 질문: "추론 모델이 필요할 때 Python을 부르면 안 될까?"

기존 시도: SFT로 도구 사용 가르치기

이전 방법들(Toolformer, Qwen-Math-TIR 등)은 "도구 사용 예시들"을 모아 supervised learning으로 가르쳤습니다. 하지만 이는 ##정답 풀이를 외우는 것##과 비슷해서, 새로운 문제에서 ##언제 도구를 부를지를 잘 못 결정합니다##.

SFT는 모방, RL은 결과 기반 학습
비유: 요리사 양성

SFT 방식: "이 레시피를 외워라"라고 시킴. 그 요리만 만들 수 있고, 응용 못 함.

RL 방식: 손님 평가로 맛있으면 +1, 맛없으면 -1. 어떤 양념이 언제 좋은지 스스로 발견. 새 메뉴도 시도.

ReTool의 핵심 아이디어

ReTool은 ##추론 중에 코드를 끼워넣고 실행 결과를 받아 다시 추론하는 사이클##을 강화학습으로 훈련합니다. 답이 맞는지만 봅니다. ##언제 코드를 쓸지, 어떻게 쓸지, 틀리면 어떻게 고칠지##는 모두 AI가 시행착오로 발견.

RL 학습 중 추론 과정

텍스트 + 코드 + 실행결과가 한 추론 흐름 안에 인터리브
비유: 시험에서 계산기 사용

학생이 수학 시험을 보면서, "이 부분은 머리로 풀고, 이 부분은 계산기로 풀자"고 결정. 계산기 결과가 오류면 다시 계산. 시험 끝나고 답이 맞았는지에 따라 보상. 그 보상으로 ##계산기 사용 timing##을 학습.

가장 놀라운 발견: "Aha Moment"

학습 데이터에 ##self-correction(자기 수정) 예시는 없었습니다##. 그런데 RL 학습 후, AI는 ##스스로 코드 오류를 발견하고 고치는 능력을 발견##했습니다.

학습 데이터 없이 emergent 능력
비유: 신입 프로그래머의 디버깅 습관 발견

신입에게 "디버깅하는 방법"을 가르치지 않았는데, ##오답에 대한 페널티만으로## 신입이 스스로 디버깅 습관을 발견. "Oops, 이 변수가 undefined네요"라고 혼잣말하며 수정.

비유로 다시 정리

비유 1: 견습 셰프와 도구

SFT는 "이 레시피 외워"라고 시키는 견습. ReTool은 "맛으로만 평가"하며 도구(칼, 믹서, 오븐) 사용 타이밍을 시행착오로 배우는 견습. 결과적으로 자기만의 스타일 개발.

비유 2: 시험 중 계산기 사용

학생이 수학 시험 중 계산기를 ##언제 쓸지##를 스스로 결정. 손으로 풀 수 있는 부분은 손으로, 계산기 필요한 부분은 계산기로. 답 채점 결과로만 학습.

비유 3: 디버깅 습관 발견

"Oops, 이 함수 정의가 빠졌네" — 신입 프로그래머가 누구도 가르치지 않은 ##self-correction 습관##을 발견. ReTool도 "aha moment"라 불리는 emergent 능력.

비유 4: 운동선수의 폼 개선

코치가 폼을 직접 가르치지 않고 ##승부 결과만 보고##. 선수가 시행착오로 자기 폼 발견. ReTool의 outcome-based reward와 같다.

비유 5: 직장에서 도구 사용 진화

신입 → 베테랑이 되면서, ##엑셀을 더 일찍 열고## (timing 앞당김), ##간결한 매크로를 자주 쓰고## (코드 줄 수 증가), ##불필요한 검토 시간 단축## (응답 길이 감소). ReTool의 학습 곡선과 동일.

비유 6: 인생 살이의 시행착오

인생도 모범 답안이 없고 결과로만 평가. 시행착오로 ##언제 친구에게 도움 청할지(도구)##, ##어떻게 실수 만회할지(self-correction)##를 배움. AI도 같은 원리.

한 줄 결론

"도구 사용은 가르치는 것이 아니라 발견하게 하는 것."

한계점

⚠️ 1. 수학 task에만 평가

AIME 2024/2025만 평가. ##코딩, 일반 agent task에서의 효과는 미지수##. 도구도 Python 1종만.

⚠️ 2. 검증 가능한 reward 필요

"정답이 명확한" 문제에만 적용 가능. ##글쓰기, 창작 같은 open-ended task에 적용 어려움##.

⚠️ 3. Sandbox 보안 고려 부재

AI가 임의 Python 코드를 실행할 수 있으므로 ##sandbox escape 위험##. 논문은 보안 isolation 메커니즘을 명시하지 않음.

⚠️ 4. Reward hacking 위험

KL coefficient = 0으로 reference policy 제약 없음. ##모델이 catastrophic drift할 위험##. general capability 보존 검증 부재.

⚠️ 5. Response length 40% 감소의 trade-off

효율은 좋아졌지만 ##깊은 reasoning이 필요한 탐색적 문제에선 손해##일 수 있음. AIME는 수렴적 문제라 괜찮지만 일반화 의문.

역사적 위치

ReTool은 ##도구 사용 학습의 두 번째 패러다임 전환##입니다.

  • 2023 2월: Toolformer — self-supervised tool learning (SFT 기반)
  • 2024 1월: PoT, PAL — prompting 기반 도구 사용
  • 2024 후반: SFT tool-integrated reasoning (TIR 등) — 데이터 imitate
  • 2025 1월: DeepSeek R1 — RL이 reasoning을 학습할 수 있음을 입증
  • 2025 4월: ReTool — RL이 도구 사용 strategy까지 학습

##"RL은 어떤 능력도 학습할 수 있다"##는 DeepSeek R1의 영향을 도구 사용에 확장한 사례. 후속 모든 ##tool-use RL agent## 연구의 토대.

FAQ

Q: 32B 모델이 어떻게 o1을 이겼나요?

A: o1도 강하지만 ##순수 텍스트 reasoning##. ReTool은 ##Python 코드의 정확한 계산 능력을 빌려옴##. 도구가 모델 크기의 갭을 메움. 마치 사람이 계산기로 천재 수학자를 이길 수 있는 것과 비슷.

Q: 일반 사용자가 어떻게 쓰나요?

A: ByteDance의 production 모델(Doubao 등)에 통합될 가능성. 또한 ChatGPT Code Interpreter도 비슷한 기능을 제공. ReTool의 방법론이 ##후속 production AI##에 영향.

Q: "Aha moment"는 무엇이 특별한가요?

A: ##학습 데이터에 없던 능력이 RL로 emerge##했다는 것. AI에게 self-correction을 가르치지 않았는데 학습 후 갖게 됨. 이는 RL이 ##metacognitive ability(사고를 사고하는 능력)##까지 길러줄 수 있다는 증거.
