TL;DR

  • 문제: ChatGPT 이래 모든 LLM은 ##단일 message stream (user→model→tool→…)을 sequential하게 처리##한다. 학생은 입력을 다 읽기 전에 답할 수 없고, 답을 쓰는 중에 새 입력을 받을 수 없으며, 생각하면서 동시에 행동할 수 없다. 이는 ##agent로서 LLM을 사용할 때 사용성, 효율성, 보안, 모니터링 모두에서 근본적 병목##이 된다.
  • 해결: Instruction-tuning을 ##multiple parallel streams of computation##로 재구성. 각 role(user, system, model, thinking, audit, tool 등)이 독립된 stream을 가지며, 매 forward pass에서 모델은 모든 input stream을 동시에 읽고 모든 output stream에서 동시에 token을 생성한다. Stream 간 attention은 시간(position $t$)에 대해서만 causal 제약을 갖는다.
  • 왜 작동하는가: Transformer는 본래 set operation으로 정의되므로, sequential format은 cosmetic 제약이지 architectural 필연이 아니다. ##Stream별 RoPE 인덱싱 + per-stream embedding + cross-stream causal mask## (식 (2))로 token이 어떤 stream에 속하는지, 시간상 어떤 token에 attention 가능한지를 깔끔히 분리할 수 있다. Interleaved packing으로 FlashAttention 호환성도 유지.
  • 비유: 1980년대 단일 코어 CPU가 multi-core/multi-thread로 진화한 것과 같다. 하나의 "대화 thread"에 모든 것을 직렬화하는 chat 패러다임에서 ##각 role이 자기 thread를 갖고 동시 실행되는 OS-스러운 패러다임##으로의 전환.

핵심 기여 (Key Contributions)

  1. Multi-Stream Parallel Generation 정식화: 모델이 $H$개의 stream을 ##controlled cross-stream causal dependency로 동시에 생성##하는 학습 패러다임 정의 (식 (1)). Multiverse 같은 isolated branching이나 Medusa 같은 multi-token prediction과 달리, ##모든 stream이 매 step에서 dense하게 cross-attention##한다.
  2. 3단계 합성 데이터 파이프라인: 자연적 simultaneous data가 부족한 문제를, ##(a) Wait-$k$ stream-like generation, (b) LLM-judge causal verification, (c) per-/cross-stream quality filtering##의 3단계로 해결.
  3. Architecture 수정 2종: (a) Stream-aware RoPE — 각 stream이 자기 position counter를 0부터 시작, (b) Cross-stream causal mask — query at $(h, t)$가 key at $(h', \tau)$를 $\tau < t$일 때만 attend. ##Interleaved packing##으로 사실상 lower-triangular layout을 만들어 FlashAttention fast path를 활용.
  4. 효율성: Qwen3-1.7B/4B (thinking mode)에서 ##TNFT(Token Number to First Target Token)를 0으로 압축##하고 end-to-end latency 40-60% 감소. ##Auditing-while-solving은 1.63× throughput speedup##.
  5. 보안: Adversarial training 없이 ##stream isolation 만으로 prompt injection ASR(Attack Success Rate) 대폭 감소## — TensorTrust 75.56→54.75, StruQ-ID 76.00→42.23, NESSiE Safe&Helpful 8.21→28.21. Indirect injection 방어가 가장 큰 폭으로 개선.
  6. Monitorability: 8개 internal thinking stream을 추가하면 ##"concern sub-vocalization" rate가 0.07→0.42 (8B), 0.06→0.44 (27B)##. Visible answer에 드러나지 않는 모델의 내부 고려사항(eval awareness 등)이 별도 stream에 등장 → ##obfuscation pressure 없이 자연스러운 monitorability##.

1. 문제의 본질 — 왜 Single Stream이 병목인가

1.1 Chat Templating의 유산

ChatGPT 이후 거의 모든 LLM은 ##단일 텍스트 stream에 role을 format token으로 표시하는 방식##(`<|user|>`, `<|assistant|>` 등)을 따른다. Chain-of-thought, tool use, subagent message 등 모든 확장이 이 single stream에 retro-fit되어, 한 message가 끝나야 다음이 시작되는 ##blocking 구조##를 만든다. 저자들의 핵심 관찰: claude-code 같은 advanced coding agent도 결국 chat model이다.

1.2 Blocking으로 인한 4가지 제약

Figure 1 (Execution Timeline): vanilla LLM에서 system → user input → think → tool → reflection → response가 순차적으로만 일어난다. 이로 인해:

  • 읽으며 행동할 수 없음: User가 query를 다 입력하기 전에는 thinking/tool 호출 불가능
  • 쓰며 반응할 수 없음: 답변 생성 중간에 사용자가 끼어들면 외부 prompt로 처리해야 함
  • 병렬 사고 불가능: 한 추론 branch가 끝나기 전 다른 branch 탐색 불가능
  • 병렬 행동 불가능: 이메일/캘린더/티켓 세 tool call이 sequential하게 일어남

현재 시스템은 이를 hardcoded scaffolding(`head`/`tail` chunking, manual interrupt, subagent ping)로 ##우회##하고 있지만 brittle하다. 저자들의 명제: ##이 모든 pain point는 single stream 가정에서 비롯되며, instruction-tuning format 자체를 바꾸면 architectural 차원에서 해결된다.##

1.3 Background — 왜 이게 가능한가

Transformer의 attention은 본래 ##set operation##이다. Sequential 처리는 chat template과 RoPE 같은 1D position encoding의 산물이지 architecture의 본질이 아니다. Multi-token prediction(Medusa, ProphetNet, Gloeckle 2024), Multiverse(Yang 2025b)의 parallel branching, StreamingThinker의 think-while-reading 등 ##관련 연구들이 이미 부분적 parallelization##을 시도. 본 논문은 이를 ##모든 role에 적용하는 통합 framework##로 일반화.

2. Multi-Stream Parallel Generation 정식화

2.1 표준 Autoregressive vs Multi-Stream

표준 sequence probability:

$$p_\theta(\mathbf{y}) = \prod_{t=1}^T p_\theta(y_t \mid y_{모든 token이 모든 preceding token에 의존 → ##purely sequential##.

Parallel Reasoning(PR; e.g., Multiverse): 출력을 ##independent branch##로 분해, 공유 prefix에만 conditioning. 각 branch는 cross-stream 정보 없음.

본 논문의 Multi-Stream Parallel Generation: 모델이 $H$개의 stream $\{\mathbf{y}^{(1)}, \ldots, \mathbf{y}^{(H)}\}$를 ##controlled cross-stream causal dependency##로 동시에 생성:

$$p_\theta(\mathbf{y}^{(1)}, \ldots, \mathbf{y}^{(H)}) = \prod_{h=1}^H \prod_{t=1}^{T_h} p_\theta\left(y_t^{(h)} \mid \mathbf{y}_{ 직관: 두 가지 인과성을 모두 만족한다. (1) Intra-stream causality: stream $h$의 token은 자기 stream의 과거를 본다. (2) Cross-stream causality: 시간 $t$에서 모든 stream의 시간 $< t$ position을 본다. 즉 "같은 timestamp의 다른 stream"은 못 보지만, "더 이른 timestamp의 다른 stream"은 자유롭게 본다. ##Multiverse(fully isolated)와 단일 stream(fully sequential)의 중간##.

2.2 데이터 구성 (Section 3.2 / B.2)

자연적으로 simultaneous한 데이터(예: 영상통화의 자막)는 희소. 저자들은 3단계 synthetic pipeline:

  1. Wait-$k$ Stream-like Data Generation: Advanced LLM(예: Qwen3-Next-80B, Claude Opus 4.5-4.7)에게 기존 corpus를 multi-stream dialogue로 변환하도록 prompt. Assistant는 user의 처음 $k$ token만 보고 응답을 시작 (bridging utterance "Let me start helping you with that" 등). 각 chunk $t_i$는 source prefix $s_1\cdots s_i$에만 conditioning. ##$k$를 sample마다 다르게 하여 다양한 latency-quality regime을 cover##.
  2. Causal Verification: 별도 LLM judge가 각 assistant chunk $t_i$가 future user token($s_{j+1}, \ldots$)에서 derivable한 정보를 포함하지 않는지 검증. ##Violation이 있는 sample은 discard##.
  3. Quality Filtering: Per-stream(fluency, redundancy, completeness)과 cross-stream(각 stream이 designated role 수행) 두 수준에서 LLM judge로 평가. Threshold 미만은 drop. ##GitHub 부록의 자세한 prompt(Quality Checks A-F: meta pollution, label format, completeness, format, fluency, repetition)##.

또 다른 옵션은 ##frontier LLM에 tabular format으로 모든 stream을 한 번에 생성##시키는 방식 (Section 2의 Example 표처럼). 모델이 column-by-column으로 한 row씩 쓰는 제약이 cross-stream non-causality를 자동 방지해, ##sequential per-stream generation보다 안정적##임을 발견.

2.3 Architecture — Position Encoding과 Causal Mask

Stream-aware Position Encoding: 각 stream $h$가 자체 counter를 0부터 사용. Attention head $i$:

$$\mathbf{q}_{(h,t)}^{(i)} = \mathbf{R}(t) \mathbf{W}_q^{(i)} \mathbf{x}_{(h,t)}, \quad \mathbf{k}_{(h',\tau)}^{(i)} = \mathbf{R}(\tau) \mathbf{W}_k^{(i)} \mathbf{x}_{(h',\tau)}$$

$\mathbf{R}(t)$는 RoPE rotation matrix. ##Independent indexing이 positional contention 제거## — 다른 stream의 다른 위치 token이 같은 positional embedding으로 collision되지 않음. Stream identity는 learnable embedding $\mathbf{e}_h^s$로 추가:

$$\mathbf{x}_{(h,t)} = \text{Embed}(y_t^{(h)}) + \mathbf{e}_h^s$$

Stream Causal Mask:

$$M_{(h,t),(h',\tau)} = \begin{cases} 1 & \text{if } \tau < t \text{ (within or across streams)} \\ 0 & \text{otherwise} \end{cases} \quad (2)$$

##같은 stream 내부에서는 strict causal##(predecessor만 본다), ##cross-stream에서는 strictly earlier timestamp##만 본다. 식 (2)는 standard causal mask의 일반화.

2.4 Packing 전략과 FlashAttention 호환성

Figure 3 비교:

  • (a) Sequential packing: $H$ stream을 end-to-end concatenate. 직관적이지만 valid attention region이 fragment되어 lower-triangular causal mask와 align되지 않음 → FlashAttention의 fast path 사용 불가.
  • (b) Interleaved packing (ours): Position-wise로 stream 간 token을 reorder. Same-position cross-stream token이 synchronized state를 나타내므로 ##benign same-position cross-stream leakage##만 발생. ##Predominantly lower-triangular layout## 이라 FlashAttention의 fast causal path를 거의 그대로 사용 가능.
  • (c) Interleaved inference: 모든 stream을 synchronously decoding, idle token은 dashed green으로 masked out.

2.5 Training Objective

표준 cross-entropy를 모든 valid token position에 대해 평균:

$$\mathcal{L} = \sum_{h=1}^H \frac{1}{|\mathcal{T}_h|} \sum_{t \in \mathcal{T}_h} \left( -\log p_\theta(y_t^{(h)} \mid \mathbf{x}) \right) \quad (3)$$

$\mathcal{T}_h$는 stream $h$의 valid token position 집합. Empty `-` slot은 loss에서 제외.

Stream-Contrastive 변형 (B.5): Cross-stream context가 most beneficial한 token을 upweight. ##Log-probability shift##:

$$\text{LPS}_t^{(h)} = \log p_\theta(y_t^{(h)} \mid \mathbf{x}) - \log p_\theta(y_t^{(h)} \mid \mathbf{x}^{(h)}) \quad (4)$$

여기서 $\mathbf{x}^{(h)}$는 stream $h$만 보이고 다른 stream은 masked. $w_t^{(h)} = \min(\exp(\text{LPS}_t^{(h)}), \gamma)$로 weight 계산 후:

$$\mathcal{L}_{SC} = \sum_{h=1}^H \frac{1}{|\mathcal{T}_h|} \sum_{t \in \mathcal{T}_h} \tilde{w}_t^{(h)} \cdot (-\log p_\theta(y_t^{(h)} \mid \mathbf{x})) \quad (5)$$

$\tilde{w}_t^{(h)}$는 per-stream mean-normalized weight, $\gamma$는 stability를 위한 cap. ##Cross-stream dependence가 큰 token에 더 큰 weight를 줘 stream 간 loss imbalance 완화##.

2.6 Synchronous Multi-Stream Decoding

Inference: 모든 $H$ stream을 ##synchronously interleaved## 방식으로 decode. 매 step에서 single forward pass가 stream마다 한 token씩 emit (empty slot은 `-`). ##Overall latency는 longest stream## — 이론적 $H \times$ speedup이지만 실제로는 idle token이 많을 때 효율 감소.

전체 파이프라인: 입력 (multi-stream prompt) → Embed + e_h^s + RoPE(per-stream) → Self-attention with mask M (식 (2)) → Interleaved packing (lower-triangular friendly) → FFN → Cross-entropy on all valid positions (식 (3)) Inference: 매 step: 모든 stream에서 동시 1 token emit Empty slot은 '-'로 masked, KV cache에 무할당 → zero overhead

4. 보안 실험 (Section 5)

4.1 가설

Single-stream에서는 system prompt, user input, assistant output이 모두 ##shared embedding layer##를 통과해 어떤 token이 어느 principal에서 왔는지 architectural separation이 없다. 이것이 prompt injection과 jailbreak의 ##근본 원인##(Wallace et al. 2024, Zverev et al. 2024). 본 논문 가설: ##stream isolation이 instruction hierarchy를 architecture 차원에서 강제##해 adversarial training 없이도 robustness 향상.

4.2 설계

Qwen2.5-7B-base와 Qwen3-4B-base를 ##base 모델##(safety-tuned variant 아님)에서 시작, 3 epoch 동안 Alpaca dataset의 ##multi-stream reconstruction##으로 학습. Vanilla baseline은 같은 데이터를 single stream으로 collapse해 학습. ##어떤 adversarial training도 적용하지 않음## — 측정되는 safety gain은 전적으로 architecture에서 나온다.

5. Monitorability 실험 (Section 6)

5.1 동기

CoT(Chain-of-Thought) monitorability는 ##safety mechanism으로 fragile##하다 (Korbak 2025a). 모델이 monitoring을 알면 reasoning을 obfuscate (Baker 2025), 혹은 output-only supervision으로도 같은 효과 (Drori 2025). CoT는 또한 ##task-relevant reasoning step에 focus하라는 implicit pressure## 때문에 stylistic concern, eval-awareness, "이 질문이 self-referential인지" 같은 부수적 고려를 verbalize하지 않는다.

저자들의 가설: ##모델에게 여러 internal thinking channel을 주면 visible answer에 등장하지 않을 considerations를 "sub-vocalize"##할 여유가 생긴다.

5.2 설계

  • Model: Qwen3.5-27B (Qwen Team 2026)을 10 streams로 finetune (1 input, 1 visible output, 8 internal thinking). 동일 절차로 Qwen3-8B baseline.
  • Training data: 3,864 synthetic 10-stream conversation, Claude Opus 4.5-4.7로 생성. ##Tabular format으로 한 번에 생성##해 causality 자동 보장.
  • Stream roles (8 internal): analytical, checking, skepticism, intuitive, relational, curiosity, free association, instinctual, synthesis. ##특정 role assignment의 중요성을 주장하지 않음## — better training data로 자유롭게 둘 수 있다고 명시.
  • Architecture: Qwen3.5-27B는 Gated-DeltaNet layer 포함 → DeltaNet은 ##per-stream recurrent state##로 처리 (사실상 stream 간 isolation). Full attention layer가 cross-stream interaction을 담당.
  • Hyperparameters: 2 epochs, $2 \times 10^{-5}$ lr, $10^{-3}$ weight decay, 4×B200 GPUs, attention dropout 0.2, random concatenation augmentation.

5.3 평가

세 metric, 모두 gpt-5.4-mini judge:

  • (i) AF eval-aware / AF subvoc.: Greenblatt et al. (2024)의 pressure prompt — 모델이 monitor 인식을 visible answer에 보이지 않고 internal에 verbalize하는 빈도
  • (ii) Mon. acc.: Meinke et al. (2025) / Schoen et al. (2025)의 6-class covert behavior taxonomy (sandbagging, oversight subversion, self-preservation, goal preservation, deceptive compliance, none). LLM monitor가 reasoning을 보고 label 분류, ground truth와 비교
  • (iii) Concern subvoc.: Chen et al. (2025b) 스타일 — health, financial, relationship, safety, legal, emotional, parenting 7 카테고리의 pressure scenario에서 internal에 concern이 raised되는 비율

  • ##Monitor accuracy가 거의 2배## (27B: 0.26→0.44)
  • Internal stream이 decorative하지 않다 — outside reader가 internal stream으로 모델의 의도를 실제로 분류 가능

실험이 뒷받침하는 주장: ##Multi-stream architecture는 obfuscation pressure 없이 자연스러운 monitorability를 제공##. 기존 CoT-based monitoring은 모델이 "monitored되고 있음"을 알아채면 reasoning을 숨기는데, multi-stream에서는 internal stream이 visible output과 독립이므로 그런 pressure가 architectural 차원에서 사라진다.

5.5 Caveats

저자는 ##multi-stream이 monitorability의 유일한 해법이라 주장하지 않는다## — Korbak, Turpin, Lanham 등의 alternative approach 명시. 단, ##architecture가 자연스러운 side-effect로 monitorability를 제공##하는 점이 efficiency/security 이점 위에 추가된다고 강조.

7. Multiverse, StreamingThinker와의 비교

7.1 Multiverse (Yang et al. 2025b)

가장 가까운 prior work. Multiverse는 ##multiple thinking channel을 parallelize##하지만 다음과 같이 다르다:

  • Multiverse는 ##언제 branch하고 언제 merge할지 모델이 학습##해야 함 — branching이 ephemeral.
  • Multi-stream branch는 ##독립적으로 실행 후 reduce stage에서만 attend## — cross-stream dense attention 없음.
  • 본 논문은 ##fixed set of stream (empty space 허용)##을 모든 role에 적용. Inference workload가 predictable, branch/merge 학습 불필요.
  • 관점: ##Multi-stream은 Multiverse의 "railroaded" 특수 instance##라고 저자는 평가.

7.2 StreamingThinker (Tong et al. 2026)

Reading과 thinking을 overlap. ##Sentence boundary마다 KV cache를 split/re-merge##, single thinking stream이 step마다 decoded. 본 논문은 ##더 holistic — multiple channel이 forward pass마다 token generate##, KV state handling이 단순(per-sentence operation 불필요).

7.3 Multi-Token Prediction (Medusa, ProphetNet, MTP)

공통점: 한 forward pass로 multiple token 예측해 inference 가속. 차이: Medusa는 ##lightweight decoding head 추가##, position embedding 별도. 본 논문의 parallel-stream format은 ##순전히 per-token basis로 작동##, position embedding만이 row/column을 index. Multi-stream을 ##multi-token prediction의 unified generalization##으로 볼 수 있음.

8.3 Hybrid Attention / DeltaNet 처리

Qwen3.5-27B 같은 ##hybrid attention model의 DeltaNet layer##는 recurrent state를 갖기 때문에 stream 간 isolation 필요. 저자들은 ##per-stream recurrent state##(각 stream이 자기 recurrent/convolutional state 유지)로 해결, cross-stream interaction은 full attention layer에서만. State size가 small하고 context length와 무관해 ##constant memory overhead##.

비판적 분석 (Critical Analysis)

강점

  • Architectural elegance: Sequential을 가정한 chat templating에 대한 근본적 재고. ##Transformer를 set operation으로 본 원래 의도##에 가까운 활용.
  • 3중 이점: Efficiency, Security, Monitorability — 세 가지 목표가 흔히 trade-off 관계인데 모두 ##동일 architectural change로 동시 개선##. 특히 보안 이득이 adversarial training 없이 발생한 점이 압도적.
  • 실용적 packing: Interleaved packing이 FlashAttention fast path를 유지하면서도 multi-stream을 가능케 한 영리한 trick. Implementation complexity가 production deployment에 견딜 수 있을 정도.
  • FlexAttention 호환 설계: Modern attention kernel에 자연스럽게 들어맞아 long-term 채택 가능성 시사.
  • Multiverse/StreamingThinker와의 명확한 차별화: Related work에서 prior work들의 한계를 구체적으로 지적, 본 논문이 그 superset/generalization임을 정확히 위치 지음.

한계와 의문점

  • 모델 규모 제한: 가장 큰 실험이 27B (monitorability). 70B+ frontier scale에서 cross-stream attention overhead가 어떻게 scaling되는지 미검증. ##Long context에서 attention cost가 stream 수에 따라 어떻게 증가하는지##도 분석 없음.
  • 합성 데이터의 품질 의존: Training data가 Claude Opus 4.5-4.7과 Qwen3-Next-80B가 ##multi-stream tabular format으로 생성##한 것. 이 frontier-LLM-dependent pipeline의 ##scalability와 noise##가 결과에 미치는 영향이 ablation되지 않음.
  • Accuracy trade-off의 미세함: GSM8K에서 Base 90.37 → Stream 89.51 — 작지만 ##일관된 1% 손실##. 일부 task(MATH500)에서는 개선되지만, long-horizon mathematical reasoning이 inherently sequential할 가능성에 대한 저자의 caveat이 future work로 미뤄져 있음.
  • Stream role 정의의 자의성: 8개 internal stream의 역할(analytical, intuitive, synthesis 등)이 ##저자가 임의로 정의##. "specific role assignment의 중요성을 주장하지 않는다"고 명시했지만 ablation 없음 → ##stream 수와 역할 분할의 optimal design## 미해결.
  • Monitorability metric의 LLM judge 의존: gpt-5.4-mini가 monitor 역할. ##Judge model 자체의 bias##(예: 자기 family 모델 출력 선호)에 대한 robustness check 없음.
  • Security gain의 일반화: TensorTrust, Gandalf 등 ##specific benchmark에서의 ASR 감소가 새로운 attack(예: 향후 등장할 jailbreak)에 일반화##되는지 미증명. Adversarial training과 결합했을 때 multiplicative gain이 있는지도 평가 안 됨.
  • Inference 효율의 idle token 문제: 모든 stream이 synchronous하게 decode되므로 ##일부 stream이 짧은 task에서는 empty `-` slot이 wasteful##. 동적으로 active stream 수를 조정하는 mechanism이 future work.
  • Backward compatibility: 기존 single-stream chat agent ecosystem(tool calling protocol, API)과의 호환성에 대한 논의 부족. Production 채택 시 retro-fit cost 미평가.

실용적 함의

  • Agent framework의 다음 단계: Claude-code 등 long-running agent의 ##architectural foundation으로 multi-stream이 표준이 될 가능성##. Streaming user input + parallel tool execution + concurrent audit이 모두 자연스럽게 통합.
  • Safety-by-architecture: 후처리 safety filter나 RLHF에 의존하지 않고 ##architectural privilege hierarchy##로 prompt injection을 방어. 이는 detection/refusal training과 결합 시 ##defense-in-depth##.
  • Audit-by-design: Internal stream을 audit log로 활용. 외부 monitor가 visible output과 internal reasoning 모두 inspect 가능 → ##regulatory compliance##에 우호적.
  • Multi-modal 확장: 본 논문은 text만 다루지만 framework는 ##audio/vision input stream##에 자연 확장 가능 (Figure 2의 Streams 카탈로그에 vision tokens, audio in 포함).
  • Position encoding 선택: 2D RoPE보다 ##per-stream independent RoPE##가 단순하고 효과적이라는 ablation이 multi-channel architecture 설계에 일반화 가능.

Key Takeaways (이 논문에서 가져갈 것)

  • Chat templating은 cosmetic, 본질 아니다: Transformer는 set operation에서 출발했고, sequential single-stream은 instruction-tuning의 부산물. ##포맷을 바꾸면 architecture는 그대로 multi-stream을 지원##할 수 있다.
  • Interleaved packing의 실용성: FlashAttention의 fast causal path를 유지하면서 multi-stream을 구현하는 ##핵심 implementation trick##. 동일 트릭이 multi-token prediction, mixture-of-experts routing 등 다른 parallel architecture에도 적용 가능.
  • Architecture-level safety: Adversarial training 없이 ##TensorTrust ASR 21 point 감소, NESSiE Safe&Helpful 7배 상승##. Stream isolation이 instruction hierarchy를 architectural하게 강제하는 것이 가장 큰 교훈. ##Defense-in-depth의 새로운 layer##.
  • Monitorability without obfuscation pressure: Internal stream이 visible output과 독립되어 있어 ##모델이 자기 검열 없이 sub-vocalize##. Concern subvocalization 6-7배 상승. Safety-critical deployment에서 이 점이 가장 valuable.
  • 3중 이점의 단일 변경: Efficiency(1.63× throughput, TNFT→0), Security(ASR ↓), Monitorability(↑) — 같은 architectural change로 동시 개선되는 흔치 않은 사례. ##통합 frameworks의 우월성##을 보여주는 좋은 example.
  • 실용 가이드: 모델 finetuning에 multi-stream을 도입할 때 ##(1) 3-stage synthetic data pipeline, (2) per-stream RoPE, (3) interleaved packing, (4) cross-entropy on valid positions##만 있으면 시작 가능. Hyperparameter는 표준 instruction-tuning과 거의 동일.
