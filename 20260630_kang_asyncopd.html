TL;DR

  • 문제: On-Policy Distillation(OPD)은 LLM post-training의 핵심 기법이지만, ##strict synchronous pipeline에서는 rollout 생성이 학습 시간을 지배##해 학습 GPU가 idle 상태가 된다. 비동기화하면 stale-policy 데이터 문제가 발생.
  • 해결: 비동기 OPD에서 KL direction·policy-gradient surrogate·teacher-cache 세 축을 체계적으로 분석. ##current student로 advantage $A_\theta$를 recompute하고 clipping 없는 reverse-KL OPD/IS surrogate + multi-sample Monte Carlo cache##가 stale에 가장 robust.
  • 왜 작동하는가: forward-KL은 teacher-weighted라 stale rollout에 robust한 반면, reverse-KL은 student-weighted라 vulnerable. 그러나 reverse-KL에서 ##advantage를 stop-gradient로 freeze하지 않고 학습 시점 student로 재계산##하면 token-level signal이 회복된다. teacher-score cache의 support-mismatch는 multi-sample MC로 분산을 줄여 해결.
  • 비유: 교사가 매번 학생 답을 받자마자 채점하면 동기적(synchronous) 학습. 대신 ##교사가 어제의 학생 답을 오늘 채점##해도 (1) 채점 기준이 학생의 어제 답에 의존하지 않게 하고 (2) 어제 학생이 한 답 외의 가능성도 여러 개 미리 채점해 두면 학습 효율은 1.6~3.8배 빨라지면서도 품질은 유지.

핵심 기여 (Key Contributions)

  1. 비동기 OPD에서 staleness 영향의 첫 체계적 연구: prefix-level/action-level staleness를 구분하고, ##teacher-cache constraint(full-vocabulary logit을 caching·transfer하기 너무 비싸다)## 하에서 estimator design을 분석.
  2. KL direction이 stale-data 문제를 결정한다는 발견 (Finding 1): forward KL은 teacher-weighted라 stale rollout에 robust하지만 ##reverse KL은 student-weighted라 vulnerable##. cached teacher score가 current student의 action을 cover하지 못하면서 발생하는 support-ownership 차이.
  3. reverse-KL surrogate 비교 (Finding 2): PPO-style clipping이나 advanced async-RL surrogate (Decoupled PPO, M2PO)를 시도해도 ##단순 OPD-specific surrogate (learner-time $A_\theta$ recompute + no clipping)##보다 낫지 않음. clipping은 오히려 유용한 signal을 제거.
  4. multi-sample Monte Carlo cache (Finding 3): stale top-$k$ support는 support-mismatch로 bias가 있지만, one-sample MC는 importance sampling으로 correctable하되 high-variance. ##각 prefix에서 multiple local sample을 caching##해 MC variance를 줄이면 large staleness에서도 robust.
  5. AsyncOPD 시스템과 오픈소스: 완전 비동기 OPD 파이프라인. Qwen3-{1.7B,4B,8B}-Base에서 ##strict sync 대비 1.6~3.8배 throughput 향상##하면서 AIME24/25, AMC accuracy를 유지.

배경 지식

On-Policy Distillation (OPD)이란?

OPD [Agarwal et al. 2024 (GKD), MiniLLM (Gu et al.), TIP, G-OPD]는 학생이 자기 자신의 rollout을 만든 뒤, teacher가 그 prefix에 대해 dense token-level feedback을 주는 post-training 방식이다. RL과 다르게 ##reward가 아니라 teacher의 분포 자체가 supervision##이라 small student model의 학습에 효율적이다. 최근에는 self-distillation, multi-teacher distillation으로도 확장.

Synchronous OPD의 병목

strict synchronous 파이프라인은 다음 순서로 진행:

  1. Stage 1: student rollout — 학생 정책 $p_\theta$로 trajectory 생성
  2. Stage 2: teacher scoring — teacher $q$가 visited prefix에 대해 logit 계산
  3. Stage 3: learner update — student가 KL-loss로 update

각 update는 새 rollout을 기다리므로 ##학습 GPU가 rollout이 끝날 때까지 idle##. reasoning task는 rollout이 길어 이 병목이 더 심각.

Asynchronous RL의 해결책

Async RL (AReaL, Decoupled PPO, M2PO 등)은 rollout worker가 ##이전 rollout을 학생이 학습하는 동안 새 rollout을 생성##하도록 stage를 decouple한다. 이는 stale-policy data를 만든다 — rollout은 $p_{\text{old}}$에서 왔지만 update는 $p_\theta$가 받는다.

OPD의 추가 제약: Teacher-Cache Constraint

RL은 reward가 scalar라 cache가 쉽지만, OPD는 ##full-vocabulary teacher logit이 필요##하다 (KL 계산용). 이를 모든 prefix·모든 token에 대해 caching·transmitting하는 것은 비현실적. 따라서 실제 구현은:

  • Sparse top-$k$: 각 prefix에서 top-$k$ token만 cache
  • Monte Carlo (MC): rollout 시점 분포에서 몇 개 action만 sample해서 cache

비동기에서는 cache가 $p_{\text{old}}$ 시점에 만들어지지만, learner는 현재 $p_\theta$에서 update해야 하므로 ##support mismatch## (cache에 없는 action을 update해야 함) 문제가 발생.

방법론 1 — OPD Preliminaries (수식 정리)

Forward-KL OPD (식 1)

fixed prefix $s$에서 forward-KL은 teacher-weighted:

$$D_F(\theta; s) = \mathrm{KL}(q(\cdot|s) \| p_\theta(\cdot|s)) = \sum_{a \in \mathcal{V}} q(a|s) (\log q(a|s) - \log p_\theta(a|s))$$

gradient는:

$$\nabla_\theta D_F(\theta; s) = -\sum_{a \in \mathcal{V}} q(a|s) \nabla_\theta \log p_\theta(a|s)$$
직관적으로 말하면, "##teacher 분포로 가중 평균##된 log-likelihood gradient"라 student가 어떤 action을 sampling했든 teacher의 $q$가 weight를 결정한다.

Reverse-KL OPD (식 2-3)

같은 prefix에서 reverse-KL은 student-weighted:

$$D_R(\theta; s) = \mathrm{KL}(p_\theta(\cdot|s) \| q(\cdot|s)) = -\sum_{a \in \mathcal{V}} p_\theta(a|s) (\log q(a|s) - \log p_\theta(a|s))$$

미분하면 (식 3):

$$\nabla_\theta D_R(\theta; s) = -\sum_{a \in \mathcal{V}} p_\theta(a|s) \underbrace{(\log q(a|s) - \log p_\theta(a|s))}_{A(a,s) = \log q - \log p} \nabla_\theta \log p_\theta(a|s)$$

이 형태는 ##policy-gradient estimator##와 동일 — $A(a,s) = \log q(a|s) - \log p_\theta(a|s)$가 token-level advantage 역할.

PPO-style Surrogate (식 4)

behavior policy $p_{\text{beh}}$에서 sample $a \sim p_{\text{beh}}$, ratio $\rho_\theta(a,s) = p_\theta(a|s) / p_{\text{beh}}(a|s)$, clipped ratio $\bar{\rho}_\theta = \mathrm{clip}(\rho_\theta, 1-\epsilon, 1+\epsilon)$:

$$L_{\mathrm{PPO}}(\theta; A_{\mathrm{beh}}) = -\mathbb{E}_{a \sim p_{\mathrm{beh}}} [\min(\rho_\theta \mathrm{sg}(A_{\mathrm{beh}}), \bar{\rho}_\theta \mathrm{sg}(A_{\mathrm{beh}}))]$$

$\mathrm{sg}(\cdot)$는 stop-gradient. behavior-time signal $A_{\mathrm{beh}}$는 frozen.

Sparse Top-$k$ / Monte Carlo Implementations

실용적 OPD는 finite support 위에서 KL 계산:

  • Sparse top-$k$: support $S(s)$ 선택 후 renormalized KL — 식 (8)-(10) (Appendix A.1): $$S_\theta(s) = \mathrm{TopK}(p_\theta(\cdot|s), k), \quad \tilde{p}_\theta^S(a|s) = \frac{p_\theta(a|s) \mathbf{1}[a \in S]}{Z_p^S(s)}$$
  • MC reverse-KL: $A_\theta(a, s) = \log q(a|s) - \log p_\theta(a|s)$로 one-sample (식 12) 또는 multi-sample (식 13): $$\hat{g}_m(s) = -\frac{1}{m} \sum_{i=1}^m A_\theta(a_i, s) \nabla_\theta \log p_\theta(a_i|s)$$

방법론 2 — Asynchronous OPD Setup

3-Stage Cached Pipeline (Fig. 1)

[Stage 1: Student Rollout] (at p_old, time t_roll) └─ Generate trajectories, cache prefixes s and actions C_old(s) │ ▼ [Stage 2: Teacher Scoring] └─ teacher q scores cached actions C_score(s) │ ▼ [Stage 3: Student Update] (at p_θ, time t_train > t_roll) └─ Compute current local OPD gradient • Can recompute log p_θ(a|s) for a ∈ C_score(s) • Cannot recover teacher logits for missing actions Staleness Δ_batch = t_train - t_roll (rollout이 얼마나 오래된 student snapshot에서 왔는가)

Teacher-Cache Constraint의 핵심

full-vocabulary teacher logit caching·transfer는 ##async pipeline에서 매우 비싸므로## 실용 구현은 sparse top-$k$ 또는 MC sample로 제한. 비동기일 때 두 가지 추가 staleness 발생:

  • prefix-level staleness: 학습 시점의 $p_\theta$가 만들었을 prefix와 다른 prefix가 cache에 있음
  • action-level staleness: cache의 action은 $p_{\text{old}}$에서 sample됐지만, current $p_\theta$의 high-mass action과 다를 수 있음

저자는 ##action-level staleness가 estimator design으로 직접 다룰 수 있는 영역##이라 명시.

핵심 발견 1 — Forward vs Reverse KL Under Staleness (Section 4)

실험 의도

같은 stale cache에 forward KL과 reverse KL을 적용하면 어떤 차이가 있는가? KL direction이 staleness 영향을 결정하는가?

실험 결과 (Fig. 2)

Qwen3-4B-Base 학생 + Qwen3-30B-A3B-Instruct teacher, DeepMath 57,630 problems로 학습.

패턴 분석: ##Reverse KL은 zero-staleness에서 더 높지만, staleness가 커지면 더 빠르게 떨어지고 결국 forward KL에 추월##됨. 이유는 forward KL이 teacher-weighted라 stale student action sample을 current-student expectation으로 convert할 필요가 없고, reverse KL은 student-weighted라 cached support 밖의 action에 의존하기 때문.

Finding 1. Forward KL is teacher-weighted and robust to rollout staleness, whereas reverse KL is student-weighted and vulnerable to rollout staleness.

저자는 이후 분석을 reverse-KL OPD에 집중 (실용 OPD에서 더 흔하고 더 vulnerable하기 때문).

핵심 발견 2 — Reverse-KL Policy-Gradient Surrogates (Section 5)

실험 의도

reverse-KL의 stale-data 문제를 ##async-RL에서 stale을 안정화하기 위해 제안된 기법들(Decoupled PPO, M2PO)##이 해결할 수 있을까? 아니면 OPD-specific한 다른 surrogate가 더 나은가?

네 가지 surrogate 비교 (식 6-7)

2×2 ablation: advantage를 $A_{\text{old}}$ (rollout 시점) vs $A_\theta$ (learner 시점) 사용 × clipping 유무.

$$L_{\text{old}}^{\text{clip}}(\theta) = -\mathbb{E}_{a \sim p_{\text{old}}} [\min(\rho_\theta \mathrm{sg}(A_{\text{old}}), \bar{\rho}_\theta \mathrm{sg}(A_{\text{old}}))]$$ $$L_{\text{old}}^{\text{noclip}}(\theta) = -\mathbb{E}_{a \sim p_{\text{old}}} [\rho_\theta \mathrm{sg}(A_{\text{old}})]$$ $$L_\theta^{\text{clip}}(\theta) = -\mathbb{E}_{a \sim p_{\text{old}}} [\min(\rho_\theta \mathrm{sg}(A_\theta), \bar{\rho}_\theta \mathrm{sg}(A_\theta))]$$ $$L_\theta^{\text{noclip}}(\theta) = -\mathbb{E}_{a \sim p_{\text{old}}} [\rho_\theta \mathrm{sg}(A_\theta)]$$

$L_\theta^{\text{noclip}}$이 ##OPD/IS identity## — exact old-to-current importance-sampling identity (식 5):

$$D_R(\theta; s) = -\mathbb{E}_{a \sim p_\theta}[A_\theta(a,s)] = -\mathbb{E}_{a \sim p_{\text{old}}}[\rho_\theta(a,s) A_\theta(a,s)]$$

핵심 발견 3 — Cached Supports Under Staleness (Section 6)

두 가지 cached-support estimator 비교

Sparse top-$k$: stale-support biased

cache는 rollout-time support $S_{\text{old}}(s) = \mathrm{TopK}(p_{\text{old}}(\cdot|s), k)$ 위에서 만들어지지만, current student의 support $S_\theta(s) = \mathrm{TopK}(p_\theta(\cdot|s), k)$와 다를 수 있다. ##stale top-$k$ support 내에서 reweighting해도 missing teacher score를 복구할 수 없다## — fixed-support mismatch.

One-sample MC: correctable but high-variance

sample $a \sim p_{\text{old}}(\cdot|s)$ + $\log p_{\text{old}}(a|s)$ cache → IS estimator가 ##expectation에서 exact##. 하지만 single sample은 high variance.

제안: Multi-sample MC (Fig. 8)

각 prefix $s_t$에서 ##multiple local next-token sample## $a_{t,1}, \ldots, a_{t,m} \sim p_{\text{old}}(\cdot|s_t)$를 함께 cache하고 average. learner-time에서 IS-corrected gradient (식 15):

$$\hat{L}_m^{\text{MC}}(\theta; s_t) = -\frac{1}{m} \sum_{i=1}^m \rho_\theta(a_{t,i}, s_t) \mathrm{sg}(A_\theta(a_{t,i}, s_t)), \quad \rho_\theta(a,s) = \frac{p_\theta(a|s)}{p_{\text{old}}(a|s)}$$

$m$ samples를 averaging해 Monte Carlo variance를 줄임. ##async에서는 prefix를 multiple action으로 branch하는 RL 비용이 크지만, OPD는 local next-token sample만 추가하면 되어 저렴##.

실험 결과 (Fig. 6, Fig. 7, Table 1b, Table 6)

Sparse top-$k$ vs one-sample MC (Fig. 6): one-sample MC + IS가 sparse top-$k$ + reweighting보다 staleness 증가 시 robust. reweighting이 sparse top-$k$의 missing-support 문제를 해결할 수 없음을 입증.

multi-sample MC (Fig. 7, Table 1b): $m \in \{4, 16, 64\}$ 비교 — $m = 4$가 이미 large staleness에서 명확한 jump를 만들며, $m=4, 16, 64$는 큰 차이 없음.

AsyncOPD 시스템 (Section 7)

Scheduler 비교 (Fig. 9)

  • Synchronous: 모든 stage가 barriered. 학습 GPU idle.
  • Step-off (VeRL [19]): $k$-step-off로 rollout lag을 고정, 그러나 ##complete rollout batch를 기다림##.
  • AsyncOPD: completed items만 teacher scoring으로 consume, scored batch가 ready되면 learner update — ##in-flight prefix는 유지##하면서 weight sync 시점에만 worker 일시정지.

실험 설정 (Table 3 — Experimental Settings)

  • Student: Qwen3-{1.7B, 4B, 8B}-Base (thinking-disabled)
  • Teacher: Qwen3-30B-A3B-Instruct-2507
  • Training data: DeepMath 57,630 problems (difficulty ≥ 6)
  • Prompt/response: 2,048 / 16,384 tokens
  • 200 training iterations, batch 256, mini-batch 64 (4 optimizer updates per step)
  • Optimizer: AdamW, lr $3 \times 10^{-6}$, weight decay 0.01
  • Rollout: temperature 1.0, top-$p$ 1.0, no top-$k$ truncation
  • Hardware: 8×B200 single node, vLLM rollout/scoring, PyTorch FSDP
  • 비교: strict sync vs two-step-off vs AsyncOPD ($\tau = 4$)

강점

  • 분석의 깊이: KL direction → policy-gradient surrogate → cache estimator의 세 축을 ##독립 변수로 분리##해 ablation하는 방식이 인상적. 각 finding이 명확한 mechanism을 통해 설명됨.
  • 이론과 실증의 연결: OPD/IS identity (식 5)가 ##exact##임을 보이고 (importance sampling을 통한 변환), 이를 실험적으로 $A_\theta$ no-clip의 우월성으로 검증.
  • 실용성: AsyncOPD는 단순 이론 분석이 아니라 ##8×B200 단일 노드에서 작동하는 시스템##. Qwen3-{1.7B, 4B, 8B}로 scale variance 검증.
  • 오픈소스: GitHub 코드 공개 (https://github.com/furiosa-ai/async-opd).
  • Async-RL 기법의 한계 드러내기: Decoupled PPO, M2PO 같은 ##async-RL stable이 OPD에 그대로 transfer되지 않음##을 실증. 이는 OPD-specific 연구의 가치를 분명히 함.

약점·의문

  • Math domain 단일 평가: AIME24/25, AMC만으로 평가. ##reasoning(코딩), instruction following, multi-turn agent## 등 다른 도메인에서도 같은 패턴이 유지될지 불명.
  • Single 8-GPU node 한계: 저자도 limitation으로 인정. multi-node distributed environment에서 ##network bandwidth가 teacher logit transfer를 더 제약##할 수 있어, dense KL 가능성이 변동.
  • Teacher size 고정: Qwen3-30B-A3B-Instruct 하나만 사용. ##teacher-student size gap이 작을 때 (예: self-distillation)##나 더 클 때 staleness 영향이 달라질 가능성.
  • Dense KL과의 비교 부재: dense full-vocabulary KL은 cache mismatch가 없지만 ##저자가 "scale 어려움" 이유로 분석에서 제외##. 만약 KDFlow [29] 같은 방식으로 dense KL을 async에 적용한다면 multi-sample MC 대비 더 나을 수도 있음.
  • $\tau = 4$ 고정: AsyncOPD experiment에서 staleness $\tau = 4$만 사용. 더 aggressive async (예: $\tau = 16, 32$)에서도 accuracy 유지 가능한지 알 수 없음 — Table 2의 추세를 보면 1.7B MC1에서 ##sync 8.65 vs async 8.44##로 약간 떨어지는 신호가 있음.
  • Multi-sample MC overhead: $m=4$ 이상에서 rollout-time forward pass cost가 늘어남. paper는 "local next-token sample은 저렴"이라 주장하지만 ##정확한 wall-clock overhead 측정은 부재##.
  • MC64 vs MC1 비교 모호: Table 2에서 MC64가 모든 throughput에서 MC1보다 좋지 않은 경우가 있음 (1.7B에서 MC1이 더 빠름). 이는 multi-sample MC의 cost가 accuracy 향상보다 크다는 신호일 수 있는데, 이에 대한 직접 분석은 없음.

한계의 자체 인정

저자는 conclusion에서 명시:

  • Dense full-vocabulary KL의 async setting은 다루지 않음 — KDFlow [29]가 hidden state transfer로 sync OPD에서 가능성을 보였지만, async로 확장은 future work.
  • Single 8-GPU node에 국한 — multi-node cluster scaling은 future work.

Key Takeaways (이 논문에서 가져갈 것)

  1. 비동기 OPD를 구현한다면 reverse-KL이 더 risky하다: forward-KL이 stale에 robust하므로 보수적 선택. 하지만 reverse-KL을 쓰고 싶다면 다음 처방을 따른다.
  2. reverse-KL OPD의 처방전: $A_\theta$ recompute + no clipping: rollout 시점 advantage $A_{\text{old}}$를 freeze하지 말고 ##학습 시점 $p_\theta$로 advantage를 다시 계산##. clipping은 오히려 유용한 signal을 제거.
  3. Sparse top-$k$ cache는 stale support mismatch가 본질적 약점: ##stale support 안에서 reweight해도 missing teacher score는 복구되지 않음##. one-sample MC + IS가 expectation에서 exact correctable.
  4. Multi-sample MC의 가성비: $m=4$만으로도 large-staleness 안정성에 큰 향상. ##async OPD에서 prefix branching이 RL보다 저렴##하기 때문에 multi-sample이 자연스러운 선택.
  5. Async-RL 기법은 OPD에 그대로 적용되지 않는다: Decoupled PPO, M2PO 등은 OPD의 teacher-cache constraint와 manage differently. ##OPD-specific surrogate가 필요##.
  6. 1.6×–3.8× throughput은 실용적으로 큰 의미: reasoning task OPD가 production-scale에서 점점 중요해지는 만큼, ##단일 노드 8 GPU에서 2배+ 향상##은 cost-effective.
  7. OPD/IS identity: $\mathbb{E}_{a \sim p_\theta}[A_\theta(a,s)] = \mathbb{E}_{a \sim p_{\text{old}}}[\rho_\theta(a,s) A_\theta(a,s)]$ — 이 exact identity가 unclipped IS surrogate의 이론적 정당성. RL의 PPO-style clipping과 다른 OPD만의 처방.
