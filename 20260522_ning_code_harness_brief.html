Code as Agent Harness — 코드가 AI 에이전트의 '작업대'가 된다

AI 에이전트가 똑똑해지는 진짜 비결은 '더 좋은 두뇌'가 아니라 '더 좋은 작업 환경'이다. 이 서베이는 코드를 그 작업 환경(harness)으로 재해석하고, 흩어진 수백 편의 연구를 하나의 지도로 묶는다.

먼저 알아야 할 배경 지식

배경 1 — "LLM"과 "AI 에이전트"는 다르다

ChatGPT 같은 LLM(거대 언어 모델)은 본질적으로 "질문을 받으면 한 번 답하고 끝나는" 모델이다. 기억도 없고, 직접 무언가를 실행하지도 못한다. 반면 AI 에이전트는 며칠에 걸친 작업을 스스로 수행한다 — 코드를 짜고, 실행하고, 에러를 보고, 고치고, 또 실행한다. 이 차이를 만드는 것이 무엇일까?

LLM은 두뇌, harness는 그 두뇌가 실제로 일하게 만드는 손·발·기억·작업대다.

배경 2 — "Harness(하네스)"란?

Harness는 원래 말이나 낙하산을 몸에 묶는 '안전벨트·마구(馬具)'를 뜻한다. AI에서는 LLM을 둘러싸고 도구·메모리·실행 환경·안전장치·피드백 채널을 제공하는 소프트웨어 계층을 말한다. 한마디로 "stateless한 모델을 진짜로 일하는 에이전트로 바꿔주는 껍데기"다.

🐴 비유 (마구 harness): 아무리 힘센 말(=똑똑한 LLM)이라도 마구(harness)가 없으면 마차를 끌 수 없다. 안장·고삐·바퀴를 연결하는 마구가 있어야 비로소 그 힘이 '운송'이라는 유용한 일로 바뀐다. AI 에이전트의 자율성을 결정하는 병목은 종종 말의 힘(모델의 똑똑함)이 아니라 마구의 품질(harness의 신뢰성)이다.

배경 3 — 이 논문의 핵심 통찰: "코드 = harness"

그렇다면 harness를 무엇으로 만들면 가장 좋을까? 이 논문의 답은 "코드"다. 왜냐하면 코드는 자연어 대화에는 없는 세 가지 마법 같은 성질을 가지기 때문이다:

코드의 세 성질. 이것이 단순한 표기법이 아니라 "harness를 기능하게 만드는" 핵심이라는 게 논문의 주장.
🗣️ 비유 (말 vs 글): 자연어로 "2,847 × 391을 계산했어, 답은 113만쯤이야"라고 말하는 LLM은 틀려도 그럴듯하게 들린다(검증 불가). 하지만 2847 * 391이라는 코드를 실제로 실행하면 1,113,177이라는 객관적 답이 나온다 — 모델이 거짓말을 할 수 없다. 코드는 "환각(hallucination)이 불가능한 객관적 진실의 채널"이다.

이 서베이의 큰 그림 — 3개의 계층

저자들은 흩어진 수백 편의 연구를 코드가 에이전트 안에서 점점 더 큰 역할을 맡아가는 3단계로 정리한다.

단일 에이전트의 도구(①·②)에서 여러 에이전트의 공유 기반(③)으로 발전.

계층 ① — 코드가 에이전트의 '입구'가 된다

가장 기초적인 단계. 코드가 모델과 작업 환경을 잇는 다리 역할을 한다. 세 가지 방식:

(1) 추론을 위한 코드 (Code for Reasoning)

LLM은 머릿속 계산에 약하다. 그래서 계산을 '코드'로 떠넘긴다.

🧮 비유 (계산기): 똑똑한 사람도 큰 곱셈은 계산기를 쓴다. PoT(Program of Thoughts)는 LLM이 "직접 암산"하는 대신 "계산할 코드를 짜서 컴퓨터에게 시키는" 방식이다. 추론의 신뢰성이 크게 올라간다. 한 발 더 나가면 Lean 같은 '수학 증명 언어'로 모든 단계를 검증한다(AlphaProof).

(2) 행동을 위한 코드 (Code for Acting)

로봇 팔을 움직이거나, 웹 버튼을 누르거나, API를 호출하는 '행동'을 코드로 표현한다.

🤖 비유 (레시피 카드): 요리사(에이전트)에게 "맛있게 만들어"라고 말하면 막막하다. 하지만 "①양파를 썬다 ②중불에 볶는다…"라는 실행 가능한 레시피(코드)를 주면 정확히 행동할 수 있다. Voyager는 마인크래프트에서 이런 '레시피(skill)'를 스스로 모아 도서관처럼 쌓아간다.

(3) 환경 모델링을 위한 코드 (Code for Environment)

에이전트가 일하는 '세상' 자체를 코드로 표현한다 — 코드 저장소, 시뮬레이터, 테스트가 곧 '세계'다.

🎮 비유 (게임 세이브 파일): 게임의 세계 상태가 세이브 파일에 저장되듯, 에이전트가 다루는 환경(코드베이스)도 파일·테스트·실행 로그로 저장된다. SWE-bench는 "GitHub 이슈를 고쳤는가?"를 유닛 테스트 통과 여부라는 객관적 세계 상태로 측정한다.

계층 ② — 코드가 '운영'된다 (Plan-Execute-Verify)

코드가 에이전트 안에 들어오면, 이제 "무엇을 다음에 할지" 결정하고, 실패를 고치는 '운영' 메커니즘이 필요하다. 핵심은 계획→실행→검증(PEV) 루프다.

디버깅을 단순 '에러 고치기'가 아니라 '통제된 상태 전이'로 재구성한다.

안전을 위한 3단계 권한 (Permission Tier)

에이전트가 위험한 일(파일 삭제, 배포)을 멋대로 하지 않도록, 행동을 위험도별로 나눈다:

🟢 읽기 전용

코드 둘러보기, 로그 분석 — 위험 없음, 자유롭게 허용

🟡 샌드박스 편집

격리된 공간에서 코드 수정·테스트 — 제한된 범위

🔴 전체 접근

네트워크·배포·삭제 — 반드시 사람의 승인(HITL) 필요

스스로 진화하는 harness (AHE)

🔧 비유 (정비공이 자기 공구를 개선): 보통 에이전트는 '코드(자동차)'를 고친다. 그런데 한 단계 위의 '진화 에이전트(Evolution Agent)'는 에이전트가 쓰는 공구·작업대(harness) 자체를 개선한다. 단, 작업대를 함부로 바꾸면 위험하므로 — "안전 규칙을 바꾸는 변경은 반드시 사람 승인" 같은 통제(governed mutation)가 붙는다.

계층 ③ — 코드가 '공유 작업대'가 된다 (멀티에이전트)

혼자서는 한계가 있다. 그래서 여러 에이전트(기획자·코더·테스터·리뷰어)가 같은 코드 저장소를 공유하며 협업한다. 사람 개발팀과 똑같다.

여러 전문 에이전트가 하나의 공유 코드 작업대를 중심으로 협업한다.
🏗️ 비유 (건설 현장의 청사진): 건축가·목수·전기공이 협업하려면 모두가 같은 청사진을 봐야 한다. 한 명만 옛 도면을 보면 벽이 어긋난다. 마찬가지로 멀티에이전트 코딩에서 가장 큰 문제는 "각자 머릿속 코드 상태가 어긋나는 것(state divergence)"이다.

이 논문이 진단한 핵심 문제와 제안

⚠️ 진단: 대부분의 시스템은 '청사진'이 없다

저자들이 조사한 멀티에이전트 시스템 대다수는 공유 상태를 그때그때 대화 기록에서 암묵적으로 재구성한다(implicit/file-only). 형식적인 공유 청사진이 없어서, 한 에이전트의 착각이 시스템 전체에 보이지 않게 퍼진다. 이것이 시스템이 자주 깨지는(brittle) 근본 원인이라는 게 핵심 진단.

💡 제안: 공유 코드 중심 기반 (Shared Code-Centric Harness Substrate)

차세대 멀티에이전트 지능은 모든 에이전트가 읽고 쓸 수 있는 형식적·영속적 공유 코드 상태가 필요하다. 코드는 멀티에이전트 영역 중 유일하게 '실행되어 객관적 신호를 만드는' 매체이므로, 이 성질을 적극 활용해야 한다.

공유 상태 형식화 단계설명대표
① 암묵적/파일만매번 대화에서 재구성 (가장 흔하고 취약)ChatDev, MetaGPT
② 저장소 기반탐색 가능한 코드 저장소를 공유MAGIS, SyncMind
③ 실행 기반"코드가 무엇을 하는가"(테스트·실행 결과)가 상태AgentCoder, MAGE
④ 블랙보드/공유상태모두가 읽고 쓰는 명시적 영속 자료구조 (이상에 가장 근접)L2MAC

어디에 쓰이나 — 5대 응용 분야

💻 코딩 어시스턴트

Claude Code, Codex, Cursor — 저장소 전체를 다루며 PR까지 작성. 2026년엔 이 도구의 사용 기록이 차세대 모델의 학습 데이터가 됨.

🖥️ GUI/OS 에이전트

화면의 모든 것이 코드(HTML)의 결과물, 모든 클릭이 코드 호출. "가장 문자 그대로의 프로그램 세계".

🔬 과학 발견

가설→실험→분석이 모두 코드로. 자율 실험실(A-Lab)이 17일 만에 41개 신물질 합성.

🎯 개인화/추천

사용자 선호를 "수정 가능한 코드 객체"로 저장 → 사용자가 자연어로 고치면 즉시 반영.

🦾 로봇/체화 에이전트

코드가 '행동의 안전 경계'. 물리 제약(충돌·도달범위)을 행동 생성 시점에 강제.

🔭 공통점

모든 분야에서 코드는 관찰·행동·평가·기억·세계모델의 공통 언어(lingua franca)가 된다.

아직 풀리지 않은 7가지 숙제

  1. 평가의 한계: "테스트 통과"만으로는 부족 — 초록불이 떴다고 진짜 잘한 게 아니다(green test ≠ 완전한 검증).
  2. 실행을 넘어선 의미 검증: 테스트가 불완전하면 에이전트가 '잘못된 신호'에 맞춰 학습한다.
  3. 퇴보 없는 자기 진화: harness가 스스로 개선하되, 이전에 풀던 문제를 망가뜨리지 않아야 한다.
  4. 트랜잭션 공유 상태: 코드는 동기화돼도 '가정(assumption)'은 동기화 안 됨 → 의미적 충돌 해결 필요.
  5. 안전과 책임을 harness 상태로: 사람의 승인·거부 이력을 일회성이 아니라 영속적 규칙으로.
  6. 멀티모달 harness: 화면·영상·센서 같은 비텍스트 상태도 다뤄야 한다.
  7. 'harness 공학'이라는 과학: 연구 대상이 모델이 아니라 전체 폐루프 시스템으로 이동.

한계점 — 이 서베이를 읽을 때 주의할 점

🔸 실험으로 증명한 게 아니다

이 논문은 새로운 실험·벤치마크가 없는 '관점 제시(position) 서베이'다. "공유 코드 기반이 더 좋다"는 핵심 주장은 통제된 비교 실험으로 검증된 게 아니라 개념적 논증이다. 오히려 저자 스스로 "LLM이 코드를 실행하지 않고도 98% 정확히 시뮬레이션하더라"는 반례를 인정한다.

🔸 'harness'의 경계가 모호하다

코드를 로그·설정·심지어 '자연어 harness 명세'까지 넓게 정의하다 보니, 거의 모든 에이전트 인프라가 harness가 되어버린다. 무엇이 harness가 "아닌지"가 불분명해 반증하기 어렵다.

🔸 산업 주장은 검증이 어렵다

2026년 최신 상용 시스템(Cursor, Codex-Max)까지 다루는 시의성은 훌륭하나, 그 정보는 주로 회사 블로그·백서 수준이라 학술적 검증이 어렵다.

"AI 에이전트의 미래는 '더 똑똑한 두뇌'만으로 오지 않는다.
그 두뇌가 일하는 '작업대(harness)'를 잘 설계하는 것 —
실행 가능하고(executable), 검사 가능하고(inspectable),
상태를 유지하고(stateful), 통제되는(governed) 코드 기반이
다음 프런티어다."

핵심을 다시 한 번

  • 🐴 AI 에이전트 = LLM(두뇌) + harness(마구). 자율성의 병목은 종종 두뇌가 아니라 마구다.
  • ⚙️ 코드가 최고의 harness인 이유: 실행·검사·상태유지가 가능해 '객관적 진실의 채널'이 된다.
  • 🪜 3계층: 코드가 입구(Interface) → 운영(Mechanism) → 공유 작업대(Scaling)로 발전.
  • 🔁 실무 패턴: Plan-Execute-Verify 루프 + 3단계 권한 + 위험 작업엔 사람 승인.
  • 🏗️ 멀티에이전트의 핵심 문제는 '공유 상태 어긋남' → 형식적 공유 코드 기반이 해법.
  • 🔭 5대 응용(코딩·GUI·과학·추천·로봇) 모두에서 코드가 공통 언어가 된다.
