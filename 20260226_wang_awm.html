TL;DR

  • 문제: LLM agent의 tool-use 능력을 RL로 학습시키려면 다양하고 실행 가능한 환경이 필요하지만, 현실 환경은 비용이 높고 확장이 어렵다. 기존 합성 데이터 연구는 task/trajectory 합성에 집중하며 환경 자체를 합성하지 않고, LLM 시뮬레이션 환경은 hallucination과 높은 inference 비용 문제가 있다.
  • 해결: ##코드 기반의 실행 가능한 환경을 자동 합성하는 파이프라인 AWM (Agent World Model)을 제안##한다. 시나리오 → 태스크 → 데이터베이스 → 인터페이스(MCP) → 검증 코드의 5단계로 1,000개 환경(35,062개 도구, 10,000개 태스크)을 생성한다. 모든 상태 전이는 SQLite DB에 의해 보장되며, code-augmented LLM-as-a-Judge가 RL reward를 제공한다.
  • 왜 작동하는가: 환경이 코드 기반이므로 LLM 시뮬레이션 대비 (1) 상태 전이가 결정적이고 일관적이며, (2) 에이전트 상호작용이 훨씬 효율적이고, (3) DB 상태 diff를 활용한 grounded verification이 가능하다. 또한 다양한 도메인(쇼핑, 금융, 여행 등)을 포괄하는 1,000개 환경이 OOD (Out-of-Distribution) 일반화를 지원한다.
  • 비유: 기존 agent 학습이 "소수의 실제 가게에서만 직접 연습"하는 것이라면, AWM은 "자동으로 수천 개의 가상 가게를 코드로 생성하여 다양한 상황을 경험하게 하는 시뮬레이터"이다. 가게 내부 상태(재고, 주문 등)가 실제 데이터베이스에 기록되므로, LLM이 상상으로 답하는 것보다 훨씬 신뢰성 있는 연습이 가능하다.

핵심 기여 (Key Contributions)

  1. AWM 합성 파이프라인: 시나리오 → 태스크 → DB → MCP interface → verification code의 5단계로 실행 가능한 tool-use 환경을 자동 생성하는 오픈소스 파이프라인. Human prior (API 문서, 기존 task set 등) 없이 100개 seed 도메인 이름만으로 확장 가능하다.
  2. 1,000개 환경 데이터셋: 35,062개 도구, 10,000개 태스크, 환경당 평균 ~2,000줄 코드. SQL-backed 상태 일관성을 보장하며, 현존 최대 규모의 오픈소스 tool-use 환경 셋이다.
  3. Agentic RL 기법: Code-augmented LLM-as-a-Judge (hybrid verification), step-level format 보상, history-aware training (학습-추론 일관성)을 결합한 multi-turn RL 학습 프레임워크.
  4. OOD 일반화 실증: AWM에서만 학습한 에이전트가 BFCLv3, τ²-bench, MCP-Universe 등 벤치마크-specific 학습 없이 강력한 일반화 성능을 보인다.

1. Introduction — 문제 정의와 동기

LLM 기반 자율 에이전트는 multi-turn tool 상호작용으로 복잡한 태스크를 수행할 수 있지만, 이를 대규모 RL로 학습시키려면 다양하고 신뢰성 있는 환경이 필수적이다. 현재 환경 확보의 세 가지 한계:

  1. 현실 환경: 공개 API가 없거나, 수천 번의 안정적 상호작용이 비현실적. 비용이 높고 확장 불가.
  2. 인간 제작 환경: τ-bench (2개), τ²-bench (3개), TheMCPCompany (5개) 등 소규모. 다양성과 확장성 부족.
  3. LLM 시뮬레이션: 매 상태 전이마다 LLM을 호출하므로 hallucination 위험과 높은 inference 비용. RL 학습에 부적합.

기존 합성 연구는 주로 task 합성 (Self-Instruct, AgentSynth) 또는 trajectory 수집 (AgentBank, AgentTrek)에 집중하며, 환경 자체를 합성하지 않는다. 환경 합성을 시도한 연구도 있으나 (AutoForge: 36개 game-like 환경, EnvScaler: 191개), 규모가 제한적이거나 human prior에 의존한다.

DeepSeek-V3.2와 Qwen Tongyi도 환경 합성 파이프라인을 사용했다고 언급하지만, ##파이프라인과 환경 모두 비공개##이다. AWM은 이 gap을 메우기 위해 파이프라인과 1,000개 환경을 모두 오픈소스로 공개한다.

2. Related Work

Tool-use Agents

Toolformer (Schick et al., 2023)는 supervised learning으로 tool-use를 학습하고, ToolLLM (Qin et al., 2024)은 실제 API를 curate하되 tool response는 시뮬레이션한다. Gorilla (Patil et al., 2024)는 API 문서로 fine-tuning하며, ReAct (Yao et al., 2023)과 SWE-agent는 interactive 환경에서 reasoning과 acting을 교대 수행한다. 그러나 대부분의 학습 데이터는 정적이거나 소규모 환경에서 수집된다. 기존 벤치마크(τ-bench, BFCLv3, MCP-Universe)는 평가에는 적합하지만 대규모 RL 학습 환경으로 사용하기는 어렵다.

Environment Synthesis

환경 합성은 두 방향으로 나뉜다:

  • LLM-based simulation: 매 상태 전이를 reasoning model로 생성. Hallucination과 비효율성 문제.
  • Programming-based synthesis: 코드와 DB로 환경을 구축. AutoForge (10개, Tool Doc 의존), EnvScaler (191개, 기존 task set에 의존), AutoEnv (36개 game-like).

AWM은 (1) 사전 정의된 task set이나 API 문서 없이 scratch에서 합성, (2) SQL-backed 상태 관리로 일관성 보장, (3) 1,000개 환경으로 최대 규모라는 점에서 차별화된다.

3. Agent World Model — 파이프라인

AWM은 환경을 POMDP (Partially Observable Markov Decision Process)로 합성한다. 각 환경 $E_i$는 5개 구성 요소를 포함한다: 상태 공간 $S_{E_i}$, 행동 공간 $A_{E_i}$, 관측 공간 $O_{E_i}$, 전이 함수 $T_{E_i}: S_{E_i} \times A_{E_i} \to S_{E_i} \times O_{E_i}$, 그리고 태스크별 보상 함수 $R_\tau$.

AWM 파이프라인 (5단계): Step 1: Scenario Synthesis 100개 seed 도메인명 → LLM 확장 → 필터링/중복제거 → 1,000개 시나리오 Step 2: Task Synthesis 각 시나리오 → LLM → 10개 task/시나리오 → 10,000개 태스크 Step 3: Database Design & Data Synthesis 시나리오 + 태스크 → LLM → SQLite 스키마 + 샘플 데이터 → 상태 공간 S Step 4: Interface Synthesis 태스크 + DB 스키마 → LLM → MCP 도구 스키마 → Python 구현 코드 → 행동 공간 A, 관측 공간 O, 전이 함수 T Step 5: Verification & Online RL 태스크 + DB → LLM → 검증 코드 + 성공/실패 기준 → 보상 함수 R → Code-augmented LLM-as-a-Judge → RL 학습

3.1 Scenario Generation

100개의 인기 도메인 이름(seed)에서 Self-Instruct 스타일로 확장한다. LLM이 stateful application (e-commerce, CRM 등)에 적합한 시나리오를 생성하고, 두 가지 필터를 적용한다:

  • LLM 분류기: CRUD (Create, Read, Update, Delete) 연산이 핵심인 시나리오만 선택. 뉴스/위키 같은 읽기 전용 시나리오 제외.
  • Embedding 기반 중복 제거: Cosine similarity 0.85 임계값으로 유사 시나리오 제거. 과잉 대표 카테고리에 cap을 걸어 다양성 유지.

결과: 금융, 여행, 소매, 소셜 미디어, 분석, 워크플로우 등을 포괄하는 1,000개 unique 시나리오. 카테고리 분포는 Other (16.8%), Analytics (8.6%), Workflow (8.6%), E-commerce (8.2%) 등으로 고르게 분포.

3.2 Task Generation

각 시나리오에 대해 LLM이 $k=10$개의 다양한 user task를 생성한다. 두 가지 설계 원칙:

  1. API-solvability: 클릭, 페이지 네비게이션 같은 순수 UI 의존 작업 배제. API로 해결 가능한 태스크만.
  2. Post-authentication context: 로그인 완료를 가정하여 깊은 기능에 집중. 인증은 현실에서 인간이 처리하므로.

결과: 10,000개의 실행 가능한 태스크. 이 태스크들이 후속 환경 백엔드 합성의 기능 요구 사항 역할을 한다.

3.3 Environment Synthesis

Database (상태 공간 $S_{E_i}$)

시나리오와 태스크 셋이 주어지면, LLM이 모든 태스크를 실행 가능하게 만드는 최소한의 relational schema를 추론한다. ##SQLite를 상태 백엔드로 사용##하여 구조화된 relational state를 관리한다. 이는 concurrent work들이 사용하는 단순화된 NoSQL/key-value store와 차별화된다.

스키마만으로는 부족하다 — 많은 태스크가 기존 레코드를 조회/갱신해야 하므로, LLM이 각 태스크의 전제 조건을 분석하여 realistic initial state $s_0$를 만드는 샘플 데이터를 합성한다.

Interface (행동/관측 공간 $A_{E_i}$, $O_{E_i}$, 전이 함수 $T_{E_i}$)

에이전트가 DB를 직접 조작하지 못하도록, ##MCP (Model Context Protocol)## 인터페이스를 통해 도구를 노출한다. MCP는 Anthropic이 2024년에 제안한 통합 agent-tool 상호작용 프로토콜이다.

합성은 두 단계: (1) 먼저 toolset schema를 설계 (도구 이름, 매개변수 타입, 응답 스키마), (2) 그 다음 Python 구현 코드를 생성. Pilot 실험에서 환경이 3,000줄 이상의 코드를 필요로 할 수 있어, 스키마 가이드 없이 직접 코드 생성은 어려웠다.

Verification (보상 함수 $R_\tau$)

각 태스크 $\tau$에 대해, 에이전트 실행 전후의 DB 상태를 비교하는 검증 모듈을 합성한다. 이 모듈이 추출하는 구조화된 검증 신호(변경 레코드, 예상 결과, 진단 신호)를 LLM-as-a-Judge에 전달하여 최종 판정을 받는다.

왜 LLM-as-a-Judge를 병용하는가? 순수 코드 검증은 합성 환경의 불완전성(부분 실행, 인프라 이슈 등)에 취약하다. 현실의 WebArena, τ²-bench, SWE-Bench도 brittle verification 때문에 수정된 바 있다. ##Code-augmented LLM-as-a-Judge는 코드 검증의 정밀성과 LLM의 유연성을 결합##하여, 구조화된 증거를 기반으로 코드만으로 해결할 수 없는 모호성을 처리한다.

최종 판정: Completed, Partially Completed, Agent Error, Environment Error 중 하나를 반환.

Self-Correction 메커니즘

모든 합성 단계에서 코드가 실패하면 오류 메시지를 LLM에 피드백하여 수정을 시도한다. 최대 5회 반복. 85% 이상의 first-attempt 성공률, 평균 1.13회 correction iteration.

4. Agentic Reinforcement Learning

4.1 Reward Design

순수 outcome-based reward는 수학 추론에서는 성공했지만, multi-turn tool-use에서는 불충분하고 비효율적이다. AWM은 hybrid reward를 사용한다:

Step-level: 매 step에서 tool call의 format 준수 여부 확인. 위반 시 즉시 rollout 종료 + $r_t = -1.0$. 이는 invalid action을 억제하고 long-horizon에서의 계산 낭비를 방지한다.

Task-level (정상 종료 시): Code-augmented LLM-as-a-Judge가 최종 보상을 결정:

$$R_\tau = \begin{cases} 1.0, & \text{if task } \tau \text{ Completed} \\ 0.1, & \text{if task } \tau \text{ Partially Completed} \\ 0.0, & \text{otherwise} \end{cases} \tag{1}$$

직관적으로, 완전 성공은 1.0, 부분 성공은 0.1 (약간의 positive signal), 실패는 0을 부여한다. Environment Error는 에이전트의 잘못이 아니므로 $r_t = 0$으로 처리하여 잘못된 negative signal을 방지한다.

4.2 History-Aware Training

에이전트 배포 시에는 긴 상호작용 히스토리를 sliding window로 truncation하는 것이 일반적이다. 그러나 기존 RL 학습 파이프라인은 full history를 사용하여 최적화하므로 학습-추론 간 distribution mismatch가 발생한다.

Full history $h_t = (o_1, a_1, o_2, a_2, \ldots, o_t)$를 사용하는 기존 방식:

$$L = -r \log \pi_\theta(o_1, a_1, o_2, a_2, \ldots, o_T, a_T) \cdot [0, 1, 0, 1, \ldots, 0, 1] \tag{2}$$

여기서 mask는 action token만 선택하고 observation은 무시한다.

##AWM은 학습 중에도 추론과 동일한 truncation을 적용##하여 일관성을 보장한다. GRPO 하에서, 각 태스크 $\tau$에 대해 $G$개의 rollout을 샘플링하고 다음을 최적화한다:

$$L_{\text{GRPO}} = \mathbb{E}_{\tau, E_i, \{y^{(k)}\}} \left[ \frac{1}{G} \sum_{k=1}^G A^{(k)} \sum_{t=1}^{T_k} \log \pi_\theta(a_t^{(k)} | h_t^{\text{trunc},(k)}) \right] \tag{3}$$

여기서 $A^{(k)} = (R^{(k)} - \bar{R}) / \sigma_R$은 group-relative advantage이고, $h_t^{\text{trunc}}$는 sliding window $w=3$으로 truncation된 히스토리이다. 직관적으로, 긴 trajectory를 여러 sub-trajectory로 분할하여, 각각이 추론 시 에이전트가 실제로 보는 context와 동일하도록 만드는 것이다.

5. Experiments

5.1 Experimental Setup

실험의 핵심 질문: 합성 환경에서만 학습한 에이전트가 벤치마크-specific 환경 없이도 일반화할 수 있는가?

벤치마크 (모두 OOD — AWM 학습에 포함되지 않음):

  • τ²-bench (verified version): 항공, 소매, 통신 3개 시나리오의 multi-turn 대화형 에이전트 태스크
  • BFCLv3 (Berkeley Function Calling Leaderboard v3): function-calling 능력 종합 평가. Non-live, Live, Multi-turn, Hallucination 4개 카테고리
  • MCP-Universe: 실제 MCP 서버 기반. Location, Financial, Browser, Web, Multi-server 5개 도메인

Baselines:

  • Base: 추가 학습 없는 원래 LLM
  • Simulator: GPT-5가 환경 전이를 시뮬레이션하는 LLM-simulated 환경에서 RL 학습
  • EnvScaler: 191개 programming-based 합성 환경에서 SFT/RL 학습

τ²-bench (Conversational Agent)

AWM은 τ²-bench에서 EnvScaler와 경쟁적이며 Simulator보다 일관되게 우수하다. 단, EnvScaler는 기존 task set에 의존하여 합성하므로 τ²-bench와 overlap이 있을 수 있어 이 비교에서 유리할 수 있다.

MCP-Universe (Real-world MCP)

##AWM이 모든 규모에서 최고 overall 성능##. 8B: 11.17% (Base 6.70%, EnvScaler 5.59%, Simulator 6.15%). 특히 Financial과 Location 도메인에서 큰 개선. Programming-based 상태 일관성이 LLM-simulated 학습보다 안정적인 learning signal을 제공하며, 학습 latency도 크게 줄인다 (Simulator는 매 interaction step마다 LLM 호출 필요).

7. Conclusion & Limitations

AWM은 실용적 소프트웨어 개발을 모방하여 실행 가능한 agentic 환경을 대규모로 합성하는 파이프라인이다. 1,000개 환경, 35,062개 도구, 10,000개 태스크를 합성하고, 3개 벤치마크에서 OOD 일반화를 실증했다.

Limitations

  • Self-Evolving 부재: 현재 파이프라인은 고정된 생성 프로세스. 학습된 에이전트가 새 환경 합성에 기여하는 self-evolving 패러다임은 향후 과제.
  • 합성 파이프라인 최적화: Self-correction이 trial-and-error에 의존하며, LLM을 활용한 proactive한 논리적 불일치 탐지가 미비. Multi-scenario spanning task도 향후 확장 방향.
  • 학습 규모: 1,000개 중 526개로 학습 (compute 제약). Qwen3 4B/8B/14B만 실험. 다른 모델 family에서의 효과는 미검증.

비판적 분석 (Critical Analysis)

강점

  • 시스템 기여의 완결성: 파이프라인, 환경, 학습 코드를 모두 오픈소스로 공개하여, 커뮤니티가 즉시 활용할 수 있다.
  • 실용적 설계: 소프트웨어 개발 패턴(요구사항 → 스키마 → 코드 → 테스트)을 따르는 합성 프로세스는 직관적이고 확장 가능하다.
  • 철저한 ablation: Verification 전략 (3가지), history management (aligned/misaligned), environment 수, format reward 등 다각도에서 분석.
  • OOD 일반화 설계: 벤치마크-specific 환경에서 학습하지 않으므로, 결과가 진정한 일반화 능력을 반영한다.

약점 및 한계

  • 합성 품질의 한계: 74% 환경에 버그 존재, 14% 태스크가 blocked. 이는 현실 환경과의 fidelity gap을 의미하며, 더 복잡한 시나리오에서는 비율이 올라갈 수 있다.
  • Hallucination 약점: BFCLv3 hallucination 카테고리에서 Base 대비 하락. Format reward가 "항상 도구를 사용하라"고 장려하므로, 도구 사용을 거부해야 하는 상황을 학습하지 못한다.
  • GPT-5 의존성: 합성과 verification 모두 GPT-5에 의존. 합성 비용 ($57/100 환경)은 합리적이나, RL 학습 중 매 step마다 GPT-5 judge 호출은 완전한 자율성을 제한한다.
  • 벤치마크 절대 성능: MCP-Universe에서 14B 모델도 12.29%에 그쳐, 실제 MCP 환경에서의 tool-use는 아직 초기 단계임을 보여준다. AWM 학습의 효과가 존재하지만 절대 성능은 낮다.
  • 반복 실험 부족: τ²-bench Pass@1만 4회 반복. BFCLv3와 MCP-Universe는 단일 실행 결과. 통계적 유의성 확인이 어렵다.
  • 모델 다양성: Qwen3 family만 실험. LLaMA, Gemma 등 다른 모델에서의 효과는 미검증.

Key Takeaways

  1. 환경 합성이 agentic RL의 병목을 해소한다: 기존에는 환경 부족이 agent 학습의 최대 제약이었다. AWM은 코드 기반 환경 합성으로 이 병목을 정면 돌파하며, 1,000개 규모까지 자동 확장이 가능함을 실증했다. 이는 데이터 합성을 넘어 "인프라 합성"이라는 새로운 패러다임을 제시한다.
  2. SQL-backed 상태 관리가 LLM 시뮬레이션보다 우월하다: LLM이 환경을 시뮬레이션하면 hallucination과 비용이 문제인 반면, DB 기반 코드 실행은 결정적이고 효율적이다. Simulator 대비 AWM의 일관된 우위가 이를 뒷받침한다.
  3. Code-augmented verification이 RL reward의 핵심이다: 순수 코드 검증도, 순수 LLM 판정도 불완전하다. 코드가 구조화된 증거를 제공하고 LLM이 context-aware 판단을 내리는 hybrid 방식이 가장 robust하다. 이 패턴은 다른 합성 환경/벤치마크에서도 적용할 수 있다.
  4. 학습-추론 일관성이 중요하다: History truncation을 추론 시에만 적용하면 distribution mismatch가 발생한다. 학습 중에도 동일한 truncation을 적용하는 history-aware training이 성능을 크게 향상시킨다. 이는 multi-turn agent RL의 일반적 가이드라인이 될 수 있다.
  5. 환경 다양성이 일반화의 핵심이다: 10개 → 100개 → 526개로 환경 수를 늘릴수록 성능이 monotonically 향상된다. EnvScaler의 일부 벤치마크에서의 하락은 환경 다양성 부족(191개, 기존 task set 의존)이 원인일 수 있다. 범용 agent 학습에는 대규모 다양한 환경이 필수적이다.
