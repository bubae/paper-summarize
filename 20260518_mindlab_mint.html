TL;DR

  • 문제: 현대 LLM post-training은 더 이상 1회성 SFT가 아니라 ##수많은 정책(policy) 변종이 소수의 비싼 base 모델 위에서 공존하는 long-running workload##다. 기존 인프라는 매 변종마다 full checkpoint를 복사·서빙해서 ##GPU 메모리와 storage가 폭발##하고, 그 결과 LoRA RL 같은 핵심 워크로드를 운영하기가 사실상 불가능하다.
  • 해결: MinT는 ##base 모델 deployment는 resident(상주)##시키고 LoRA adapter revision만 학습/서빙/롤백/평가의 단위로 삼는 ##managed 서비스##다. 학습-서빙 boundary를 가로지르는 artifact가 ##full checkpoint(GB급)에서 LoRA adapter(<1% MB급)##로 줄어든다.
  • 왜 작동하는가: (1) Scale Up — Megatron-style 모델병렬과 sparse routing(MoE/DSA) 위로 LoRA를 매핑해 1T+ 모델까지 지원, (2) Scale Down — adapter-only handoff가 base checkpoint materialization 단계를 ##통째로 제거##, (3) Scale Out — durable catalog(10^6)을 CPU/GPU hot working set과 ##분리##해 cold load를 scheduled work로 격상.
  • 비유: 매 학생마다 학교 건물을 새로 짓던(full fine-tune) 시대에서, 한 캠퍼스 위에 ##여러 강좌(adapter)가 시간을 나눠 쓰는## 운영체제로 전환. 강좌 등록부(policy record)는 학사 행정이, 강의(adapter revision)는 강의실(resident base)이 따로 맡는다.

핵심 기여 (Key Contributions)

  1. Adapter Revision = Policy Unit: ##trained behavior를 carrying하는 payload##인 LoRA adapter revision과, 그 payload를 ##schedulable/durable하게 만드는 service state##인 policy record를 분리. 학습-서빙 boundary를 가로지르는 artifact가 full checkpoint가 아니라 fixed adapter revision이 되도록 system property 자체를 재정의.
  2. Time-sliced Multi-LoRA RL Training: 한 resident base 위에서 여러 LoRA 정책을 ##gradient step 단위로 time-slice##. 동일한 base 메모리 할당으로 ##concurrent GRPO가 wall time을 4B에서 1.77×, 30B MoE에서 1.45× 단축##하면서도 peak memory를 늘리지 않음. Single-worker PEFT와 distributed Megatron 양쪽 경로 모두 지원.
  3. Three-tier Adapter Cache: ##(addressable catalog 10^3–10^6 entries · CPU adapter cache hundreds · GPU batch ≤64)##의 3-tier 분리. 각 tier는 lifetime, promotion/eviction 규칙, 측정 가능한 latency 특성이 다르며 ##durable policy addressability를 hot working set과 분리##해 first-touch 비용을 scheduled work로 다룬다.
  4. Packed MoE LoRA Representation: MoE LoRA adapter가 만드는 ##37,248개 tiny tensor object##를 ##672개로 패킹(55.4×)##하여 file size는 1.05× 변동에 그치면서 live engine loading을 ##8.5–8.7× 가속##(N=4/8/16 모두에서 0.2초 미만의 live-load 중앙값).
  5. Adapter-Only Handoff 측정: Qwen3-4B rank-32에서 adapter ##252 MiB vs merged full checkpoint 8.061 GB## — adapter 경로가 4B에서 ##18.3×##, 30B MoE에서 ##2.85×## 빠른 step time. 4-billion parameter bf16 base의 ##3.3%##, rank-1 설정에서는 ##0.10%## 미만.
  6. Scale 검증: Qwen3 series (4B dense, 30B-A3B, 235B-A22B), Moonlight-16B-A3B & Kimi K2 1.04T countdown-task LoRA RL, GLM-5/GLM-5.1 (MLA, DSA, MTP) — ##LoRA RL을 1T+ 파라미터 스케일까지 실증##.
  7. AutoResearch Pipeline: LawBench 사례에서 ##28개 실험 candidate를 proxy로 screening, 6개를 kept candidate로 promote, full-benchmark confirmation##까지 같은 adapter lifecycle로 처리. proxy vs full divergence가 명시적 신호로 활용됨.

배경 (Background)

Post-training의 워크로드 변화

2024년까지의 post-training은 "SFT 한 번, RL 한 번"의 ##선형 파이프라인##이었다. 2025–26년의 frontier 모델들(DeepSeek-V4, Kimi K2 1T, GLM-5, Qwen3, MiniMax M1 등)이 등장하면서 post-training은 다음과 같이 변형되었다:

  • 지속적 fine-tuning: agentic tool-use, executable environment, multi-turn evaluation을 위한 task-specific 변종이 ##수십에서 수백 개##.
  • Multi-tenant policy populations: 한 frontier base 위에 product branch, experimental version, rollback point, 사용자별 personal adapter가 누적.
  • Asynchronous agent RL: rollout과 update가 비동기 진행되며 각 policy variant가 ##고유한 rollout record/checkpoint/exported revision##을 가짐.

기존 Multi-LoRA 시스템의 한계

LoRA 자체는 Hu et al. (2022)의 표준 기법이지만, ##학습+서빙 lifecycle을 통합 관리##하는 인프라는 부재했다:

vLLM LoRAInference batchingadapter lifecycle 정책 없음

"왜 Full-Checkpoint 모델이 깨지는가"

Full fine-tuning 워크플로우는 모든 trained variant에 대해 ##independent base checkpoint##를 학습-서빙 사이로 옮긴다 (예: 30B 모델이면 ##60+ GB per variant##). Merge-based LoRA는 학습 단계의 메모리는 줄이지만 ##merged full checkpoint를 inference worker로 이동##해야 하므로 같은 storage/bandwidth 비용이 든다. 이는:

  • 매 task variant마다 ##별도 full-model server##가 필요 — agentic 워크로드에서는 수백 개의 server를 띄워야 함.
  • Rollback / personalization branch가 storage cost를 곱셈으로 누적.
  • Frontier-class MoE/MLA 모델은 base가 너무 커서 ##variant 하나 더 띄우는 것조차 GPU pool에 부담##.

Adapter Revision vs Policy Record

MinT의 가장 핵심적 추상화 분리:

Adapter revision

  • ##특정 training step에서 frozen된 LoRA adapter##의 serving-tensor-layout snapshot.
  • Rollout, evaluation, serving, rollback이 ##consume하는 executable payload##.
  • Inference engine이 이미 holding하고 있는 base와 ##compatibility metadata##로 binding.

Policy record

  • ##Lifecycle state##를 담는 service-owned entity.
  • 다음 정보 포함: compatible base version, LoRA rank, target modules, latest training checkpoint, rollout records (for later updates), exported adapter revisions (fixed behaviors available).
  • Live training session은 ##하나의 worker에 그 record를 temporary restoration##한 상태.

이 분리가 가능하게 하는 것:

  • 같은 policy record가 ##여러 cache tier에 동시에 존재##할 수 있다 (durable storage / CPU cache / GPU batch).
  • Worker가 ##exited되거나 cache가 evicted된 후##에도 나중 요청이 같은 adapter revision을 selectable.
  • 학습 도중 export된 revision이 서빙되는 동안 ##trainer는 계속 진행 가능##.

Time-Sliced Multi-LoRA Training

핵심 통찰: ##RL request는 보통 충분한 rollout token을 포함##하므로 한 trainer에서 efficient per-policy batch를 형성할 수 있다 (mLoRA의 multi-adapter same-batch와 다른 접근).

한 trainer actor의 lifecycle: t_1: LoRA A 활성 (base는 항상 resident) ↓ switch (save A / load B) t_2: LoRA B 활성 ↓ switch (save B / load A) t_3: LoRA A 활성 Switch 시점에 옮기는 state: LoRA tensors / optimizer moments / scheduler position / accumulated gradients / unconsumed rollout records Base weights: 상주 → 옮기지 않음

왜 mLoRA가 아닌 time-slicing인가: mLoRA-style fused multi-LoRA kernel은 ##fine-tuning 워크로드##(여러 small adapter를 동시 update)에 최적화되어 있지만, RL에서는 한 rollout이 수천 토큰이므로 ##single-policy batch가 이미 충분히 크고##, kernel fusion gain보다 ##policy 간 schedule overlap gain##이 더 크다.

Distributed (Megatron) LoRA Training

MoE/MLA/DSA base에서 base 모델 자체가 multi-GPU에 sharding됨:

  • Tensor parallel (TP): dense module의 LoRA tensor가 base shard와 같은 partition을 따름.
  • Expert parallel (EP): per-expert LoRA tensor는 ##expert id로 keyed##; EP shard가 자기 expert의 LoRA를 owns.
  • Shared-expert LoRA: EP shard마다 deduplicate, export 시 single copy로 materialize.
  • Policy switching: base shard는 resident, adapter tensor와 optimizer moment만 다음 update를 실행할 rank로 restore.

MoE Router Replay & DSA Provenance

R3 (Ma et al., 2025)가 보인 문제: rollout에서 token이 expert A로 routed되었는데 training에서 ##implementation/precision 차이로 expert B로 routed##되면 gradient가 wrong path를 따라간다.

MinT는 ##rollout record에 selected expert ids를 저장##하고, training backend가 expert-parallel layout으로 그 id를 map할 수 있을 때 expert path를 ##replay##한다. 매핑 불가능한 token은 policy-gradient term에서 ##mask##.

DSA(Dynamic Sparse Attention, GLM-5)에는 ##router replay가 더 어렵다##: indexer top-k 결정이 small numerical difference로 token set을 바꿀 수 있음. MinT는 IcePop-style correction (Ling Team 2025) 사용 — training/rollout probability ratio가 trusted band를 벗어난 token은 ##importance weight을 0##으로. 이는 model을 mask out하는 것이지 정확한 sparse-attention token set을 reconstruct하지 않음.

비판적 분석 (Critical Analysis)

  • 저자/소속 익명성: "Mind Lab"으로만 명시되며 individual authorship 미공개. 기업 백서 형태로, ##peer review를 통한 method scrutiny가 어렵다##. 코드는 mint-cookbook으로 일부 공개되지만 ##management plane의 핵심 구현##은 closed.
  • 학습-서빙 latency trade-off의 단편적 측정: adapter handoff가 18.3× 빠르지만 ##warm sample speed는 merge path가 약간 우월##(20.6 vs 15.6 tok/s on 4B). Long-lived production deployment에서 ##누적 inference cost가 transfer saving을 잠식##하는 break-even point가 명시되지 않음.
  • "Million-scale catalog" 주장의 measurement gap: paper의 실측은 100k까지. 10^6은 ##extrapolation (Appendix Table 13의 sizing model)##이며 ##실제 measured 값이 아님##. fleet-level routing model 가정 하에서의 capacity estimate라는 점을 명시는 했지만 abstract는 다소 단정적.
  • Time-slicing vs mLoRA fusion 비교 누락: RL request의 ##rollout token이 충분히 크다##는 가정 하에 time-slicing을 선택했으나, ##short rollout (e.g., per-step preference scoring)## 환경에서 mLoRA fusion이 더 유리한 regime을 정량적으로 보이지 않음.
  • Cold-load staircase의 1.36 s/adapter는 절대값 의존: 이 measurement는 ##특정 cluster shape, network bandwidth, storage IO##에 강하게 의존. 다른 환경에서 packed representation의 8.5× gain이 재현될지는 ##infrastructure-specific##.
  • DSA correction의 정확도 검증 부재: IcePop-style correction이 ##unsafe scoring term을 mask##하지만, 그 mask가 학습 신호를 얼마나 손상시키는지 ablation 없음. R3 같은 explicit replay와 IcePop mask의 ##수렴 quality 비교##가 누락.
  • Tinker-compatibility의 정확한 boundary: paper는 Tinker-compatible API를 표방하지만 Tinker의 모든 primitive에 대한 ##compliance matrix##는 없음. SkyRL tx 같은 다른 Tinker backend와의 ##semantic 차이##가 mint-cookbook 외에서 검증되지 않음.
  • Adapter shape limits: Same resident base에서 different rank/target-module은 worker가 ##configured shape limits##을 광고. 이 제약을 넘는 request는 다른 worker로 라우팅되어야 함. 실제 다양한 shape mix를 가진 production load의 ##scheduling efficiency##는 미보고.

Key Takeaways

  • "Post-training은 단일 단계가 아니라 운영체제다": long-running multi-tenant policy population을 관리하는 OS-like abstraction이 필요. MinT의 ##adapter revision = payload, policy record = service state## 분리는 다른 RL/serving 시스템에도 transferable.
  • "Time-slicing은 fusion보다 RL friendly": RL의 큰 rollout batch가 per-policy efficient kernel을 가능하게 하므로, ##mLoRA 같은 multi-adapter fused kernel보다 simple time-slice##가 동일 memory budget에서 더 효율적. 이 통찰은 LoRA training 일반론과 다른 RL-specific design 결정.
  • "MoE adapter representation이 cache miss latency를 결정한다": byte size가 같아도 ##tensor object fanout##이 loader overhead를 지배. Packed representation으로 8.5× live-load 가속을 얻는 system trick은 ##다른 sparse model serving에도 즉시 적용 가능##.
  • "3-tier cache는 명시적 SLA": addressable catalog (10^6) vs CPU cache (hundreds) vs GPU batch (≤64)의 분리가 ##각 tier의 promotion/eviction과 latency 특성을 측정 가능한 표면으로 노출##. cold-load가 service work가 되어 throttle/dedup/prewarm 가능.
  • "Trillion-parameter LoRA RL은 운영 가능하다": Kimi K2 1.04T 사례로 입증. ##frontier-scale post-training이 academic-size compute로도 접근 가능##해진 첫 명확한 demonstration. 단 이는 base가 cluster-resident일 때의 이야기 — base 자체를 띄울 수 있어야 함.
  • "Proxy-vs-full divergence가 explicit research signal": AutoResearch가 v11의 proxy-high를 full-confirm 단계에서 reject한 사례는 ##unsupervised proxy가 distribution shift를 만들 때 그 차이가 hypothesis test 신호##라는 점을 보여줌. 같은 adapter lifecycle이 이를 자연스럽게 carry.
