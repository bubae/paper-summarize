The Conductor — 7B 지휘자가 GPT-5와 Gemini, Claude를 모두 능가하는 법

베이스 한 음도 안 치는 오케스트라 지휘자처럼, 7B 모델이 강한 LLM 워커들의 협업을 자연어로 직접 설계해 LiveCodeBench 1위 + 평균 SOTA 달성. 단 2장의 H100으로 학습 가능.

한 줄 요약

"제일 똑똑한 AI 한 명을 만드는 게 아니라, 강력한 AI들이 어떻게 협업할지를 똑똑하게 설계하는 작은 AI를 만든다. 그 결과 7B Conductor가 GPT-5 + Claude + Gemini를 각자 능가한다."

왜 이 문제가 중요한가

"하나의 만능 LLM"은 없다

2026년 현재 frontier LLM들은 각자 다른 영역에 특화되어 있다:

🧮 GPT-5

수학(AIME), 코딩(LiveCodeBench)에 강함. knowledge-rich domain.

🔬 Gemini 2.5 Pro

과학 reasoning(GPQA-Diamond), long context. 다양한 학문 정보.

💻 Claude Sonnet 4

다양한 function call이 들어간 코드 생성(BigCodeBench). 소프트웨어 엔지니어링.

🚀 DeepSeek-R1

오픈소스 reasoning. 비용 효율 좋음.

🌟 Qwen3 (thinking)

중국어 + 다국어. 오픈 가중치. 구조화된 reasoning.

그러나 어떤 단일 LLM도 모든 task에서 universally optimal하지 않다. 그리고 하나의 task 안에서도 (예: 코딩) 서브 스텝마다 다른 모델이 더 잘한다 — 계획은 Gemini, 구현은 Claude, 검증은 GPT-5처럼.

기존 multi-agent 방법의 한계

① 인간이 설계한 scaffold에 의존

MASRouter, GPTSwarm 같은 방법은 미리 정해진 topology pool에서 선택. 인간이 상상한 워크플로우 밖으로 못 나간다.

② 단순 router classifier로 환원

RouterDC는 query→best agent로 1대1 매칭. multi-agent collaboration 자체를 못한다.

③ Mixture-of-Agents는 비용 폭발

MoA는 8개 agent를 모두 부른 뒤 aggregator로 합성. 평균 8회 호출, candidate가 부정확하면 misled.

④ 표현력의 본질적 제약

모든 기존 방법은 인간이 정한 routing function 또는 fixed scaffold 구조에 묶임. 자연어가 표현할 수 있는 임의의 워크플로우를 카버하지 못한다.

배경 지식: 알아두면 좋은 6가지 개념

1) LLM 워커 (Worker Agent)

GPT-5, Claude, Gemini, Qwen3 같은 강한 LLM 한 명. 각자 도메인이 다르고 가격도 다르다. API 호출로 사용.

2) Multi-Agent 시스템

여러 LLM이 협업해서 한 문제를 푸는 시스템. 인간 팀처럼 기획자/구현자/검증자 역할 분담.

3) Routing

"이 질문은 어떤 LLM에 보낼까?"를 결정하는 classifier. 가장 간단한 multi-agent 방식이지만 한 명만 선택 가능.

4) Topology (협업 구조)

여러 agent가 누가 누구와 정보를 주고받는지의 구조. Chain (A→B→C), Tree (A,B → C), 또는 더 복잡한 그래프.

5) GRPO (Group Relative Policy Optimization)

DeepSeek-R1이 사용한 RL 방법. 한 문제에 여러 답을 만들어 group 내 상대 우수성을 reward로. 본 논문도 같은 알고리즘.

6) Prompt Engineering

LLM에게 어떻게 질문할지를 잘 설계하면 같은 모델로도 더 나은 답이 나온다. Conductor는 이걸 자동으로 한다.

핵심 아이디어: 자연어로 워크플로우를 설계하는 RL Conductor

Conductor는 LLM이다. 그러나 user 질문에 직접 답하지 않는다. 대신 다른 LLM들에게 어떻게 답할지를 자연어로 지시한다.

▲ Conductor의 동작: (subtask 텍스트, model id, access list) 트리플 시퀀스를 자연어로 출력 → 실행 → reward로 학습.

Output의 3가지 요소

1Subtask 텍스트

그 step에서 worker가 할 일을 자연어로 구체적으로 지시. "Develop algorithm…", "Verify the previous solution…"

2Model ID

어느 워커를 부를지. 0~6의 정수 (총 7개 worker).

3Access List

이전 step 중 어느 것의 답을 이 worker에게 보여줄지. [] (empty), ["all"] (전체), 또는 특정 index만.

🎼 비유: 오케스트라 지휘자와 같다. 지휘자는 베이스도, 첼로도, 바이올린도 직접 안 친다. 그러나 누가 언제 들어오고, 어떤 음량과 표정으로 연주할지, 누가 누구의 멜로디를 듣고 화답할지를 지휘봉 하나로 만든다. 단원들이 아무리 명연주자여도 지휘자가 없으면 100명이 따로 노는 소음일 뿐이다.

Conductor가 모든 topology를 표현할 수 있는 이유

3-tuple (model_id, subtask, access_list)의 sequence는 놀랍게도 모든 일반적 워크플로우를 표현할 수 있다.

모든 구조를 동일 형식으로 표현 access_list가 어떤 정보를 다음 step에 노출할지 결정 = 정보 흐름 그래프 자유롭게 설계
▲ 같은 3-tuple 시퀀스 형식이 모든 topology를 표현한다. access_list가 정보 흐름을 결정.
📜 비유: 음악 악보 기호도 결국 음표(어떤 음), 악기 지정, 연주 지시 (forte, piano)의 3요소가 시퀀스로 나열되는 것. 그 단순한 형식이 베토벤 9번 교향곡까지 표현한다. 마찬가지로 (model_id, subtask, access_list)의 3-tuple이 모든 LLM 협업 패턴을 표현한다.

학습 방법: 단순한 RL 레시피

Conductor의 학습은 놀랍도록 단순하다 — DeepSeek-R1의 RL 레시피를 그대로 가져온다.

1Base 모델

Qwen2.5-7B에서 시작.

2Worker pool 준비

Gemini, Claude, GPT-5, DeepSeek-R1, Gemma3, Qwen3 등 7개 API/오픈 worker.

3학습 데이터 960문제

MATH500, MMLU, RLPR, LiveCodeBench V1에서 4개 도메인 다양하게 선별.

4GRPO 학습

한 문제에 64개 rollout, 200 iteration, batch 256. KL regularization 없음.

5Reward 단순

format 실패: 0, 오답: 0.5, 정답: 1.

핵심 디자인: 자연어가 output

다른 모든 multi-agent 방법이 classifier 또는 graph 구조를 학습하는 반면, Conductor는 LLM 자체가 자연어로 워크플로우를 출력. 이 차이가 만드는 expressivity 폭이 결정적.

학습 중 발현하는 패턴들

▲ Iteration이 진행되면서 단순 호출 → 다단계 분해 → 정교한 collaboration까지 자연 발현. 명시적 supervision 없음.

흥미로운 발견 4가지

① 학습 데이터의 OOD few-shot이 더 효과적

Counter-intuitive: 학습 시 in-distribution 예시보다 out-of-distribution 예시를 few-shot으로 주면 더 좋은 결과. M500 88.2 → 89.33, LCB 58.32 → 64.29. OOD example이 lazy repetition reward-hacking을 막고 exploration을 유도.

② 모델 크기가 prompt engineering 품질을 결정

3B vs 7B Conductor는 agent selection은 동일. 그러나 7B가 더 정교한 subtask 문자열을 만들어 LCB에서 큰 격차. base LLM의 자연어 능력이 직접 coordination quality로 변환.

③ Frontier model을 Conductor로 써도 학습된 7B가 우월

GPT-5/Gemini를 zero-shot prompted Conductor로 쓰면 worker 능력에 대한 잘못된 가정(open-source over-reliance) 때문에 학습된 7B Conductor에 못 미친다. 학습이 worker capability landscape를 학습시킨다.

④ Task complexity별 step 수 자동 조절

LiveCodeBench (어려운 코딩): 3–4 step. MMLU (사실 회상): 1–2 step. 난이도가 높을수록 더 많은 compute를 자동 할당. emergent task adaptivity.

확장: 재귀적 Conductor와 새로운 test-time scaling

Conductor 자신을 worker pool에 포함시키면 자기 자신을 부를 수 있는 재귀가 가능해진다.

▲ Conductor가 자기 답을 본 뒤 worker를 다른 모델로 재분배해 개선. BigCodeBench에서 +2.2점.

Table 2의 결과: BigCodeBench 37.8 → 40.0 (+2.2), GPQA-D 81.31 → 82.32 (+1.0). 분석해보면 BigCodeBench에서 GPT-5가 strangely 약함을 Conductor가 첫 번째 시도에서 발견하고, 재귀 호출에서 Claude/Gemini로 worker 재분배.

🔄 비유: 작가가 초고를 쓰고 자기가 다시 읽어보며 고쳐쓰는 과정. 첫 시도에는 평소 쓰던 stylebook(GPT-5 의존)을 따르지만, 결과를 보고 "이 부분은 시인의 손길이 더 낫겠다"(Claude로 전환)고 판단해 재할당.

한계점

① API 비용 의존

Worker pool이 GPT-5, Claude, Gemini 같은 expensive API를 포함. Conductor 자체는 저렴하지만 inference 단가는 worker 호출 비용에 비례. open-only subset도 작동하지만 절대 성능 gap이 존재.

② Worker ceiling에 종속

Conductor 성능 한도는 worker 조합의 best 가능치. worker가 약하면 Conductor도 한계가 있다.

③ Verifiable task에만 적용

Math, code, MMLU 같은 binary 정답이 있는 task에서만 reward 정의 가능. open-ended creative writing이나 multi-modal task는 reward shaping이 추가로 필요.

④ API 모델의 시간적 가변성

worker가 GPT-5 같은 closed API이므로 시간에 따라 진화/변경. 재현성이 long-term challenge.

⑤ Domain transfer 미검증

training data가 math/code/MMLU/RLPR 4개 도메인만. translation, planning, web agent로의 domain transfer는 추후 검증 필요.

활용 분야

▲ Conductor 패러다임의 응용 영역.

한 줄 결론

Frontier LLM의 성능을 끌어올리는 두 가지 방법:
① 더 크고 더 비싸게 학습시킨다 (지금까지의 방법).
② 그들을 똑똑하게 조합할 작은 지휘자를 학습시킨다 (Conductor).
7B Conductor가 GPT-5/Claude/Gemini를 모두 능가했다 — 2장의 H100으로.

가져갈 인사이트 3가지

  • "Collective intelligence는 single-model scaling의 진정한 대안": 더 큰 모델을 만드는 대신 기존 모델을 더 똑똑하게 조합하면 frontier를 돌파할 수 있다. 비용/접근성 측면에서 학원 연구의 새 가능성.
  • "자연어가 ultimate output medium": classifier나 graph 같은 구조화된 output space는 expressivity 한계. 자연어로 워크플로우 자체를 출력하는 것이 임의의 topology를 표현하는 유일한 방법.
  • "Coordination strategy는 emergent capability": 명시적으로 가르치지 않은 planner/verifier/refiner 패턴이 end-to-end reward maximization만으로 자연 발현. RL이 LLM에게 가르칠 수 있는 것의 범위가 prompt engineering까지 확장된다는 발견.
