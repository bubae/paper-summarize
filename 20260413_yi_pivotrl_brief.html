먼저 알아야 할 것: AI 에이전트란?

PivotRL을 이해하려면, "AI 에이전트"가 무엇인지, 그리고 AI를 훈련하는 두 가지 방법(SFT와 RL)이 왜 각각 한계가 있는지를 먼저 알아야 합니다.

개념 1: AI 에이전트 (Agentic AI) — 도구를 사용하는 AI

ChatGPT에게 "오늘 날씨 알려줘"라고 물으면 한 번에 대답합니다. 하지만 "이 코드의 버그를 찾아서 고쳐줘"라고 요청하면, AI는 파일을 읽고, 코드를 실행하고, 에러를 확인하고, 수정하고, 다시 테스트하는 등 여러 단계에 걸쳐 환경과 상호작용해야 합니다. 이렇게 도구와 환경을 직접 조작하며 복잡한 작업을 수행하는 AI를 AI 에이전트라고 부릅니다.

AI 에이전트는 한 번의 대답이 아니라, 여러 단계에 걸쳐 환경과 상호작용하며 문제를 해결한다
비유: AI 에이전트 = 신입 사원

일반적인 AI(ChatGPT 등)가 "질문에 답변하는 상담원"이라면, AI 에이전트는 "직접 컴퓨터 앞에 앉아서 업무를 처리하는 신입 사원"과 같습니다. 신입 사원은 이메일을 읽고, 프로그램을 실행하고, 보고서를 작성하는 등 여러 단계를 스스로 진행해야 합니다. 이런 AI 에이전트를 잘 훈련시키는 것이 이 논문의 주제입니다.

개념 2: AI를 훈련하는 두 가지 방법 — SFT와 RL

AI 에이전트를 잘 훈련시키려면, "정답 예시를 보여주는 방법"과 "스스로 시행착오를 겪게 하는 방법"이 있습니다. 각각 SFT (Supervised Fine-Tuning, 지도 학습 미세조정)와 RL (Reinforcement Learning, 강화학습)이라 부릅니다.

SFT는 저렴하지만 기존 능력을 잃고, E2E RL은 능력을 보존하지만 비용이 매우 높다
비유: 수학 과외의 딜레마

SFT는 선생님이 풀이를 보여주고 학생이 그대로 따라 적는 것입니다. 빠르지만, 새로운 유형의 문제가 나오면 대처하지 못합니다. 게다가 수학 문제만 연습하다 보면 이전에 잘 하던 영어까지 까먹게 됩니다 (catastrophic forgetting).

E2E RL은 학생이 직접 시험을 처음부터 끝까지 치르며 배우는 것입니다. 실력은 확실히 늘지만, 매번 긴 시험을 반복해야 하므로 시간과 비용이 엄청나게 듭니다.

핵심 딜레마: 저렴하면 까먹고, 까먹지 않으려면 비싸다!

개념 3: OOD 성능 — "배우지 않은 것도 잘 해야 한다"

OOD (Out-of-Domain, 도메인 외)란, AI가 훈련받지 않은 영역에서의 성능을 의미합니다. 예를 들어, "코딩"을 집중 훈련했을 때, 이전에 잘 하던 "수학 문제 풀기"나 "번역" 능력이 얼마나 유지되는가를 측정합니다.

좋은 훈련 방법이란 In-Domain 성능은 올리면서 OOD 성능은 떨어뜨리지 않는 것

PivotRL의 핵심 아이디어

PivotRL은 두 가지 핵심 혁신으로 이 딜레마를 해결합니다: (1) "유의미한 연습 문제"만 선별하는 Pivot Filtering과 (2) "결과가 같으면 정답으로 인정"하는 Functional Reward입니다.

전체 흐름: 3단계 파이프라인

핵심 아이디어 1: Pivot Filtering — "풀 수도, 못 풀 수도 있는" 문제만 골라라

AI 에이전트의 작업 과정(trajectory)은 여러 개의 행동 단계(turn)로 이루어집니다. 그런데 모든 단계가 훈련에 유용한 것은 아닙니다. 연구진이 실제로 분석해보니, 무작위로 선택한 단계의 71%가 훈련에 아무 도움이 되지 않았습니다.

전체 turn 중 29%만이 훈련에 유의미한 "Pivot" — 이것만 선별하면 비용 대비 효과가 극대화
비유: 과외 선생님의 문제 선별

수학 과외를 생각해 보세요. 학생이 이미 100% 풀 수 있는 문제(1+1=?)를 연습해도 실력이 늘지 않습니다. 반대로, 절대 못 푸는 문제(대학원 수준 미적분)를 연습해도 좌절만 합니다. 실력 향상에 가장 효과적인 것은 "풀 수도 있고 못 풀 수도 있는" 경계선의 문제입니다. PivotRL은 이런 문제만 골라서 집중 훈련합니다.

핵심 아이디어 2: Functional Reward — "결과가 같으면 정답"

기존의 단순한 접근(Naive Local RL)에서는 AI의 답변이 모범답안과 글자 하나하나 완전히 일치해야만 정답으로 인정했습니다. 하지만 현실에서는 같은 결과를 만드는 방법이 여러 가지입니다.

Functional Reward는 "결과의 동등성"을 기준으로 채점하여, 다양한 올바른 행동을 인정한다
비유: 시험 채점 방식

Exact Match는 "답안지의 글씨체까지 모범답안과 동일해야 정답"이라고 채점하는 것입니다. 답이 맞아도 다르게 썼으면 오답 처리합니다.

Functional Reward는 "최종 답이 맞으면 풀이 과정이 달라도 정답"이라고 채점하는 것입니다. 이렇게 하면 AI가 다양한 전략을 시도해볼 수 있고, 더 풍부한 학습 신호를 받게 됩니다.

왜 PivotRL은 기존 능력을 보존할까?

PivotRL의 Functional Reward에는 중요한 수학적 성질이 있습니다: AI 모델이 "올바른 행동"의 확률을 높이면서도, 나머지 행동들 사이의 상대적 순서는 바꾸지 않습니다. 즉, 코딩 관련 행동만 재배치하고, 수학이나 번역과 관련된 행동의 순서는 그대로 둡니다.

PivotRL은 "올바른 코딩 행동"의 확률만 높이고, 수학/번역 등 다른 영역의 행동 비율은 거의 보존한다

각 구성 요소는 얼마나 중요한가?

PivotRL의 두 핵심 요소(Pivot Filtering + Functional Reward)를 하나씩 제거하면 성능이 어떻게 변하는지 확인합니다.

Functional Reward(+6.47)가 Pivot Filtering(+4.13)보다 더 큰 기여를 하지만, 두 요소 모두 필수적

핵심 발견: Functional Reward 없이 Pivot Filtering만 사용하면(Exact Match), SFT보다도 성능이 낮습니다 (57.34 vs 58.44). 두 구성 요소가 함께 작동할 때만 최고 성능(63.81)을 달성합니다.

한계점

SWE-Bench에서 SFT 대비 열세: 4개 도메인 중 Software Engineering(SWE-Bench)에서만 PivotRL(32.67)이 SFT(37.40)보다 낮은 성능을 보였습니다. 이 도메인에서는 E2E RL이 여전히 우위에 있어, 모든 도메인에서 만능은 아닙니다.

도메인별 검증기(Verifier) 필요: Functional Reward를 사용하려면 각 도메인마다 "결과가 올바른지" 판정하는 검증기를 별도로 설계해야 합니다. 현재는 규칙 기반 검증기만 사용하며, LLM을 활용한 범용 검증기는 아직 미래 과제입니다.

단일 모델에서만 검증: 실험이 Qwen3-30B 모델 하나에서만 수행되었습니다. 다른 크기나 종류의 모델에서도 동일한 효과가 나타나는지는 아직 확인되지 않았습니다.

Pivot 선별이 정적: 훈련 시작 전에 한 번 Pivot을 선별하면, 이후 AI 모델이 발전해도 Pivot 목록이 업데이트되지 않습니다. 학생의 실력이 올라가면 문제 난이도도 함께 조정해야 하는데, 현재는 그런 동적 조정이 없습니다.

실험 반복 미보고: 결과가 단일 실험 run을 기반으로 하며, 여러 번 반복했을 때의 편차(표준편차)가 보고되지 않았습니다.

비유: 과외 선생님의 한계

PivotRL은 마치 "첫날 레벨 테스트로 문제를 선별한 뒤 같은 문제 세트만 계속 내주는 과외 선생님"과 같습니다. 학생이 성장하면 문제도 바뀌어야 하는데, 현재는 첫날의 선별이 끝까지 고정됩니다. 향후 "실시간으로 문제 난이도를 조정하는" 동적 Pivot 선별이 개선 방향입니다.
