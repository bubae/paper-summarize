TL;DR

  • 문제: LLM은 few-shot 능력이 뛰어나지만 ##산술 계산, 사실 검색, 최신 정보, 날짜 인지##에서 작은 specialized model보다 못함. 기존 tool-use 접근은 (a) 막대한 human annotation 필요하거나 (b) task-specific setup에 한정.
  • 해결: ##Self-supervised 방식으로 LLM 자체가 tool API call 시점·인자·통합 방법을 학습##. 핵심 아이디어: in-context learning으로 후보 API call 대량 생성 → 실제 호출 → ##해당 호출이 future token loss를 줄이는지 perplexity 기반 filtering## → 통과한 호출만 inline 삽입한 데이터셋으로 fine-tuning.
  • 왜 작동하는가: 정답 라벨이 아니라 ##LM 자체의 perplexity##가 "useful API call"의 자기지도 신호. 무엇이 사람에게 유용한지가 아니라 ##무엇이 모델에게 useful한지##를 직접 측정.
  • 비유: 평범한 학생에게 "계산기, 사전, 검색 엔진"을 책상에 놓고 ##본인이 풀기 어려운 문제일 때 알아서 도구를 집어 들도록## 가르치는 것. 단 "어떤 문제에 어떤 도구를 써라"고 알려주지 않고, ##학생이 푼 문제의 풀이 시간을 줄여주는 도구 사용 패턴만 보상##.

핵심 기여 (Key Contributions)

  1. Self-supervised tool-use 학습 패러다임: human annotation 없이, ##LM perplexity가 자체 라벨##. 어떤 API call이 모델에게 유용한지 모델 자신이 결정.
  2. 3-step 데이터 증강 파이프라인: (a) in-context prompt로 후보 API call 위치/내용 샘플링, (b) 실제 API 실행, (c) ##perplexity-based filtering##으로 유용한 call만 유지.
  3. 다양한 도구 통합: Question Answering (Atlas), Calculator, Wikipedia Search (BM25), Machine Translation (NLLB 600M), Calendar 5종 도구를 단일 모델이 자율적으로 선택.
  4. 강력한 zero-shot 성능: GPT-J 6.7B Toolformer가 ##GPT-3 175B을 능가##. LAMA SQuAD 17.8 → 33.8, MAWPS math 9.9 → 44.0, NQ 12.8 → 17.7, TempLAMA 13.7 → 16.3. core LM 능력은 보존.
  5. Scaling law of tool use: tool 활용 능력은 ##~775M parameter##부터 emerge. 그 이하 모델은 도구 제공해도 성능 향상 없음.

1. 문제 정의 (Background)

1.1 LLM의 inherent limitations

Brown et al. 2020(GPT-3), Chowdhery et al. 2022(PaLM)이 LLM의 few-shot 능력을 입증했지만 다음 한계는 ##scaling으로 해결되지 않음##:

  • 최신 정보 접근 불가 (Komeili et al. 2022)
  • 사실 hallucination (Maynez et al. 2020, Ji et al. 2022)
  • 저자원 언어 이해 부족 (Lin et al. 2021)
  • 정밀 계산 불가 (Patel et al. 2021)
  • 시간 인지 부족 (Dhingra et al. 2022)

1.2 기존 tool-use 접근의 한계

  • Heavy supervision 접근 (WebGPT, Komeili et al., Thoppilan et al.): 사람이 직접 도구 사용 데모를 annotation. 비용 비싸고 ##사람이 유용하다고 생각하는 것 ≠ 모델에게 유용한 것##.
  • Task-specific 접근 (PAL, TALM): 특정 task에 hard-coded prompt. ##generality 손실##, 자유 응답에서 사용 어려움.

1.3 Toolformer의 desiderata

두 가지 목표:

  1. Self-supervised tool 학습: human annotation 없이. 사람이 유용하다고 판단하는 것이 아니라 ##모델이 useful하다고 판단하는 것##을 신호로.
  2. Generality 보존: LM이 ##언제 어떤 도구를 쓸지 스스로 결정##. 사전 설정된 task에 한정되지 않음.

2. 방법 (Approach)

2.1 API call 표현

API call을 tuple $c = (a_c, i_c)$로 표현 ($a_c$: API 이름, $i_c$: 입력). result $r$이 있을 때:

$$e(c) = \texttt{} a_c(i_c) \texttt{}$$ $$e(c, r) = \texttt{} a_c(i_c) \to r \texttt{}$$

실제로는 token vocabulary 수정 없이 "[", "]", "->" 같은 일반 토큰 시퀀스로 매핑.

2.2 3-Step Data Augmentation Pipeline (Figure 2)

[Figure 2 설명]: 입력 텍스트 $\mathbf{x}$ = "Pittsburgh is also known as ___ the Steel City". (1) ##Sample API Calls##: position $i$ 선택 후 후보 호출 $c_i^1, \ldots, c_i^k$ 생성 (예: "What other name is Pittsburgh known by?", "Which country is Pittsburgh in?"). (2) ##Execute API Calls##: 실제 호출해서 $r_i^1$="Steel City", $r_i^2$="United States" 결과 획득. (3) ##Filter API Calls##: $L_i(c_i^1 \to \text{Steel City}) < \min(L_i(c_i^1, \varepsilon), L_i(\varepsilon))$인 call만 유지. 통과한 호출만 inline 삽입해 $\mathbf{x}^*$ 생성.

Step 1: Sampling API Calls

Few-shot prompt $P(\mathbf{x})$로 LM이 annotation 생성. Position $i$의 시작 확률:

$$p_i = p_M(\texttt{} | P(\mathbf{x}), x_{1:i-1})$$

Threshold $\tau_s$ 이상 위치 set $I = \{i | p_i > \tau_s\}$ 선정 (최대 $k$개). 각 위치마다 최대 $m$개 후보 API call $c_i^1, \ldots, c_i^m$ 샘플링.

Step 2: Executing API Calls

각 호출 $c_i$를 실제 실행해 single text $r_i$ 획득. 도구별로 Python script, retrieval, MT 모델 등.

Step 3: Filtering API Calls (핵심)

가중 cross-entropy loss:

$$L_i(\mathbf{z}) = -\sum_{j=i}^n w_{j-i} \cdot \log p_M(x_j | \mathbf{z}, x_{1:j-1})$$

여기서 $\mathbf{z}$는 prefix sequence, $w$는 position weight. 두 비교:

$$L_i^+ = L_i(e(c_i, r_i))$$ $$L_i^- = \min(L_i(\varepsilon), L_i(e(c_i, \varepsilon)))$$

$L_i^+$: API call과 결과 모두 prefix. $L_i^-$: (a) API call 안 함, (b) API call하되 결과 제공 안 함 중 ##더 작은 loss##. 직관적으로 말하면, ##API call의 결과가 future token 예측에 도움이 될 때만 $L_i^+ < L_i^-$##.

Filtering criterion:

$$L_i^- - L_i^+ \geq \tau_f$$

이를 통과한 call만 유지. ##단 결과 r이 token 예측 loss를 충분히 줄여야 함##.

2.3 Model Finetuning

필터링 후 남은 API call들을 원본 텍스트에 inline 삽입:

$$\mathbf{x}^* = x_{1:i-1}, e(c_i, r_i), x_{i:n}$$

이를 통해 ##원본 $\mathcal{C}$와 동일 content## + ##API call 삽입##된 $\mathcal{C}^*$ 데이터셋 구성. Standard LM objective로 fine-tuning. ##원본 content가 유지되므로 core LM 능력 보존##.

2.4 Inference

Regular decoding 진행 → "$\to$" token 생성 시 ##decoding 중단 → API 호출 → response 삽입 → "$$" token 추가 후 decoding 재개##.

2.5 알고리즘 흐름

[Data construction phase] Input: dataset C = {x^1, …, x^|C|}, tools T For each x in C: For each tool t in T: 1. Sample k positions i with p_i > τ_s 2. At each position, sample m candidate API calls 3. Execute each candidate → get result r 4. Compute L_i^+ (with result) vs L_i^- (without) 5. Keep calls where L_i^- - L_i^+ ≥ τ_f Merge filtered calls inline → x* Output: augmented dataset C* with API calls [Training phase] Fine-tune LM M on C* with standard LM objective [Inference phase] Regular autoregressive decoding When "→" token emitted: pause → call API → insert response → resume

4. 실험 (Experiments)

4.1 Setup

  • Base LM: GPT-J 6.7B (Wang & Komatsuzaki 2021)
  • Dataset: CCNet subset (Wenzek et al. 2020)
  • Training: batch 128, lr 1e-5, linear warmup 10%, $w_t = \max(0, 1 - 0.2 \cdot t)$
  • Heuristics: 계산기 도구 후보는 ##최소 3개 숫자 포함 텍스트##만 고려 (annotation cost 절감)
  • Comparison: GPT-J + CC (no API), Toolformer, Toolformer (disabled — inference 시 API token prob 0), OPT 66B, GPT-3 175B

5. 종합 평가

강점

  • 패러다임 정의: ##Tool-use LLM의 시대를 연 foundational paper##. 이후 ReAct, Toolllama, Gorilla, LangChain, GPT-4 plugins, ChatGPT tool use 모두 이 lineage. ChatGPT의 plugin 시스템이 사실상 Toolformer의 production 구현.
  • Self-supervised의 우아함: human annotation 0개. ##LM perplexity가 자체 라벨##이라는 통찰. "유용한 도구 호출"의 정의를 사람의 직관에서 모델의 효용으로 옮김.
  • Filtering criterion의 수학적 명료성: $L_i^- - L_i^+ \geq \tau_f$ 한 식으로 도구 효용 측정. 두 reference loss(no call vs call without result)의 min을 비교군으로 잡는 디자인이 ##엣지 케이스 처리에 robust##.
  • Generality 입증: 5가지 도구를 단일 모델이 자율 선택. ##LAMA QA tool 98%, math calculator 97.9%, multilingual MT 63~94%##로 ##task-tool mapping을 모델이 학습##함을 보임.
  • Core LM 보존: API disabled perplexity가 baseline과 거의 동일. ##finetuning이 일반 능력 깎지 않음##.
  • Scaling law of tool use 발견: 775M emergence threshold. ##future agent 연구의 모델 선택 기준##.
  • Negative result honesty: GPT-3 미달(QA, MLQA), sample inefficient(million docs → thousand calls), prompt sensitivity 등 한계를 명시적으로 보고. ##정직한 과학##.

한계 및 의문점

  • Tool chaining 불가: 한 tool 결과를 다른 tool에 feed하는 multi-step 사용 못함. annotation이 independent하기 때문. ##Calendar로 오늘 날짜 얻고 QA에 넘기는 패턴이 자연스러우나 학습 불가##. 후속 ReAct, Toolllama가 해결.
  • Interactive 사용 불가: Search tool 결과가 부족할 때 ##query 재구성, multi-page browsing 불가##. ##WebGPT 스타일이 안 됨##. Toolformer의 search engine 단점의 근본 원인.
  • Sample inefficiency: million docs → thousand calls (calculator). API token sampling rate가 매우 낮음. 후속 연구에서 prompt engineering으로 일부 완화.
  • Prompt sensitivity: LM의 일반적 약점인 prompt wording에 매우 sensitive. zero-shot setting에서 task description 표현에 따라 성능 큰 변동.
  • 도구 cost 미고려: 호출 의사결정에 ##API의 latency/cost 반영 안 됨##. perplexity 이득이 작아도 calling이 학습될 수 있음. production에선 cost-aware 필요.
  • 도구별 heuristic 의존: calculator는 "≥3 numbers" heuristic으로 후보 텍스트 좁힘. ##모든 도구를 동일하게 처리하지 못함##.
  • MLQA 부분 실패: finetuning data가 영어 위주여서 multilingual 성능이 base 대비 떨어지는 언어 존재. fundamental data balance 문제.
  • QA에서 GPT-3 175B 미달: 본문이 정직하게 보고. ##간단한 BM25 + 모델 크기 제약##이 원인. scale로 부분 해결 가능.
  • LLaMA 7B에서 도구 효용 감소 (Appendix G): stronger base에서 Wikipedia search 도구의 utility가 ##vanish##. ##도구 강도 < 모델 강도가 되면 tool 비활성화이 더 빠름##. 후속 연구의 도구 선정에 중요한 시사.
  • 2023 시점의 한계: GPT-4(2023), Claude 등의 후속 모델은 이미 도구 사용을 base에서 더 잘함. Toolformer 이후 ChatGPT plugins 등 production 시스템이 등장하면서 ##연구 가치는 historical##.

6. Key Takeaways

  1. "LM perplexity가 라벨이다": 자기지도 학습의 신호로 LM 자체의 효용을 사용하는 아이디어는 ##chain-of-thought self-improvement, RLAIF 등 후속 self-improvement 연구##에 직접 영향.
  2. Tool use는 emergent capability: 775M parameter threshold. ##sub-Billion 모델에는 도구 줘봐야 소용없음## — 이후 agent 연구의 모델 선택 기준.
  3. "What the model finds useful ≠ what humans find useful": Tool annotation의 패러다임 전환. 사람이 라벨링하는 게 아니라 모델이 스스로 발견. ##RLHF/instruction tuning에서도 적용 가능한 원리##.
  4. Toolformer의 limitations이 후속 연구의 로드맵: chaining → ReAct, interactivity → AutoGPT/BabyAGI, sample efficiency → ToolBench/Gorilla, cost-aware → economic tool selection.
  5. Generality 보존이 production 가치: API disabled 시 perplexity 보존이라는 negative result는 ##LLM의 multi-purpose nature를 잃지 않는다##는 deployment guarantee. 이 패러다임이 ChatGPT plugins, GPT-4 function calling으로 직결.
