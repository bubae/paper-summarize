TL;DR

  • 문제: 기업이 HIPAA/ITAR/변호사-의뢰인 비밀 등 규제 환경에서 LLM을 쓰려면 (a) 데이터를 평문으로 클라우드 API에 보내거나, (b) 동형암호(HE)/MPC/TEE 같이 100~1000배 느려지는 보안 기법을 써야 한다. WAN(광역망) 환경에서 토큰별 round-trip 비용이 80~100ms에 달해 split inference도 8~11 tok/s밖에 안 나온다.
  • 해결: 트랜스포머의 ##임베딩·언임베딩 층은 로컬, 중간 층은 클라우드##로 분할하여 raw 토큰이 네트워크를 떠나지 않게 하고, ##lookahead decoding을 WAN split inference에 처음 적용##해 매 round-trip당 1.2~1.6 토큰을 통과시켜 8.7~9.3 tok/s (Mistral 7B)와 7.8~8.7 tok/s (NeMo 12B)를 달성한다. 기본 split (2 local layers) 시 ~59% 토큰이 inversion 공격에 복원되나, 8 local layers로 늘리면 ~35%로 감소 (throughput 영향 미미).
  • 왜 작동하는가: WAN RTT(~80ms)가 클라우드 forward pass(~16~21ms)의 3~5배라서 GPU가 네트워크 대기로 idle한다. Jacobi 반복으로 여러 후보 토큰을 한 round-trip에 검증하면, 단일 토큰을 처리하는 비용 안에서 여러 토큰이 채택될 수 있다. Greedy argmax 하에서 lookahead 출력은 sequential과 token-identical하므로 품질 손실 0.
  • 비유: 미국-서울 간 비행기에 사람을 한 번에 한 명만 태워 보내면 비효율적이다. 좌석을 다 채워 보내면, 어차피 같은 비행 시간 안에 더 많은 사람이 도착한다. lookahead는 한 round-trip 안에 5개 후보 토큰을 묶어 보내 그중 채택되는 것만 커밋한다.

핵심 기여 (Key Contributions)

  1. 비대칭 layer split의 명시적 동기 부여: 임베딩/언임베딩 층을 로컬에 두고 중간 transformer 층만 클라우드로 보내, ##raw 토큰 ID와 vocabulary 매핑이 신뢰 경계를 절대 넘지 않는다는 architectural privacy 보장##.
  2. Lookahead decoding을 split inference에 처음 적용: Fu et al.의 lookahead decoding[6]을 WAN 환경에 이식하여 ##round-trip latency를 1.2~5 accepted tokens에 amortize##한다.
  3. 경험적 inversion attack 평가: split depth가 privacy-performance trade-off의 조절 노브임을 정량화 — 2 local layers (split after layer 1) 시 ~59% 토큰 복원, 8 local layers (split after layer 7) 시 ~35%, throughput 영향 미미.
  4. n-gram speculation의 통계적 안정성 입증: 7B와 12B에서 acceptance rate 1.21~1.25 tok/step으로 ##모델 크기에 거의 독립적##임을 확인 (코드에서 최대 1.57).
  5. 품질의 형식적 검증: greedy argmax 하에서 lookahead 출력이 sequential과 ##token-identical (zero quality degradation)##임을 4개 prompt × 200 token × 2 모델로 100% 일치 확인.
  6. NeMo 12B 스케일링 검증: 40-layer 12B를 4.9GB 로컬 VRAM으로 운용, 7B와 동일 RTT에서 throughput 일치 (RTT-to-compute 비율의 우호적 변화).
  7. RTT decomposition 모델: per-step time = RTT + fixed_overhead 분해로 <6.2% cross-validation error로 임의의 RTT에서 throughput 투영 가능 (공정한 cross-provider 비교).
  8. 실제 deployment engineering 기록: VAST.ai의 Virginia proxy routing 같은 cloud provider 특유의 함정과 SSH tunnel 엔지니어링을 학술 문헌에서 보기 드물게 명시적으로 보고.

배경 지식 (Background)

Split Computing이란

Split computing은 DNN 추론을 신뢰 경계(edge ↔ cloud)에서 분할 실행하는 기법으로, Neurosurgeon (ASPLOS 2017)이 CNN 분야에서 시작했다. 핵심 trade-off는 다음과 같다:

  • Early split (얕은 곳에서 분할): 로컬 연산량은 적지만, 중간 representation이 raw input과 비슷해 압축이 어렵고 복원 공격에 취약.
  • Late split (깊은 곳에서 분할): 중간 representation이 추상화되어 압축률·privacy가 좋지만, 로컬 연산 부담이 큼.

LLM에 적용된 split computing 선행 연구로는 Sung et al.의 mixed-precision quantization 기반 split, EdgeShard의 dynamic programming 배치, MDI-LLM의 recurrent pipeline parallelism, exo의 P2P 추론 등이 있으나, ##대부분 throughput·resource를 최적화 대상으로 삼고 privacy를 1차 design goal로 두지 않았다##.

Speculative Decoding과 Lookahead Decoding

Speculative decoding (Leviathan et al., Chen et al., 둘 다 2023)은 작은 draft 모델이 후보 토큰을 만들고 target 모델이 병렬 검증하여 2~3× 가속을 얻는다. 그러나 draft 모델을 별도로 유지·배포해야 하는 부담이 있다.

Jacobi decoding (Santilli et al., ACL 2023)은 autoregressive 생성을 ##fixed-point iteration 문제##로 재구성한다. $k$개 미래 토큰 위치를 임의로 초기화하고, transformer를 통과시켜 새로운 $k$ 토큰을 얻기를 수렴까지 반복한다. CLLMs (ICML 2024)는 consistency 학습으로 수렴을 개선했고, Fu et al.의 lookahead decoding [6]은 별도 draft 모델 없이 ##Jacobi 반복 trajectory에서 등장하는 n-gram을 후보로 캐싱##한다.

이 논문에서 "draft"가 의미하는 것은 별도 small draft model이 아니라 ##target model 자체의 과거 Jacobi iteration 상에서 우연히 등장한 n-gram 시퀀스##다. 이러한 후보들이 후속 iteration에서 한 forward pass 내에 검증되면, 채택된 토큰들이 한꺼번에 커밋된다.

왜 WAN 환경이 단일 머신과 다른가

단일 머신에서 lookahead는 ##GPU idle time##을 메우지만, WAN split inference에서는 ##network idle time##을 메우는 역할로 바뀐다. 단일 머신의 $k$-token forward pass는 1-token과 거의 같은 비용 (GPU 병렬성)인 반면, WAN에서는 ##forward pass 비용 자체보다 network round-trip이 5배 이상 큰## 비대칭 구조다. 이 때문에 lookahead의 효과가 1.5~2×에서 더 극대화된다.

Threat Model (위협 모델)

본 논문은 ##semi-honest cloud## (= honest-but-curious)를 가정한다. 클라우드는:

  • 레이어 2-29(7B 기준)의 모델 가중치를 알고 있다.
  • 해당 레이어들을 통과하는 중간 activation을 모두 본다.
  • 임베딩 행렬, LM head 가중치, 레이어 0-1과 30-31의 가중치는 모른다.
  • 계산 자체는 정직하게 수행하나, 입력 정보를 추론하려 시도한다.

핵심 가정: 로컬 레이어 가중치가 ##public weights와 구별##되어야 한다 (예: LoRA fine-tuning). 이 가정이 깨지면 (클라우드가 동일한 공개 모델의 layer 0-1 가중치를 그대로 다운로드 가능하다면) privacy 경계가 크게 약화된다.

Model Inversion 공격 — SipIt 정리

Nikolaou et al. [32]의 SipIt 결과는 ##transformer hidden state가 input token에 대해 injective##임을 증명한다 — 즉 서로 다른 입력은 서로 다른 hidden representation을 만들며, 전체 모델 가중치에 접근 가능하면 hidden state에서 linear time으로 정확한 토큰을 복원할 수 있다. 본 논문이 이 결과를 어떻게 회피하는지가 5.2절의 핵심이다.

System Architecture

레이어 분할 (Layer Partitioning)

핵심 원칙: ##embedding과 unembedding 레이어는 절대 클라우드에 두지 않는다.## 가운데 transformer block만 클라우드로 보내고, 입력/출력 양쪽에 일부 transformer block을 로컬에 둘 수 있다 (privacy depth 조절 가능).

Mistral 7B Instruct (32 layers, hidden_dim=4096):

ComponentLocationLayersPurpose
Token embeddingLocal (3090)-Token IDs → hidden states
Layers 0-1Local (3090)0, 1Initial transformer processing
Layers 2-29Cloud2-29Bulk computation (28 layers)
Layers 30-31Local (3090)30, 31Final transformer processing
RMS norm + LM headLocal (3090)-Hidden states → logits → tokens

Mistral NeMo 12B Instruct (40 layers, hidden_dim=5120, GQA 8 KV heads): 동일한 구조로 layers 2-37이 클라우드.

이 partitioning이 보장하는 것:

  • Raw 토큰이 로컬을 떠나지 않는다: 토큰 ID → 벡터 변환은 로컬에서만.
  • 중간 activation만 네트워크 통과: 4096-dim float16 vector per token (~8KB/token for 7B, ~10KB/token for 12B).
  • 클라우드는 vocabulary mapping에 접근 불가: 임베딩/LM head 가중치가 없어 어떤 activation이 어떤 토큰에 대응하는지 직접 알 수 없다.

Transport Layer

HTTP/JSON, HTTP/binary, WebSocket을 비교한 끝에 ##WebSocket binary protocol## 채택. 프레임 포맷:

[4 bytes: header length][JSON header][tensor bytes (float16)]

JSON header는 tensor shape, prompt/generation 플래그, KV-cache crop 위치, attention mask shape를 담는다. Tensor는 ##base64 없이 raw float16 bytes##로 전송 — base64의 ~33% 팽창과 encode/decode CPU 비용을 회피.

Transport 최적화 단계별 throughput (Mistral 7B):

WebSocket + lookaheadbinary float1613~17 tok/s

KV-Cache 관리

핵심 결정: ##레이어 2-29(클라우드 측)의 KV-cache는 클라우드에서 유지##. 네트워크로 전송하지 않음 — KV-cache 크기는 sequence length에 선형으로 증가하여 곧 대역폭을 압도하기 때문.

HuggingFace의 DynamicCache로 세션별 캐시 관리. Non-prompt generation step에서는 ##새 토큰의 hidden state(~8KB)만 전송##. 세션 ID로 lifecycle 관리, 5분 비활성 시 자동 만료.

Jacobi/lookahead용 추가 기능:

  • Cache cropping: 투기 블록이 거부될 때 체크포인트로 롤백.
  • Cache relocation: n-gram speculation으로 검증 윈도우가 이동할 때 KV entries를 이동.
  • Custom attention masks: 병렬 토큰 검증에 필요한 비표준 causal mask 지원.

중요 발견: ##cloud provider의 네트워킹 아키텍처가 지리적 거리보다 latency를 훨씬 더 좌우한다.## Texas의 동일한 RTX 4090 GPU에 대해:

Tunnel 아키텍처: 로컬 GPU 서버는 NAT 뒤에 있고 클라우드 인스턴스 IP는 일시적이므로, 로컬에서 클라우드로 SSH tunnel을 열어 HTTP health-check 포트와 WebSocket 포트를 forward한다. 이는 ##학술 문헌에서 거의 다루지 않는 deployment 현실##이지만 public cloud 인프라에서는 필수.

Decoding Strategies for Split Inference

Sequential Decoding (Baseline)

매 토큰마다 한 번의 완전한 local-cloud-local round trip:

1. Local: Embed token, process through layers 0-1 → hidden state (8KB) 2. Network: Send hidden state + position embeddings to cloud 3. Cloud: Process through layers 2-29 with KV-cache → hidden state (8KB) 4. Network: Return hidden state to local 5. Local: Process through layers 30-31, norm, LM head → next token

~85ms RTT (HyperStack)에서 7.8 tok/s, ~80ms RTT (RunPod)에서 10.9 tok/s.

Jacobi Parallel Decoding

$k$개 미래 토큰 위치를 초기화하고 수렴할 때까지 반복. 매 iteration에서 $k$ 위치를 한 forward pass로 병렬 처리. Effective throughput:

$$\text{tok/s} = \frac{k}{\text{iterations} \times \text{RTT}}$$

직관적으로 말하면, $k$개 토큰을 동시에 다듬어 가다가 수렴하면 $k$개를 한꺼번에 커밋한다. 그러나 WAN에서 ##vanilla Jacobi는 수렴이 나빠 iteration당 1-2 토큰만 커밋##되어 ~1.5 tok/s — sequential보다도 느리다 (큰 텐서 전송 오버헤드 때문).

Lookahead Decoding (논문의 주요 모드)

Lookahead decoding [6]은 Jacobi 수렴 문제를 ##trajectory 자체에서 n-gram 후보를 수확##하여 해결. Jacobi iteration 도중 등장하는 토큰 시퀀스를 ##n-gram cache##에 보관하고, 이후 iteration에서 매칭되면 1-5 토큰을 한 round trip에 커밋.

왜 split inference에서 특히 효과적인가:

  1. 매 round trip이 비싸다 (~100ms vs ~3ms locally) — 다중 토큰에 amortize하면 효과 극대.
  2. 검증 비용이 미미하다: 5 candidate 검증 비용 = 1 토큰 생성 비용 (한 round trip).
  3. 구조적 텍스트에서 n-gram speculation이 자주 성공: 코드, 템플릿, 반복 콘텐츠 — 기업 use case에 흔함.

A100 클라우드 ~80ms RTT에서 8-9 tok/s 달성 (sequential ~8 tok/s 대비), acceptance rate 1.2~1.6 tok/step. RunPod의 낮은 RTT에서는 13~17 tok/s.

Attention Mask Engineering

중요한 구현 디테일: PyTorch의 scaled_dot_product_attention에 is_causal=True를 쓰면 ##query/key 길이를 기준으로 relative causal mask를 만든다.## KV-cache에 이미 커밋된 토큰이 있으면 이 relative mask가 잘못된 attention 패턴을 생성 — speculative position $i$가 attend하면 안 되는 KV entries에 attend하게 됨.

해결: 명시적 attention mask 구성:

attn_mask = torch.full(
    (1, 1, k, kv_len), float('-inf'),
    device='cuda', dtype=torch.float16)
for i in range(k):
    attn_mask[0, 0, i, :committed_len + i + 1] = 0.0

이 mask를 WebSocket 프레임에 함께 전송 (~2KB 추가 대역폭) — 작은 비용으로 ##catastrophic attention 오류를 예방##.

Privacy Analysis

Threat Model 상세

Semi-honest cloud 가정. 클라우드가 ##모델 가중치를 안다는 것## 자체가 큰 정보 — Nikolaou et al. [32]의 SipIt는 ##full weights가 있으면 hidden state에서 정확한 토큰 복원이 linear time 가능##함을 증명. 본 논문이 이를 회피하는 메커니즘이 5.2절의 핵심.

Malicious cloud (활성 공격)는 ##scope 밖## — 이를 막으려면 verifiable computing이나 다중 provider redundancy가 필요.

Privacy Properties and Limitations

What is guaranteed: Raw 토큰 ID와 token embedding이 로컬을 떠나지 않는다 — ##structural property##이지 statistical property가 아니다.

What is NOT guaranteed: SipIt 결과에 의해 hidden state가 input에 대해 injective. 다만 본 논문 setting은 다음 요소들로 완화:

  1. Weight access requirement: SipIt 공격은 input ↔ observation 사이의 모든 레이어 가중치를 요구. 본 시스템에서 클라우드는 layer 0-1, embedding matrix, LM head, 그리고 결정적으로 ##로컬 레이어에 적용된 임의의 fine-tuning##에 접근 불가.
  2. Depth of transformation: He et al. [4]은 split depth에 따라 inversion 정확도가 감소함을 보임. 본 논문은 이를 ##직접 측정##:
Split After LayerLocal LayersTop-1 AccuracyTop-5 Accuracy
Layer 1 (default)258.8%62.0%
Layer 3444.3%49.8%
Layer 5644.8%56.6%
Layer 7834.8%46.2%
Random baseline-0.003%0.016%

공격은 3-layer MLP (4096→2048→2048→32000)를 ##~880 (activation, token ID) 페어##로 학습한 결과. 다양한 텍스트(일반 지식, 코드, 의료, 법률, 대화)에서 추출. 이는 ##lower bound##: sequence-level context를 쓰는 더 정교한 공격자나 큰 학습 셋은 더 높은 정확도를 얻을 수 있다.

해석: 기본 2-layer split에서 ##~59% 토큰이 복원 가능## — 이론적 취약성 확인. 8-layer split로 늘리면 ##~35%##로 떨어지지만 0이 되지는 않는다.

  1. Symmetric vulnerability: 출력 측도 동일하게 2 layers (LM head 직전) — 입력과 출력 양쪽에서 동등한 깊이를 두어야 보호가 대칭.

Privacy 강화 방안

  • Local layer 증가: 2→8 layers로 inversion 35% 감소, 추가 비용 layer당 ~3ms (RTT ~100ms 대비 무시 가능).
  • Local layer fine-tuning: LoRA 등으로 layer 0-1 + embedding을 organization-specific하게 수정 — SipIt 공격 벡터 직접 차단. 다만 ##random LoRA perturbation만으로는 부족## (adaptive attacker가 perturbed activations에 retrain 가능); 진정으로 private한 가중치여야 함.
  • Split point randomization: 세션/요청마다 split depth를 무작위로 — 모든 transformer 레이어가 동일한 hidden dim을 공유 (4096)하므로 tensor shape만으로는 알 수 없음. 다만 activation magnitude 분포가 layer별로 다를 수 있어 통계적 식별 가능성은 추가 연구 필요.
  • Calibrated noise: Split-and-Denoise [26]처럼 transmitted activation에 differential privacy 노이즈 — formal $(\epsilon, \delta)$-DP 보장 가능, 품질 trade-off.
  • Activation compression: int8 quantization으로 lossy하게 정보 함량 감소.

관찰: ##네트워크 RTT가 63~64%로 지배##. 12B가 36 cloud layers를 7B의 28 layers 대비 5ms (21 vs 16ms) 더 쓰는 데 그치는 것은 A100 80GB의 컴퓨트 활용도를 보여줌.

RTT projection model: acceptance rate는 RTT 독립적이므로 임의의 target RTT에서 throughput 예측 가능:

$$\text{projected\_tok/s} = \frac{\text{acceptance\_rate}}{(\text{target\_RTT} + \text{fixed\_overhead})/1000}$$

직관적으로 말하면, 네트워크 비용은 줄일 수 있지만 fixed_overhead는 floor — 같은 RTT에서 7B와 12B가 비슷한 throughput을 내는 이유.

Cross-validation error: sequential 1.2~2.4%, lookahead 4.5~6.2%. 12B lookahead는 더 큰 speculative batch 때문에 naive projection을 따르지 않아 measured-adjusted 값으로 보정.

Prefill latency: 초기 prompt 처리는 전체 prompt의 hidden states를 단일 forward pass로 클라우드에 전송 — 7B에서 100-token prompt 기준 ##~800 KB## (8 KB × 100), 12B에서는 ~1 MB (10 KB/token). 단일 round trip에 완료되며 generation per-token throughput에는 영향 없음. 그러나 ##time-to-first-token에 100~500ms 추가##되어 (Discussion §8.3 한계로 명시) — chunked prefill 또는 prefill-decode disaggregation으로 개선 가능.

비율 감소의 메커니즘:

  1. Cloud-side layer 수가 모델 깊이에 선형 증가.

직관적으로 말하면, ##모델이 커질수록 split inference 오버헤드의 비율이 더 작아진다.## 큰 모델일수록 GPU가 network 대기로 idle한 비율이 줄고 compute로 메워진다.

Discussion

Novelty

  1. Privacy 동기 + embedding boundary 분할 + empirical evaluation: 선행 split computing은 latency/energy 최적화 중심. Split-and-Denoise [26]는 embedding에 noise를 추가해 전송하나 ##본 논문은 token-to-vector mapping 자체를 로컬에 유지##하고 post-transformer hidden states만 전송. ##Inversion attack 측정##으로 split depth가 privacy-throughput trade-off의 dial임을 정량화.
  2. Lookahead decoding for network latency amortization: lookahead는 GPU 병렬성을 위해 설계되었으나, ##round-trip 비용이 forward pass보다 30-100× 큰 환경##에서 더 효과적.
  3. Empirical deployment engineering: SSH tunneling, ephemeral IP, cross-machine SSH key 관리 — 학술 split computing 문헌과 deployable systems 사이의 gap을 메움.
  4. Inverse scaling of overhead: 7B와 12B의 measured 결과가 ##모델이 커지면 RTT-to-compute 비율이 좋아짐##을 처음 정량 증명.
  5. RTT decomposition model: <6.2% CV error로 cross-provider 비교와 throughput 투영 가능.

Concurrent Work와의 관계

DSSD (ICML 2025): 가장 가까운 concurrent work — split inference + speculative decoding for edge-cloud. 본 논문과의 차이: (1) 별도 draft 모델 대신 ##lookahead 사용 (추가 모델 불필요)##, (2) privacy를 primary motivation으로 명시, (3) 실제 WAN 측정(시뮬 X).

CROSS-SEC: WAN latency를 layerwise KV-cache overlapping으로 다룸 — 본 접근과 ##complementary##, 결합 가능.

Limitations (저자가 명시)

  • No formal privacy guarantees: architectural privacy ≠ cryptographic. 8-layer split로 ~35% 복원, 0이 아님.
  • Two models tested: Mistral 가족(MistralForCausalLM)만. LLaMA, GPT-NeoX 등 다른 아키텍처에서의 검증 필요.
  • Consumer hardware: RTX 3090 사용. Enterprise는 A6000/L40으로 더 많은 local layer 가능.
  • Lookahead effectiveness varies: code 1.43~1.57, creative 1.06~1.12. ~80ms RTT의 break-even은 1.15~1.20.
  • Prefill latency not optimized: 전체 prompt를 단일 forward pass로 처리 — 100~500ms TTFT 추가. Chunked prefill, prefill-decode disaggregation으로 개선 가능.

[Sequential Decoding] 입력 토큰 → Local: embed + layer 0-1 → hidden(8KB) → [Network ~80ms] → Cloud: layer 2-29 + KV cache → hidden(8KB) → [Network ~80ms] → Local: layer 30-31 + norm + LM head → next token (반복 → ~8 tok/s @ 80ms RTT) [Lookahead Decoding] n-gram pool 초기화 (빈 상태) 반복: Local: 현재 토큰 + 후보 n-gram k개 → embed + layer 0-1 → hidden block → [Network ~80ms] → Cloud: layer 2-29 with custom attention mask → hidden block + KV cache cropping (rejected branches) → [Network ~80ms] → Local: layer 30-31 + LM head → next token + verify candidates → Jacobi trajectory에서 새 n-gram 수확하여 pool에 추가 → 채택된 후보들 commit (1~5 tokens per round trip) → (acceptance rate 1.2~1.6 → 8~13 tok/s @ 80ms RTT)

비판적 분석

강점

  • Engineering 충실도: HTTP → WebSocket → binary float16 → 3090 relay → lookahead까지 1.4 → 17 tok/s의 ##optimization chain을 단계별로 기록##. 학술 논문에서 보기 드물게 실무 deployment에 즉시 도움이 되는 내용.
  • RTT decomposition 모델의 실용성: <6.2% CV error로 임의의 RTT에서 throughput 투영 가능 → provider 선택 및 SLA 계획에 직접 활용 가능. 8.4절의 partnership scaling 시나리오가 단순한 추측이 아니라 ##validated projection##.
  • Privacy claim의 honesty: "architectural privacy"라는 신중한 용어 선택, computational/cryptographic privacy와의 명시적 구분, ##inversion attack을 직접 실행하여 35~59% 복원##됨을 정직히 공개. 학술적 책임 의식이 돋보임.
  • Acceptance rate의 모델 크기 독립성 발견: 7B와 12B가 1.21 tok/step으로 거의 동일. 이는 70B/405B 투영의 정당한 근거가 되며, n-gram speculation의 일반화 가능성을 시사.
  • Output quality의 형식적 검증: greedy argmax 하에서 token-identical을 4 prompt × 200 token × 2 model로 100% 확인. speculative decoding의 가장 흔한 우려에 대한 명확한 답.

한계와 의문점

  • 단독 저자, 소속 미상: 학술 환경의 peer review 없이 단독 개인이 작성. Github 저장소 coder903/split-inference의 신원도 익명. 결과의 재현성은 코드 공개로 보장되나, 연구 환경의 검증 부재는 한계.
  • Inversion attack의 약점: 3-layer MLP, 880 sample, position-independent. ##sequence-aware attacker## (예: language model prior 사용)나 10K+ sample로 학습 시 35~59% 수치가 크게 오를 수 있음을 ##저자도 인정##. 즉, ##보고된 inversion accuracy는 매우 보수적 lower bound##.
  • Critical assumption의 fragility: 로컬 layer 가중치가 public weights와 구별되어야 한다는 가정. ##공개 모델 그대로 쓰는 경우 privacy 경계가 사실상 무너짐##. LoRA로 fine-tuning한 layer가 진정으로 private한 가중치인지(adaptive attacker retrain 불가)는 5.3절에서 추가 연구로 미룸 — 핵심 가정이 검증 안 됨.
  • Lookahead의 한계 시나리오: creative text에서 1.06~1.12 acceptance, structured에서 0.96~0.99× sequential — ##기업 use case에 많은 dialog/customer support 같은 비반복 텍스트에서 lookahead의 실제 이득이 제한적##. 코드 use case에 편향된 marketing 가능성.
  • RTT decomposition 모델의 가정: per-step time = RTT + fixed_overhead가 lookahead에서 4.5~6.2% error. 12B lookahead 투영은 naive projection이 안 맞아 측정값으로 보정 — ##큰 모델/큰 speculative batch에서 모델의 정확도가 어떻게 떨어지는지##는 불명확. 405B 투영(0.4~0.8× RTT-to-compute)의 신뢰도는 낮음.
  • 실험 규모의 작음: ablation 8 prompts (category당 2), inversion 1100 sample (test 221), seed 반복 미보고. ±10% run-to-run variance를 저자도 명시. 더 큰 규모와 통계적 유의성 검증이 필요.
  • Real-world WAN 다양성 부재: 미국 내부 RTT 78-100ms만 측정. 대륙 간(예: 한국→미국 ~150ms, 한국→유럽 ~250ms), 모바일/위성 link 등에서의 검증 없음.
  • Threat model의 narrow함: semi-honest만 다룸. 실제 enterprise에서 클라우드가 malicious일 가능성, 또는 traffic analysis side-channel은 미고려.
  • vs DSSD 비교 부재: 가장 가까운 concurrent work인 DSSD (ICML 2025)와의 ##직접 throughput 비교 없음## — "다르다"는 정성적 언급만. 정량 비교 절실.

개선 제안

  • Sequence-aware inversion attack 평가: GPT-2 small을 attacker로 사용해 inter-token dependency 활용. 실제 worst-case privacy를 더 정직히 측정.
  • Fine-tuned local layer의 inversion robustness: LoRA로 fine-tune한 layer 0-1로 split하여 inversion attack을 재훈련 — 5.3절의 가설을 실험으로 검증.
  • LLaMA/Qwen 등 다른 architecture로 일반화 검증: 현재 Mistral 가족만. acceptance rate의 architecture 독립성이 정말 성립하는가?
  • Multi-region 측정: 대륙 간 link (≥150ms RTT)에서 lookahead 효과 검증.
  • Stochastic decoding에서의 acceptance: temperature 0.7 등 실무에 흔한 setting에서 reported 1.2~1.5×가 얼마나 유지되는지.
  • DSSD/CROSS-SEC와 정량 비교: 동일 hardware/network에서 head-to-head.
  • Privacy depth의 dynamic 조절: 사용자/요청 sensitivity에 따라 split point를 동적으로 결정하는 메커니즘 (현재는 정적).

Key Takeaways

  1. WAN split inference는 RTT amortization 문제다: 80~100ms RTT가 per-token wall time의 60~64%를 차지하는 환경에서, lookahead처럼 ##round-trip당 multiple token을 통과##시키는 기법이 1차 leverage. 일반적인 GPU-utilization 중심 speculative decoding 사고방식과 다른 contextualization 필요.
  2. Embedding boundary split = architectural privacy의 적정 sweet spot: cryptographic 기법(100~1000× 느림)과 plaintext API(0× privacy) 사이의 ##3~5× overhead로 raw token leakage 차단##. Formal guarantee는 아니지만, 규제 산업의 실용적 baseline.
  3. RTT-to-compute ratio가 inverse scaling: 모델이 커질수록 split inference의 상대적 overhead가 감소 — 405B에서 ratio가 1× 이하로 떨어져 ##사실상 native에 가까운 효율##. 따라서 이 접근은 작은 모델보다 ##큰 모델에 더 적합##하다는 역설.
  4. Acceptance rate는 model-size invariant: 7B와 12B의 동일한 1.21 tok/step. n-gram speculation이 ##언어 통계에 의존##하고 모델 capacity에 의존하지 않음을 시사 — 70B+에서도 유효할 강력한 prior.
  5. Deployment engineering이 알고리즘만큼 중요: VAST.ai proxy 같은 ##cloud provider 특유의 latency 함정##, SSH tunnel, ephemeral IP — 학술 논문이 좀처럼 다루지 않는 영역이 실 throughput을 결정. Optimization chain 1.4→17 tok/s에서 ##transport 최적화가 5.6× (절반 이상)## 기여.
