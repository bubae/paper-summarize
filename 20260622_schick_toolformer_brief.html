"정답을 가르치지 마라.
도구 사용이 도움이 되는지 AI가 스스로 판단하게 하라."

먼저 알아야 할 것: LLM은 못하는 게 많다

ChatGPT 같은 큰 AI는 글쓰기, 번역, 요약은 잘하지만, 다음과 같은 ##기본적인 일에 의외로 약합니다##:

  • 계산: "27 + 4 × 2" 같은 단순 산수도 자주 틀림
  • 최신 정보: "오늘 환율" 같은 실시간 정보 모름
  • 정확한 사실: 위키피디아에 있는 사실도 자주 거짓 답변(hallucination)
  • 저자원 언어: 힌디어, 베트남어 번역 부정확
  • 날짜 인지: "오늘 무슨 요일?"에도 답 못함

반면 ##계산기, 위키피디아 검색, 번역기, 캘린더## 같은 작은 도구들은 이 일들을 완벽하게 처리합니다. 그래서 자연스러운 질문: "AI가 이 도구들을 알아서 쓰면 안 될까?"

기존 접근법의 한계

"AI에게 도구를 가르치자"는 생각은 새롭지 않습니다. 하지만 기존 방법은 두 가지 큰 문제가 있었어요.

기존 방법들은 비싸거나, 특정 task에 갇혀 있다
비유: 신입사원 교육의 두 방식

방법 1 (사람 라벨링): 선배가 "이런 상황엔 엑셀, 이런 상황엔 캘린더"라고 일일이 가르침. 시간 오래 걸리고, 선배 기준이 신입에게 안 맞을 수 있음.

방법 2 (task별 setup): 신입에게 "재무 회의용 매뉴얼"만 줌. 영업회의에선 못 씀.

Toolformer 방법: 신입 책상에 도구들을 놓아두고, 일을 빨리 끝내게 해준 도구 사용 패턴만 보상. 신입이 스스로 발견.

Toolformer의 핵심 아이디어

Toolformer는 사람이 라벨링 없이 ##AI가 자체 판단으로## 도구 사용을 학습합니다. 핵심 통찰: "AI가 다음 단어를 예측할 때 도구를 쓰면 더 잘 예측되는가?" 이 질문에 자체 답할 수 있습니다.

3단계 학습 파이프라인

단계 1
후보 호출 생성
"여기서 도구 쓰면 좋을 듯" 위치 샘플링

유용성 필터의 수학

"도구가 도움 됐는지"를 어떻게 측정할까요? AI는 다음 단어를 예측할 때 ##확률##을 계산합니다. 도구 결과를 본 후 정답에 대한 확률이 ##더 높아지면## 도구가 도움 된 것이죠.

3가지 시나리오를 비교해 도구의 진짜 효용 측정
비유: 신입의 도구 사용 평가

신입사원이 작성한 보고서를 평가할 때, "엑셀을 썼는지"가 아니라 "엑셀을 써서 보고서가 더 정확해졌는지"를 평가하는 것. 도구 자체가 목적이 아니라 도구로 인한 결과 개선이 평가 기준.

비유: 아이의 도구 사용

3살 아이에게 계산기를 줘도 못 씀. 8살부터는 단순 계산기 사용 가능. 12살부터는 복잡한 도구도 활용. AI도 마찬가지로 ##일정 capacity 이상부터## 도구를 의미 있게 활용.

활용 분야

Toolformer가 연 길: 후속 모든 tool-using AI 시스템의 시발점

비유로 다시 정리

비유 1: 신입사원의 사무용품 사용

신입 책상에 계산기, 사전, 시계, 전화기를 놓고, "일을 더 잘하게 한 도구 사용 패턴만 보상". 신입은 알아서 어떤 일에 어떤 도구를 쓸지 발견.

비유 2: 학습된 어휘력

아이가 "내일 비 와?" 같은 질문에 답할 때, 책에서 외운 게 아니라 "엄마에게 물어보면 더 잘 안다"를 학습. 도구(엄마)가 어떻게 도움 되는지 자체 발견.

비유 3: 작가의 자료 조사

소설가가 "한국 수도는?"이라고 쓸 때 매번 위키피디아를 펴는 게 아니라, 본인이 모르는 부분만 정확히 검색. AI도 "내가 모르는 부분"에서만 도구 호출.

비유 4: 시험에서 계산기 사용

학생이 시험에서 계산기를 쓸지 결정할 때 기준은 "내가 손으로 풀 수 있나?". Toolformer도 "내가 다음 단어 예측할 수 있나?"가 기준.

비유 5: 아이의 emergence

아이는 일정 나이 이상이 되어야 도구를 효과적으로 씀. 너무 어리면 계산기 줘도 단순 누르기만 함. AI도 ##775M 파라미터 이상##에서 도구 활용 능력이 나타남.

비유 6: 학습 자료 vs 자가 학습

"선생님이 만든 모범 답안 외우기"(사람 라벨링) vs "내가 풀어본 시험 중 정답률 높았던 풀이 방식 분석"(자가 학습). 후자가 본인에게 진짜 맞는 방식. Toolformer가 후자.

한계점

⚠️ 1. 도구를 연속해서 못 씀

"오늘 날짜를 캘린더로 알아낸 후 그 날짜에 QA로 정보 검색" 같은 ##chain##을 못 함. 각 호출이 독립.

⚠️ 2. 검색 결과 재조회 못 함

위키 검색 결과가 부족할 때 쿼리를 ##재구성하거나 다른 페이지로 이동## 못 함. WebGPT/ChatGPT browse처럼 ##interactive 사용 불가##.

⚠️ 3. Sample inefficient

100만 문서에서 ##계산기 유용 사례 1,000건##만 추출. 학습 데이터 효율 낮음.

⚠️ 4. 도구 호출 비용 무시

API 호출의 ##비용/latency가 학습 신호에 반영 안 됨##. 작은 perplexity 이득만 있어도 도구 호출. production에선 cost-aware 필요.

⚠️ 5. 강한 모델에서 도구 효용 감소

LLaMA 7B 같은 강한 base에서는 ##Wikipedia search tool의 효용이 거의 사라짐##. 강한 모델은 도구 없이도 잘 함. ##도구 강도 < 모델 강도이면 도구 무의미##.

역사적 위치

Toolformer는 ##LLM agent 시대의 시작##입니다.

  • 2023 2월: Toolformer (NeurIPS 2023) — self-supervised tool learning 패러다임
  • 2023 3월: ChatGPT plugins 출시 — Toolformer의 production 구현
  • 2023 6월: GPT-4 function calling — 표준화된 tool interface
  • 2023 후반: ReAct, LangChain, AutoGPT — multi-step / interactive 도구 사용
  • 2024+: ToolBench, Gorilla, MetaGPT — 수천 개 API 라이브러리

Toolformer의 ##self-supervised + perplexity filtering## 아이디어는 후속 RLAIF, self-improvement, agent fine-tuning 등에 직접 영향.

FAQ

Q: ChatGPT의 도구 사용과 어떻게 다른가요?

A: ChatGPT plugins는 OpenAI가 fine-tune한 production 시스템. Toolformer는 그 이전의 academic baseline. 핵심 패러다임(self-supervised tool learning + filtering)이 ChatGPT plugins에 영향을 줬습니다.

Q: 일반 사용자가 어떻게 활용?

A: 직접 활용은 없습니다. 하지만 ChatGPT, Claude, Gemini의 도구 사용 능력의 ##연구적 토대##. 사용자가 LLM에게 "위키 검색해줘"라고 부탁할 수 있는 능력의 시작점.

Q: 왜 GPT-3 175B을 이겼나요?

A: ##도구가 작은 specialized model의 능력을 빌려옴##. 계산기는 큰 모델보다 정확한 계산을 함. AI가 그 도구를 활용하면 자기 크기보다 큰 능력을 발휘.
