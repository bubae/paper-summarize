왜 이 모델이 중요한가

요즘 AI 챗봇은 이미지도 보고 답변할 수 있습니다. 하지만 실제로는 "이미지를 본다"가 어색한 부속 기능인 경우가 많아요. 비전 인코더라는 작은 모듈을 붙여 "이미지 → 텍스트 설명"으로 변환해서 LLM에 넘기는 식이죠.

이래서는 스크린을 보고 클릭하거나, 차트를 정확히 읽어 분석하거나, GUI 흐름을 따라 자동화하는 일에 한계가 있습니다. 자세한 시각 정보가 텍스트 설명으로 압축되며 손실되기 때문이죠.

GLM-5V-Turbo는 시각을 reasoning, planning, tool 사용, 실행의 핵심 컴포넌트로 통합한 새 멀티모달 파운데이션 모델입니다. Z.ai (전 Zhipu AI)와 Tsinghua University의 collaboration으로, 이미지·웹·문서·GUI를 직접 이해하고 행동하는 agentic AI의 다음 단계를 제시합니다.

배경 지식 — 5가지 개념을 차근차근

① 멀티모달 모델이란?

텍스트만 처리하는 LLM(GPT-4의 텍스트 모드)과 달리, 이미지·비디오·음성 등 여러 데이터 타입을 동시에 다루는 AI입니다. 사용자가 사진을 보여주며 "이건 뭐야?"라고 물으면 답변할 수 있죠.

② "비전 부착형" vs "Native 멀티모달"

두 가지 접근이 있습니다:

  • 부착형 (Bolt-on): 텍스트 LLM에 비전 인코더를 나중에 붙임. 빠르고 효율적이지만 reasoning과 perception이 분리됨.
  • Native: 처음부터 비전과 언어를 함께 학습. 더 깊이 통합되지만 학습 비용 큼.

GLM-5V-Turbo는 후자에 가까운 설계—pretraining 단계부터 멀티모달.

Bolt-on은 카메라로 사진을 찍어 친구에게 말로 설명하는 것과 비슷합니다. Native는 친구도 같은 사진을 직접 보면서 함께 결정을 내리는 것이죠. 후자가 훨씬 정확하겠죠.

③ "Agentic AI"란?

단순히 질문에 답하는 챗봇을 넘어, 스스로 task를 수행하는 AI입니다:

  • 웹 브라우저를 자동으로 조작하기 (정보 수집, 폼 작성)
  • 화면을 보고 GUI 클릭 (Mobile App 자동화)
  • 코드 작성 + 실행 + 디버깅
  • 긴 문서·논문 읽고 보고서 작성

이 모든 작업은 시각 정보를 정확하게 읽는 능력이 필수입니다.

④ Vision Encoder (CogViT)

이미지를 AI가 이해할 수 있는 숫자 벡터로 변환하는 모듈입니다. GLM-5V-Turbo는 새로운 인코더 CogViT (403M params)을 만들었습니다. 더 큰 SigLIP2 (427M)나 DFN-H (632M)보다 더 작은데 더 잘하는 것이 특징.

비결: Dual Teacher Distillation—SigLIP2(의미)와 DINOv3(질감) 두 선생님으로부터 동시에 배움.

한 분야 전문 선생님(SigLIP2 = 의미 박사)과 다른 분야 전문 선생님(DINOv3 = 질감 박사)에게 동시에 배우는 학생을 떠올려 보세요. 더 적은 시간 공부해도 더 다재다능해집니다.

⑤ Multi-Token Prediction (MTP)와 그 멀티모달 확장

일반 LLM은 한 번에 한 토큰씩 예측합니다. MTP는 한 번에 여러 토큰을 동시에 예측해 더 빠르고 더 정확. 그러나 이미지 입력에서 어떻게 작동시킬지가 도전.

GLM-5V-Turbo의 답: <|image|> placeholder. 이미지 위치 정보는 보존하되, 실제 시각 임베딩을 균일한 placeholder로 대체. 학습이 더 안정적이고 인프라도 단순해집니다.

핵심 아이디어 — 4단계 통합

1단계
CogViT 비전 인코더

두 단계 학습: 마스킹 이미지 학습 (35% 마스킹) + 대조 학습 (8B 한·영 이미지·텍스트 코퍼스). dual teacher로 적은 파라미터에 더 풍부한 표현.

2단계
MMTP (Multimodal MTP)

multi-token prediction을 멀티모달로 확장. <|image|> placeholder 방식으로 학습 안정성 + 인프라 호환성 동시에.

3단계
30+ task RL

perception/reasoning/agentic 30+ 도메인을 동시에 RL. cross-task transfer (한 도메인 학습이 다른 도메인 향상) 발견.

4단계
에이전트 프레임워크 통합

Claude Code(터미널 logic) + AutoClaw(GUI hands) + GLM-5V-Turbo(vision-language brain)의 perception → planning → execution 풀 loop.

30+ Task RL의 cross-task transfer

이 논문의 가장 흥미로운 발견 중 하나는 multi-task RL이 single-task보다 안정적이고 효과적이라는 것입니다.

  • Single-task RL은 narrow distribution에서 학습이 oscillate (왔다갔다 진동)하는 경향.
  • 여러 도메인을 동시에 학습하면 strategy 분포가 풍부해져 더 robust한 해결책으로 수렴.
  • 한 도메인의 thinking pattern이 다른 도메인으로 전이됨—예: UI-to-code 학습이 multi-turn coding 능력 향상.
수영 한 종목만 훈련하는 선수보다 수영·달리기·자전거 모두 훈련하는 트라이애슬론 선수가 종합 체력이 더 좋다는 것과 비슷합니다. 한 종목만 했을 때 생기는 편중·과적응이 다중 훈련에서 줄어들죠.

Real-world 통합 — 실제 task 흐름

예시: NVIDIA 주식 분석 보고서 생성

  1. 사용자: "NVIDIA 주식을 분석해서 영문 리포트로 작성해줘"
  2. GLM-5V-Turbo가 glmv-stock-analyst skill 호출
  3. OpenClaw가 여러 출처에서 정보 수집 (텍스트 + 차트)
  4. GLM-5V-Turbo가 차트를 직접 시각적으로 분석 (기술적 분석)
  5. fundamental + sentiment + action plan 종합
  6. professional 보고서 출력

예시: 웹사이트 자동 복제

"이 웹사이트 URL을 받아 비슷한 HTML/CSS로 복제해줘"라는 명령. GLM-5V-Turbo가 GUI를 직접 탐색하며 layout, interaction, asset을 추출 → 95점대 코딩 정확도로 재현.

4가지 Design Lens (개발 과정의 통찰)

Lens 1
Perception이 가장 fundamental

최근에 reasoning, planning을 강조하지만, 결국 fine-grained perception 오류가 high-level 실패의 시작. perception을 끝까지 다듬는 것이 결정적.

Lens 2
계층적 최적화

long-horizon task만 RL하면 unstable. element 인식 → grounding → 단일 action → trajectory로 계층 분해해 학습.

Lens 3
명세 + 검증의 명확성

End-to-end task는 stable specification, reliable verification이 핵심. PRD + mockup + reference로 풍부하게 정의.

Lens 4
자기 비판 학습

모델이 자기 perception 오류를 스스로 비판하도록 학습하면 hallucination 감소 (특히 GUI agent).

Lens 1을 비유하자면, 셰프가 아무리 레시피(reasoning)에 능통해도 재료의 신선도를 보지 못하면(perception) 요리는 망친다는 것입니다. 본 논문은 perception 자체가 ceiling을 결정한다고 강조합니다.

활용 분야

한계와 도전

① 더 나은 agentic strategy의 emergence

RL이 cold-start trajectory의 변형 안에 머물러 있음. sub-agent decomposition, multi-agent collaboration을 모델이 스스로 발견하게 하는 것은 미해결.

② 멀티모달 컨텍스트 관리

이미지·비디오는 텍스트보다 context budget을 훨씬 많이 소비. 긴 trajectory에서 earlier visual observation을 단순 drop하면 정보 손실. 텍스트처럼 압축하기도 어렵습니다 (layout, spatial, temporal 보존 필요).

③ Model + Harness 동시 진화

모델 단독이 아니라 도구·메모리·검증 루프가 capability boundary를 함께 결정. 같은 모델이 다른 harness 하에서 매우 다르게 행동. 따라서 model 개선만 측정하면 progress가 misleading.

④ Frontend full-stack의 약점

Vision2Web (full-stack 웹 복제)에서 31.0 vs Claude의 43.5—12.5pt 차이로 밀림. 가장 복합적인 multimodal coding integration에서 한계.

실무자/연구자가 가져갈 인사이트

  1. "Perception ceiling"의 발견: high-level reasoning RL보다 fine-grained visual perception을 다듬는 것이 결정적. SVG/frontend coding을 perception proxy로 활용하는 idea가 영리하다.
  2. Multi-task RL의 cross-domain transfer: single-task RL이 oscillate할 때, 다중 task로 동시 학습하면 안정화 + strategy unification 효과.
  3. 계층적 capability optimization: long-horizon만 학습하지 말고 element → grounding → step → trajectory 분해.
  4. <|image|> placeholder 디자인: 직접 visual embedding보다 균일한 placeholder가 학습 안정성 + 인프라 호환성 양쪽에서 우월.
  5. Task specification + verification 설계: instruction만이 아니라 PRD, mockup, reference를 함께 — workflow-based verification.
  6. Harness가 capability boundary의 일부: model만 보지 말고 도구 ecosystem, memory, verification도 함께 설계.
