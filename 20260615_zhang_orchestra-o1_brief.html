Orchestra-o1

텍스트, 이미지, 오디오, 비디오를 한꺼번에 다루는 AI 에이전트의 "지휘자(Orchestrator)" 만들기

📄 전문가용 상세 요약 보기 →
한 명의 만능 천재가 모든 일을 떠맡는 대신, 작업을 잘게 쪼개 전문가들에게 동시에 맡기고 결과를 합치는 'AI 프로젝트 매니저'

📚 배경 지식 — 왜 이 문제가 어려운가?

1. 멀티모달 시대의 도전

인간은 일상에서 자연스럽게 여러 감각을 동시에 처리합니다. 친구와 대화할 때 우리는 말소리(오디오), 표정과 몸짓(영상), 주변 환경(이미지), 대화 내용(텍스트)을 모두 한 번에 이해합니다. AI가 이 수준에 도달하려면 단순히 "사진을 분석한다", "음성을 인식한다"를 따로 잘하는 것을 넘어, 여러 종류의 정보를 동시에 통합 추론(omnimodal reasoning)할 수 있어야 합니다.

2. AI 에이전트(Agent)란?

"AI 에이전트"는 단순히 질문에 답하는 챗봇과 달리, 스스로 계획을 세우고, 도구(웹 검색·코드 실행·이미지 분석 등)를 사용해, 여러 단계로 문제를 해결하는 AI 시스템입니다. 마치 인턴이 "이 보고서 좀 작성해줘"라는 지시를 받고 (1) 관련 자료를 검색하고 (2) 차트를 그리고 (3) 글을 다듬는 것처럼 말이죠.

💡 비유: 챗봇이 "요리 레시피를 알려주는 책"이라면, AI 에이전트는 "장 보러 가서 재료 사고, 요리하고, 설거지까지 하는 도우미"입니다. 행동하고, 도구를 쓰고, 결과를 보고 다음을 결정합니다.

3. 기존 멀티모달 에이전트의 두 가지 길

두 접근 모두 실세계 멀티모달 과제(이를테면 "이 영상에서 무슨 일이 일어났고, 그 사람의 발언이 사실인지 검증해줘")를 풀기에는 부족했습니다.

4. 강화학습 기반 에이전트 학습 (GRPO란?)

최근 LLM 학습에서 자주 쓰이는 GRPO(Group Relative Policy Optimization)는, 같은 질문에 대해 여러 답을 생성하고 그 중 어느 게 "더 잘했는지" 비교해서 학습 신호를 만드는 방식입니다. 마치 학생들이 같은 시험을 보고 "옆자리보다 잘한 만큼" 점수를 받는 것과 비슷합니다.

💡 GRPO 비유: 코딩 면접에서 5명의 후보가 같은 문제를 풀면, 평균 점수보다 잘한 사람은 +가, 못한 사람은 - 점수가 됩니다. 이걸 모델에게 적용해 "그룹 평균 대비 잘한 정도"로 학습시키는 것입니다.

그런데 GRPO를 에이전트에 그대로 쓰려면 문제가 있습니다. 에이전트는 한 답을 내기까지 여러 단계의 도구 호출을 거치는데, 매번 전체 과정을 실행하면 너무 비싸고, 최종 답만 보고 점수를 매기면 중간 단계의 좋고 나쁨을 알 수 없습니다.

🎯 핵심 아이디어 — Orchestra-o1의 3가지 혁신

1

Modality 인식 작업 분해

매니저가 작업을 받자마자 "이 sub-task는 이미지가 필요하다", "이건 오디오 + 텍스트가 필요하다", "이건 웹 검색이 있어야 한다"를 명시적으로 결정. 또한 어떤 sub-task가 어떤 sub-task의 결과를 기다려야 하는지(의존성)도 그래프로 그립니다.

2

실시간 전문가 매칭

매 작업마다 "어떤 모델"과 "어떤 도구"를 쓸지 동적으로 결정. 쉬운 정보 추출은 싼 모델로, 어려운 추론은 GPT-5 같은 강한 모델로. 모델별 가격표(In: $0.0125, Out: $0.01 등)와 능력 점수를 함께 고려.

3

병렬 실행

독립적인 sub-task들은 동시에 실행. 영상 분석과 오디오 분석이 서로 의존하지 않으면 함께 처리하고, 그 결과를 모아 다음 단계로 넘어감. 같은 작업이 약 30% 더 빨라짐.

🔄 실제 작동 예시 — 케이스 스터디

문제: "이 오디오와 이미지를 보고, 오디오에 언급된 계절 행사가 일어나는 시간을 UTC로 알려줘"

비용 vs 정확도 — 두 마리 토끼를 모두 잡다

왼쪽 위로 갈수록 좋은 위치 — Orchestra-o1은 비용을 줄이면서 정확도도 크게 향상

💡 왜 비용까지 줄어들까? 두 가지 이유 — (1) 쉬운 sub-task는 싼 모델로 라우팅해서 GPT-5 호출 횟수가 줄고, (2) 병렬 실행으로 wall-clock 시간이 짧아져 API 사용 시간이 감소합니다.

🚀 더 작은 모델도 강하게 — DA-GRPO

오픈소스 8B 모델로 30B 모델 능가

저자들은 Qwen3-8B(작은 모델)를 base로 새로운 강화학습 기법 DA-GRPO로 학습한 Orchestra-o1-8B를 공개. 결과는 놀랍습니다:

12.5% 기본 Qwen3-8B (ReAct)
26.3% + Orchestra 프레임워크
30.0% + DA-GRPO 학습

같은 8B 모델이 12.5% → 30.0%로 2.4배 성능 향상. 더 큰 OmniAtlas-Qwen-3-30B-A3B(20.8%)도 능가합니다.

DA-GRPO가 왜 다른가?

💡 DA-GRPO 비유: 기존 GRPO가 "전체 프로젝트 끝낸 후에야 인턴을 평가"하는 거라면, DA-GRPO는 "매 작업 결정마다 즉시 피드백"입니다. 형식이 맞는지 / 행동이 유효한지 / 도구 선택이 합리적인지 / 전체 결정 품질이 좋은지 — 4가지를 매번 채점합니다.

⚠️ 한계점 — 솔직한 시각

1. 단일 벤치마크 의존

모든 주요 결과가 OmniGAIA 한 벤치마크에 의존합니다. 다른 멀티모달 에이전트 벤치마크에서도 같은 우위가 나올지는 미지수입니다. OmniGAIA의 작업 분포가 Orchestra-o1의 의존성 그래프 패러다임과 잘 맞을 가능성이 있습니다.

2. Capability Score의 추정 방법 미공개

"이 모델은 이미지에 강하다 / 오디오에 강하다"라는 점수가 어떻게 정해지는지 구체적이지 않습니다. 실제로는 prompt에 적어주는 방식인데, 이 prompt engineering의 robustness가 평가되지 않았습니다.

3. Reward Model 의존

DA-GRPO의 4차원 점수를 매기는 reward model이 Claude-Haiku-4.5입니다. Haiku의 평가 편향이 학습된 정책에 그대로 전파될 수 있고, reward hacking 가능성이 분석되지 않았습니다.

4. 학습 데이터 1200개 — 작은 규모

최종 학습 데이터셋이 1200 examples입니다. 더 큰 scale에서 saturation이나 over-fitting 양상이 어떨지 분석이 부재합니다.

5. Hard task 절대 성능 여전히 낮음

Open-source 8B 모델의 Hard task 정확도가 26.9%로, ceiling이 아직 멉니다. 의존성 그래프가 깊어질 때 어디에서 실패하는지 error analysis가 부재합니다.
