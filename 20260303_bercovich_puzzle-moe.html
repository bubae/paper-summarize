Extending Puzzle for Mixture-of-Experts Reasoning Models with Application to GPT-OSS Acceleration

  • 저자: Akhiad Bercovich, Nir Ailon, Vladimir Anisimov, Tomer Asida, ..., Ran El-Yaniv (총 24명)
  • 소속: NVIDIA
  • 학회/저널: arXiv preprint (Technical Report), February 2026
  • 연도: 2026
  • 링크: arXiv:2602.11937

TL;DR

  • 문제: Reasoning LLM은 긴 reasoning trace를 생성하여 답변 품질을 높이지만, 추가 토큰이 serving 비용을 크게 증가시킨다. 특히 MoE (Mixture-of-Experts) 아키텍처와 128K 이상의 long-context에서는 KV-cache가 메모리와 bandwidth를 지배하여 throughput이 급격히 저하된다.
  • 해결: Puzzle이라는 post-training NAS (Neural Architecture Search) 프레임워크를 MoE reasoning 모델에 확장 적용하여, gpt-oss-120B에서 gpt-oss-puzzle-88B를 파생. Heterogeneous expert pruning, selective window attention, FP8 KV-cache quantization, RL + checkpoint merging을 결합.
  • 왜 작동하는가: Puzzle의 decomposed replace-1-block scoring이 각 layer의 중요도를 독립적으로 평가하여, layer별로 최적의 압축 수준을 결정한다. 초기 MoE layer가 후기보다 중요하다는 비대칭성을 포착하여 heterogeneous pruning을 가능하게 하고, long-context-aware scoring으로 window attention 전환에 안전한 layer를 식별한다.
  • 비유: 퍼즐 맞추기처럼, 각 layer(퍼즐 조각)에 대해 "가벼운 대체 조각"을 시도해보고, 전체 그림(모델 품질)이 가장 잘 보존되는 조합을 MIP (Mixed-Integer Program)으로 찾는 것이다. 무작정 모든 조각을 작게 만드는 대신, 중요한 조각은 원본을 유지하고 덜 중요한 조각만 교체한다.

핵심 기여 (Key Contributions)

  1. MoE 모델에 Puzzle NAS 확장: Expert-parallel 배포 제약을 존중하면서 layer별 expert 수를 다르게 pruning하는 heterogeneous MoE expert removal 방법을 제안
  2. Long-context-aware window attention scoring: Activation-based 대신 AA-LCR (Artificial Analysis Long-Context Reasoning) 벤치마크 기반 scoring으로, window attention 전환에 안전한 layer를 식별. 40% KV-cache 절감 달성
  3. RL + Checkpoint merging 전략: High-effort RL (정확도↑, 길이↑)과 balanced RL (길이↓, 제어성↑) 두 변형을 학습 후 weight averaging으로 결합하여, 정확도와 생성 길이를 동시에 최적화
  4. FP8 KV-cache quantization with calibrated scales: Max calibration + power-of-2 rounding으로 KV-cache를 2배 압축하면서 정확도 유지
  5. Request-level efficiency 메트릭 제안: Per-token throughput이 reasoning 모델에서는 부적절함을 지적하고, 생성 토큰 수를 반영하는 request-level 효율성 프레임워크를 제시

1. 배경: Puzzle 프레임워크와 Reasoning 모델의 과제

Puzzle NAS 프레임워크

Puzzle (Bercovich et al., 2025d)은 학습된 parent 모델에서 출발하여, 배포 효율성 제약(메모리, latency, throughput) 하에서 정확도를 최대한 보존하는 파생 아키텍처를 탐색하는 decomposed NAS 프레임워크이다. 핵심 아이디어는:

  1. Block library 구축: 각 transformer layer(block)에 대해 대체 가능한 "퍼즐 조각"들의 집합을 정의. 예를 들어 attention은 {global, window}, MoE FFN은 {128, 96, 64, 32, 16, 8 experts}
  2. Replace-1-block scoring: 각 대체 조각을 원래 위치에 교체하고 나머지는 유지한 상태에서, parent 모델과의 output divergence를 측정. 전체 아키텍처의 품질은 개별 replace-1-block 점수의 합으로 추정
  3. MIP (Mixed-Integer Program) 탐색: 주어진 비용/점수 프로파일에서, 배포 제약을 만족하면서 품질을 최대화하는 조합을 최적화

Puzzle은 이전에 Llama-3 Instruct 시리즈에서 Llama-Nemotron으로 파생할 때 사용되어, 1.7~2.1× speedup을 달성한 바 있다.

Reasoning 모델로의 확장 과제

최근 LLM에 Puzzle을 적용할 때 새로운 도전이 발생한다:

  • MoE FFN의 복잡성: MoE layer에서는 routing 동작과 expert-parallel 배포 제약을 존중하면서 "얼마나 pruning할 것인가"를 layer-dependent하게 결정해야 함
  • Long-context에서 attention bottleneck: Reasoning 모델은 128K 이상의 context를 사용하므로, KV-cache가 메모리와 bandwidth를 지배. Window/streaming attention이 KV-cache를 줄여주지만, 잘못된 layer에 적용하면 long-range dependency를 손상
  • Scoring 신호의 locality bias: 기존 activation-MSE 기반 replace-1-block scoring은 next-token prediction의 local 특성으로 인해 long-context 의존성 손상을 제대로 감지하지 못함

2. 방법론: Puzzle 최적화

2.1 Search Space 설계

gpt-oss-120B의 각 transformer layer는 attention 모듈과 MoE FFN 모듈로 구성된다. Layer $i$의 search space는 $A_i \times F_i$로, attention과 FFN 선택의 곱으로 정의된다.

  • Attention 대안: {원래 parent attention, window attention (W=8192)}. Parent 모델 gpt-oss-120B는 128-token sliding window와 global attention을 번갈아 사용하는 구조
  • MoE FFN 대안: 원래 128개 expert에서 {8, 16, 32, 64, 96, 128}개로 축소된 변형들. Expert-parallel inference에 호환되도록 설계

2.2 Heterogeneous MoE Expert Pruning

각 MoE layer에서 expert의 중요도를 ranking하여, layer별로 다른 수의 expert를 유지하는 heterogeneous pruning을 수행한다.

Expert contribution score: 각 MoE layer $f$에 대해, expert $i$를 제거했을 때의 output 변화를 MSE로 측정:

$$\text{expert\_score}_i = \mathbb{E}_x \left[ \text{MSE}\left(f(x), f^{(i)}(x)\right) \right]$$

여기서 $f^{(i)}$는 expert $i$를 제거한 MoE layer, $x$는 입력 hidden state이다. 직관적으로, 제거 시 output 변화가 큰 expert일수록 중요하다.

중요한 발견: ##초기 MoE layer가 후기 layer보다 훨씬 더 중요하다## (Figure 2). 따라서 Puzzle은 초기 layer는 128개 expert를 유지하고, 후기 layer는 64개 이하로 대폭 줄이는 heterogeneous 구조를 선택한다. 전체적으로 약 25%의 expert가 pruning되어 88B parameter 모델이 생성된다.

구현 세부사항: Parent 분포에 일치하는 데이터를 사용하는 것이 중요도 추정에 매우 중요. LNPT-gpt-oss 데이터셋을 사용하여 12K packed sequences (8K 길이)로 expert contribution score를, 128 packed sequences (32K 길이)로 replace-1-block score를 계산.

2.3 Selective Window Attention

Window attention은 각 토큰이 최근 $W$개 토큰만 참조하도록 제한하여, KV-cache를 sequence length $L$이 아닌 window size $W$에 비례하게 만든다. 이는 very long context에서 특히 유리하다.

그러나 모든 global attention layer를 window attention으로 전환하면 long-range dependency가 심각하게 손상된다. 따라서 "window-able"한 layer만 선택적으로 전환하는 것이 핵심이다.

Long-context-aware scoring: 기존 activation-MSE scoring은 inherent locality bias가 있어 long-context 손상을 제대로 감지하지 못한다. 이를 해결하기 위해 ##AA-LCR (Artificial Analysis Long-Context Reasoning) 벤치마크의 정확도 하락을 replace-1-block 점수로 사용##한다. 즉, layer $i$를 window attention으로 교체했을 때의 AA-LCR 점수 하락이 작을수록 해당 layer는 window attention에 적합하다고 판단한다.

결과: 18개 global attention layer 중 8개가 window attention (W=8192)으로 전환되어, ##KV-cache 크기가 parent 대비 40% 감소##.

RoPE scaling 조정: Window attention 도입 후 long-context 동작을 지원하기 위해, YaRN positional encoding의 RoPE scaling factor를 32에서 56으로 증가. 128K 길이에서의 phase wrapping을 줄여 long-range attention을 안정화.

2.4 Multi-Constraint Optimization

Puzzle을 단일 실행으로 두 가지 배포 시나리오를 동시에 최적화:

Puzzle 입력: gpt-oss-120B (parent, 128 experts × 36 layers) ├─ [제약 1] 64K/64K (long-context): 1.6× throughput 개선 목표 │ └─ KV-cache 지배 → Selective window attention 주력 │ → 18개 중 8개 global attention → window (8192) │ → KV-cache 40% 감소 ├─ [제약 2] 4K/4K (short-context): 1.2× throughput 개선 목표 │ └─ MoE 지배 → Expert pruning 주력 │ → 128 → layer별 8~128 experts (평균 ~25% 제거) └─ 출력: gpt-oss-puzzle-88B (heterogeneous architecture)

3. 학습 및 평가 설정

3.1 Knowledge Distillation

Puzzle phase 이후, blockwise substitution으로 인한 품질 저하를 회복하기 위해 knowledge distillation을 수행한다:

  • 데이터: LNPT-gpt-oss 데이터셋 (nvidia/Llama-Nemotron-Post-Training-Dataset 기반). Mathematics, code generation, general reasoning, instruction following 프롬프트에 대해 parent 모델이 high/medium reasoning effort로 생성한 응답
  • 토큰 수: 총 84B 토큰
  • Sequence length: 128K
  • Batch size: 33M tokens (global)
  • 프레임워크: Megatron-LM
  • 고정: MoE experts와 router는 frozen. Distillation은 high + medium effort 응답을 동등하게 혼합하여, 다양한 사용 패턴에서의 정확도를 보존

3.2 Reinforcement Learning + Checkpoint Merging

Distillation 이후 RL을 통해 reasoning 정확도를 추가로 개선한다. 핵심 통찰은 ##RL 데이터의 reasoning effort 구성이 정확도와 생성 길이 사이의 근본적 tension을 야기한다##는 것이다:

RL 변형정확도생성 길이제어성
High-effort only최고모든 effort에서 길어짐effort length ratio 증가 → 비효율적
Balanced mix (all efforts)더 낮음짧음 (length regularization 효과)effort 간 길이가 수렴 → 제어성 약화

해결책: 두 RL checkpoint를 weight averaging으로 결합. 이를 통해 high-effort RL의 정확도를 유지하면서도, balanced RL의 짧은 생성 길이와 parent 수준의 effort length ratio를 복원한다.

RL 세부사항: Learning rate 1e-6 (constant), mathematics/coding/general reasoning environment 사용 (tool-use 제외), MoE experts와 router는 frozen.

3.3 FP8 KV-Cache Quantization

Parent 모델 gpt-oss-120B의 MXFP4 expert quantization을 유지하면서, KV-cache에 FP8 quantization을 추가 적용:

  • KV scales 없이 quantization하면 정확도가 저하됨을 발견
  • LNPT-gpt-oss 데이터셋을 사용한 max calibration으로 KV scales를 계산
  • ##Scales를 power-of-2로 반올림##하여, rounding error가 "scales 없는 경우"와 유사하게 동작하면서도 dynamic range adaptation을 통해 underflow 방지
  • 효과: ~2× KV-cache 토큰 용량 + 더 빠른 attention 모듈

Request-Level Efficiency (Q3)

실험 의도: Per-token throughput이 reasoning 모델에서 실제 end-to-end 효율성을 정확히 반영하는가?

Per-token throughput만으로는 reasoning 모델의 효율성을 판단할 수 없다. 모델 변형이나 reasoning effort에 따라 생성 토큰 수가 달라지므로, ##2× throughput 이득이 trace가 2× 길어지면 상쇄##된다. 반대로, throughput 이득을 더 많은 reasoning 토큰에 "투자"하여 정확도를 높일 수도 있다.

저자들은 relative request rate를 제안한다:

$$\text{Relative Request Rate} = \frac{\text{Max Token Throughput} / \text{Avg Tokens Generated per Request}}{\text{Baseline Request Rate}}$$

직관적으로, 초당 처리할 수 있는 요청 수를 정규화한 것이다.

HyperNova-60B (Multiverse Computing, 제3자 gpt-oss-120B 압축 모델)와의 비교가 이를 잘 보여준다: HyperNova-60B는 architectural efficiency가 높지만, 더 긴 reasoning trace를 생성하여 request-level에서의 이점이 상쇄된다. 반면, gpt-oss-puzzle-88B는 ##accuracy-speed frontier 전체에서 parent를 능가하며, 최대 1.29× higher request-level efficiency##를 달성한다.

5. 전체 파이프라인 흐름도

6. 비판적 분석

강점

  • End-to-end 실용적 파이프라인: NAS → KD → RL → quantization의 완전한 파이프라인을 제시하여, 실제 배포에 즉시 적용 가능한 수준의 완성도
  • Multi-constraint 최적화: 단일 Puzzle 실행으로 long/short context 두 시나리오를 동시에 최적화하는 실용적 접근
  • Request-level efficiency 프레임워크: Per-token throughput의 한계를 명확히 지적하고, reasoning 모델에 적합한 평가 메트릭을 제안. HyperNova-60B 비교가 이를 설득력 있게 보여줌
  • RL checkpoint merging의 참신성: 정확도-길이 tension을 weight averaging으로 해결하는 간단하지만 효과적인 방법
  • 하드웨어별 분석: 8×H100, 1×H100, 8×B200 등 다양한 배포 환경에서의 결과 제시

한계 및 질문

  • gpt-oss 특화 결과: 모든 실험이 gpt-oss-120B 단일 모델에 대한 것. 다른 MoE 모델 (DeepSeek-V3, Qwen-3-MoE 등)로의 일반화 가능성은 불확실. Puzzle이 Llama-3에서도 작동했다는 점이 간접적 증거이나, MoE 특화 기법의 transferability는 미검증
  • AA-LCR 의존성: Window attention scoring에 AA-LCR을 사용하고, 평가에도 AA-LCR을 포함. 이 벤치마크의 representative성에 대한 분석 없이, scoring과 evaluation의 circular dependency 리스크가 존재. RULER 결과가 이를 보완하지만 AA-LCR에서의 일관된 하락 (High: 46.75→40.75)은 우려
  • Standard deviation 보고가 제한적: 그래프에서 ±1σ 밴드를 보여주지만, 정확도 Table에서는 개별 결과만 보고. "여러 번 측정하여 평균"이라고 명시하나 구체적 반복 횟수와 표준편차 미제시
  • 생성 길이 증가의 trade-off: Parent 대비 1.09~1.57× 더 많은 토큰 생성. Medium effort에서 1.57×는 상당한 증가로, 일부 사용 사례에서는 architectural throughput 이득을 상쇄할 수 있음
  • Ablation의 제한적 범위: Expert pruning 단독, window attention 단독, 결합 효과를 분리한 ablation이 부족. Figure 4b에서 "MoE pruning only" 곡선이 일부 이를 보여주지만 체계적 분석은 아님
  • Training cost 미보고: KD 84B 토큰 + RL 학습의 총 GPU-hours가 명시되지 않아 재현 비용 판단이 어려움

7. Key Takeaways

  • Post-training NAS가 MoE reasoning 모델에도 효과적: Puzzle 프레임워크의 decomposed scoring + MIP 접근이 MoE의 heterogeneous 특성을 잘 포착하여, 초기 layer는 보존하고 후기 layer를 공격적으로 pruning하는 최적 전략을 자동으로 발견.
  • Per-token throughput은 reasoning 모델 평가에 부적절: 생성 토큰 수가 모델과 effort에 따라 다르므로, request-level efficiency 메트릭이 실제 배포 비용을 더 정확히 반영한다. 압축 모델 개발 시 반드시 생성 길이 변화를 모니터링해야 한다.
  • Task-aware scoring이 aggressive 압축에서 중요: Window attention 전환 비율이 높을수록 activation-MSE와 AA-LCR scoring의 차이가 커진다. 압축이 보수적이면 어떤 scoring이든 유사하지만, 더 높은 압축률을 목표로 할 때 task-aware signal이 필수적.
  • RL checkpoint merging은 정확도-효율성 Pareto 개선의 간단한 방법: 서로 다른 objective를 가진 RL 변형들을 weight averaging으로 결합하는 것은 비용이 거의 없으면서도 양쪽의 장점을 취할 수 있다.
  • Single GPU에서의 이득이 가장 극적: 메모리 제약이 심한 환경(1×H100)에서 2.82× speedup은, expert pruning + window attention의 메모리 절약이 batch size 확대로 이어지는 선순환 효과를 보여준다.
