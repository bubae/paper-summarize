"LoRA vs Full Fine-Tuning"이라는 잘못된 질문

MoLF — 두 가지를 함께 학습시키되, 매 순간 더 유리한 쪽에만 발언권을 주는 지능형 옵티마이저

"두 전문가에게 같은 자료를 주고,
매 결정마다 더 적합한 한 명에게만 펜을 쥐어 주자."

왜 이 논문이 중요한가

요즘 거대 언어 모델(LLM)을 특정 분야에 적응시킬 때 두 가지 방식이 있습니다:

  • Full Fine-Tuning (FFT): 모델의 모든 파라미터를 업데이트. 가장 강력하지만 메모리·연산 비용 큼.
  • LoRA: 작은 추가 행렬만 학습해 효율적. 메모리 절감 + 일종의 정규화 효과.

학계는 둘 중 어느 게 좋은지 수년간 논쟁해왔습니다. 한쪽은 "LoRA는 충분하다"라 하고, 다른 쪽은 "LoRA Learns Less"라 외칩니다.

이 논문은 둘 다 부분적으로 맞다는 것을 실증으로 보입니다: task에 따라 답이 다르다는 거죠. 사실 주입(fact)에서는 FFT가, 의료 추론(med)에서는 LoRA가, SQL에서는 둘이 비슷합니다.

해결책: 둘을 같이 학습시키고, 매 step마다 더 유리한 쪽에 weight 업데이트를 routing하는 MoLF (Mixture of LoRA and Full) 프레임워크. 결과: 9개 setting에서 더 나은 baseline의 1.5% 이내를 항상 달성. 게다가 학습 후엔 두 expert가 하나의 dense matrix로 합쳐져 추론 비용 0.

배경 지식 — 5가지 개념을 차근차근

① Fine-Tuning이란?

거대 언어 모델은 인터넷의 방대한 데이터로 사전 학습됩니다. 그런데 특정 분야(의료, 법률, 코딩, 회사 내부 지식)에 쓰려면 추가 학습이 필요합니다. 이를 fine-tuning이라 합니다.

② Full Fine-Tuning (FFT) vs LoRA

FFT는 모델 전체 weight(예: 70억 개)를 직접 업데이트합니다. 가장 자유롭지만 메모리·시간 비용이 크죠.

LoRA(Low-Rank Adaptation)는 영리한 아이디어: 모델 weight $W$를 그대로 두고, 두 개의 작은 행렬 $A, B$만 학습합니다. 새 weight는 $W + BA$. 학습 가능한 파라미터를 1%까지 줄일 수 있어요.

FFT는 책 전체를 다시 쓰는 것, LoRA는 포스트잇만 붙이는 것. 포스트잇이 더 효율적이지만, 책 내용을 크게 바꿔야 하는 일에는 한계가 있죠.

③ 왜 둘 다 필요한가? — 세 가지 task regime

이 논문이 밝힌 핵심: task마다 어느 쪽이 좋은지가 다르다.

이는 "글쓰기 도구"의 선택과 비슷합니다. 새 사실을 잔뜩 기록해야 한다면 공책 전체(FFT)가 필요하고, 기존 지식을 보존하면서 메모만 한다면 포스트잇(LoRA)이 적합하고, 단순한 형식 변환이라면 둘 다 잘 됩니다.

④ Mixture-of-Experts (MoE)란?

여러 sub-network(전문가)를 두고 입력에 따라 일부만 활성화하는 구조. 메모리 효율적이지만 forward 시 sparsity—일부 전문가만 데이터를 보기 때문에 활성화 안 된 전문가의 gradient 통계가 부족해집니다.

⑤ AdamW Optimizer

가장 널리 쓰이는 LLM 학습 옵티마이저. 각 파라미터마다 별도의 momentum (m)과 variance (v)를 유지해 step 크기를 자동 조절. MoLF의 핵심도 이 AdamW의 preconditioned step direction을 활용합니다.

핵심 아이디어 — MoLF의 4단계

단계 1
Forward는 dense하게

모든 expert (FFT + LoRA들)가 항상 데이터를 본다. 입력 $x$가 모든 expert를 통과해 합쳐진 출력 $y$ 생성. 전통적 MoE의 gating 없음.

단계 2
모든 expert의 momentum 추적

선택 여부와 무관하게 모든 expert가 매 step AdamW moment를 업데이트. 따라서 dormant expert도 mature, debiased momentum을 갖춤.

단계 3
EPD 점수로 winner 선정

각 expert의 예상 손실 감소량(Expected Preconditioned Descent)을 계산. AdamW의 실제 step direction과 learning rate를 모두 고려한 점수.

단계 4
Top-K 만 실제 weight 업데이트

점수 1등 expert만 weight 변경. 나머지는 weight 그대로—단 momentum은 계속 누적. 다음 step에 다시 후보가 됨.

핵심 통찰: "sparsity를 forward가 아닌 optimizer에"

전통적 MoE는 forward에서 일부만 활성화 → gradient 통계가 단편화. MoLF는 forward는 모두 통과, 단지 weight 업데이트만 sparse. 이 작은 위치 변경이 큰 차이를 만듭니다:

  • 모든 expert가 full-batch gradient를 받음 → cold-start AdamW 문제 없음.
  • FFT와 LoRA가 같은 feature를 두고 oscillate하지 않음(sparse update가 conflict 해결).
  • 학습 후 single dense matrix로 collapse 가능—추론 latency penalty 없음.
회사 회의를 생각해 보세요. 전통적 MoE는 "팀원 중 일부만 회의에 참석"(forward sparse). MoLF는 "모든 팀원이 회의에 참석해 정보 공유하지만, 결정 권한은 한 명에게만"(update sparse). 정보가 끊기지 않고 흐릅니다.

EPD 점수 — 그래디언트가 아니라 "예상 손실 감소"

특히 인상적: FFT의 catastrophic forgetting 회피

Med Gemma-1B에서: FFT는 42.15, LoRA는 46.19. MoLF는 45.40. FFT가 기존 지식을 망가뜨리는 경향(catastrophic forgetting)을 MoLF가 자동으로 피한 것입니다. 사람이 "이 task는 LoRA로 가야지"라고 결정하지 않아도 routing이 알아서 LoRA expert를 선호.

왜 새 expert가 활성화될 때 무너지지 않는가?

기존 AdaLoRA 같은 adaptive 방법의 문제: 새 rank를 promote할 때 AdamW의 momentum과 variance가 0(cold start). 학습 안정성이 흔들립니다.

MoLF는 다릅니다: 모든 expert가 항상 forward에 참여하므로 어떤 step에도 momentum/variance가 fresh. 어느 순간 routing이 새 expert를 선택해도, 이미 warm-up된 상태라서 즉시 효과적인 step.

MoLF-Efficient — 메모리 제약 환경용

FFT expert는 메모리가 크기 때문에 (전체 모델 weight × 2 = m, v) 항상 가능하지 않습니다. 그래서 저자들은 MoLF-Efficient (MoLF-E)를 별도 제안:

  • FFT expert permanently freeze (optimizer state tracking 안 함)
  • 대신 2개 다른 rank의 LoRA expert (예: rank 64 + rank 128) parallel
  • EPD가 매 step 더 유용한 subspace를 선택

결과: AdaLoRA, AdaMix 대비 Fact +11.70~+20.01%p 향상. memory footprint는 LoRA 두 개 수준.

활용 분야

한계와 도전

① 소형 모델 (≤3B)에서만 검증

3B 이하 모델 3개만 테스트. 7B, 13B, 70B의 frontier 모델에서 같은 양상이 유지되는지 불확실.

② Top-K=1 고정

매 step 한 expert만 update. K=2 또는 dense update와의 sweet spot 탐색 부재.

③ FFT expert가 있으면 메모리 큼

MoLF는 FFT의 dense AdamW state (m, v) 추적이 필수. 메모리 제약 환경에서는 MoLF-E 사용 필요.

④ 좁은 task 범위

Fact, Med, SQL 세 task에서만 검증. instruction tuning, RLHF, multimodal fine-tuning 같은 다른 시나리오 결과는 미지수.

실무자/연구자가 가져갈 인사이트

  1. "FFT vs LoRA"는 false dichotomy: task에 따라 답이 다름. 하나만 선택하지 말고 task regime을 먼저 파악.
  2. "Sparsity in optimizer, density in forward": 전통 MoE의 forward sparsity 대신 업데이트 단계만 sparse. AdamW state cold-start 회피, oscillation 방지.
  3. EPD > PFN: routing에 raw gradient 크기 대신 expected loss reduction을 쓰자. AdamW의 preconditioning과 learning rate 둘 다 반영.
  4. MoLF-E를 default로: 메모리 제약 환경이라면 FFT 없이 2개 rank의 LoRA를 parallel로 둬도 AdaLoRA·AdaMix를 크게 능가.
  5. 학습 후 모델은 single dense matrix: 추론 시점에서는 LoRA의 latency penalty조차 없음. 배포 친화적.
  6. Hyperparameter search 부담 감소: 더 이상 rank를 sweep할 필요 없이 MoLF가 자동으로 적합한 expert를 선택. compute budget의 큰 절감.
