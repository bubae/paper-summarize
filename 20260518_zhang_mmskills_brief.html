MMSkills — 시각 AI에게 "텍스트 매뉴얼" 대신 "사진 첨부 매뉴얼"을 주자

데스크탑 GUI와 게임 환경에서 일하는 AI 에이전트에 "언제, 무엇을 보고, 무엇을 하지 말지"를 담은 multimodal skill 패키지를 제공. OSWorld에서 weak 모델 성능 2.4×.

한 줄 요약

"AI 에이전트가 컴퓨터를 다룰 때 '어떤 버튼을 누르는가'만으로는 부족하다. '지금 화면이 그 단계가 맞는지' '어떤 단서를 봐야 하는지' '무엇을 하면 안 되는지'까지 사진과 함께 묶인 매뉴얼이 있어야 한다 — 그게 MMSkills다."

왜 이 문제가 중요한가

AI 에이전트의 실제 실패 모드

2025–26년의 visual agent (Gemini, Claude Computer Use, Qwen3-VL 등)는 desktop, GUI, 게임을 다룰 수 있지만 자주 어리석은 실수를 한다:

📊 잘못된 시트에 차트 생성

"Sheet2에 차트 만들어줘"라고 했는데 현재 활성 시트가 Sheet1이라 거기에 만든다. 결과는 wrong.

⏰ 다이얼로그 안 떴는데 클릭

창이 아직 로딩 중인데 다음 버튼을 누른다. 무반응.

🎮 진행과 완료를 혼동

게임에서 목표를 달성했는데 아직 진행 중인 줄. 같은 액션을 반복.

이런 실수의 공통점: "지금 화면 상태"를 텍스트로만 추론하면 잘못 판단한다. 시각 단서가 결정적이다.

기존 Skill 라이브러리의 한계

기존 agent system (Voyager, ReAct 등)이 사용하는 skill은 텍스트로만 표현된다:

▲ 기존 텍스트 매뉴얼 vs MMSkills: state card와 multi-view evidence가 추가된다.

배경 지식: 알아두면 좋은 6가지 개념

1) Visual Agent (시각 에이전트)

screenshot이나 게임 화면을 직접 보고 키보드/마우스 액션을 수행하는 AI. Gemini 3, Claude Computer Use, Qwen3-VL 등이 대표. 텍스트 base agent보다 applicable scope가 훨씬 넓지만 실수도 잦다.

2) Skill / Procedural Knowledge

재사용 가능한 "이 task를 하는 방법"의 추상화. 인간에게도 운전 면허 매뉴얼, 요리 레시피처럼 반복되는 절차가 있듯이, AI도 task-specific skill을 가진다.

3) OSWorld

desktop GUI 환경에서 AI agent를 평가하는 벤치마크. Chrome, GIMP, LibreOffice (Calc/Impress/Writer), VS Code 등 실제 Ubuntu 데스크탑에서의 task를 측정.

4) VLM (Vision-Language Model)

이미지와 텍스트를 동시에 받는 LLM. GPT-4V, Gemini, Qwen3-VL 등이 대표. 본 논문은 VLM을 base agent로 사용.

5) Branch / Sub-agent 패턴

메인 agent의 reasoning context를 보호하기 위해 temporary sub-agent에 정보 처리를 위임. 처리 결과만 main으로 반환. 본 논문의 핵심 메커니즘 중 하나.

6) State Card

본 논문 고유 개념. "지금 어떤 단계인가"를 시각 단서 + 진행 검증 + when-not-to-use로 명시한 agent-facing description. 인간의 체크리스트와 유사.

MMSkill의 구조

각 MMSkill은 4개 component로 구성: $M = (D, P, S, K)$

📝 D: Descriptor

이 skill이 언제 사용되는지의 한 줄 요약. 이름표 역할.

📋 P: Procedure

재사용 가능한 텍스트 절차. "1. Insert 메뉴를 연다, 2. Chart Wizard 선택..." 같은 step list.

📊 S: State Cards

각 단계의 시각적 상태 설명. when_to_use, when_not_to_use, visible_cues, verification_cue 포함.

🖼 K: Keyframes

각 state에 대한 4종 view: full_frame, focus_crop, before, after.

4종 Multi-View: 왜 이렇게 나누나?

gated selection으로 load — 4종 모두 항상 보여주지 않는다 실측: focus_crop이 3/4 setting에서 가장 자주 selected
▲ 4종 view는 coordinate를 복사하기 위한 게 아니라 평가 evidence다. 각자 다른 진단 목적.
📷 비유: 사진 첨부 요리 레시피를 떠올려보자. 전체 식탁 사진(full_frame)으로 "지금 어떤 단계인지"를 보고, 손 부분 확대(focus_crop)로 "어떻게 칼질하는지"를 보고, 볶기 전(before)/후(after) 사진으로 "다 익었는지"를 판단한다. 4종 view가 각각의 역할.

Skill을 만드는 5단계 파이프라인

handwritten skill이나 raw demonstration replay가 아니라, public non-evaluation trajectory에서 자동 추출한다 (Generator $\mathcal{G}_\mathcal{F}$).

text rationale을 먼저 고정한 뒤 Phase 4에서 image를 grounding한다 → visual grounding이 rationale에 anchor되도록 강제 (image overfitting 방지)
▲ 5-phase pipeline. 핵심 design: image를 보기 전에 text rationale을 먼저 굳힌다.
📚 비유: 요리책 저자가 레시피 글을 먼저 다 써놓고 마지막에 사진작가가 단계별 사진을 찍어 끼워넣는 워크플로우. 사진이 글을 만들지 않고, 글이 사진의 진단 기준을 정한다.

실행 메커니즘: Branch Loading

Skill을 사용하는 방식이 중요하다. 전체 skill package를 main agent에 그대로 inject하면 세 가지 문제가 생긴다.

Direct loading의 3가지 문제

  • ① Context pressure: state card + 4종 keyframe + transition example이 main context를 잡아먹는다.
  • ② Image 경쟁: reference image와 live observation이 시각 attention을 두고 경쟁.
  • ③ Over-anchoring: main agent가 reference 시나리오를 따라하려고 함 (live와 다른데도).
planning signal로만 사용 — action은 live screenshot에서 자기가 결정 → reference image가 live observation을 override하지 못한다
▲ Branch는 보조 agent가 multimodal evidence를 정리해 5-field tuple만 메인에 돌려준다.
🏥 비유: 의사(main agent)가 환자(live observation)를 진료할 때, 매뉴얼을 직접 읽지 않고 비서(branch)가 매뉴얼을 보고 "이 환자에게 적용 가능한가, 다음 단계, 하지 말 것, 확인 기준"만 1쪽으로 정리해 전달한다. 의사는 그 1쪽을 참고하면서도 앞에 있는 환자의 실제 상태를 직접 본다.

가장 흥미로운 발견: 행동 패턴이 바뀐다

MMSkills의 효과는 단순 success rate 향상이 아니다. agent의 행동 양식 자체가 변한다.

▲ 행동 모드의 질적 전환. click-heavy 탐색이 keyboard input + completion judgment로 바뀐다.

이 변화의 함의:

  • Step 수 감소: VAB-Minecraft에서 평균 -7.67 step. shorter task-solving path.
  • Repetitive loop 탈출: exact repeated actions 21.8% → 6.2%.
  • DONE 액션 증가: 완료 판단이 더 명시적으로 이뤄짐.

한계점

① Skill 생성 비용 측정 부재

5-phase Generator는 LLM agent를 반복 호출. per-skill generation cost가 보고되지 않아 scale-up 정량 평가 어렵다.

② Branch loading의 추론 overhead

매 step skill을 invoke할 때 추가 VLM call이 branch에서 발생. step 수는 줄어들지만 총 token cost / latency는 보고되지 않음.

③ Generator quality에 대한 의존

when_to_use / when_not_to_use가 잘못 생성되면 skill 자체가 wrong state에서 invoked. Audit phase가 이를 막는다고 주장하지만 audit 자체도 LLM 기반의 circular validation.

④ Focus crop이 정적

skill package가 focus_crop region을 미리 정의 → 새 UI layout이나 다른 해상도에서는 적용 불가. adaptive crop 메커니즘 부재.

⑤ Safety 분석 없음

잘못 적용된 skill로 destructive action (파일 삭제 등)이 일어날 수 있는 GUI 시나리오에 대한 error recovery 정의가 없음.

활용 분야

▲ MMSkills 패러다임이 적용 가능한 영역들.

한 줄 결론

시각 agent가 잘 하려면 '무엇을 하는가'뿐 아니라 '무엇을 보고 판단하는가'를 알아야 한다.
MMSkills는 그 시각 단서 + 검증 + negative constraint를 하나의 reusable package로 묶은 첫 시도.
Branch loading으로 main context를 보호하면서 동시에 reference 시나리오에 over-anchoring되는 것을 방지한다.

가져갈 인사이트 3가지

  • "Text-only skill은 distractor가 될 수 있다": Gemini 3.1 Pro에서 텍스트만 사용 시 오히려 no-skill보다 낮은 점수. procedural knowledge는 visual evidence와 함께일 때만 안정적 향상을 준다.
  • "When-not-to-use가 가장 중요한 negative constraint": 인간 매뉴얼에서도 자주 누락되는 정보. agent의 exploration을 줄여 step 수와 repeat을 동시에 감소.
  • "Branch loading은 distillation의 새 패턴": 모든 정보를 main context에 inject하는 대신 sub-agent가 정리한 요약본만 전달. context 압박과 attention 경쟁을 동시 해결.
