SABER — AI 에이전트가 "되돌릴 수 없는 행동" 직전에만 멈춰 확인하는 법

전체 행동의 14–18%만 차지하는 'mutating step'(취소, 환불, 삭제)이 실패의 92–96%를 만든다. 그 순간에만 verification + reflection을 거는 것만으로 GPT-5/Claude/Qwen3에서 +5~20pp.

한 줄 요약

"모든 행동에 안전벨트를 매면 차를 못 탄다. 되돌릴 수 없는 행동(취소·삭제·환불)에만 안전장치를 걸어라 — 그게 실패의 거의 전부니까."

왜 이 문제가 중요한가

AI 에이전트가 일상 task에서 자주 실패한다

2026년 현재 GPT-5, Claude Sonnet 4, Qwen3 같은 frontier LLM도 multi-turn tool-using task에서는 깨지기 쉽다. 항공 예약, 환불, 파일 편집, 코드 수정 같은 task에서:

📌 제약을 잘못 해석

"basic economy는 변경 불가" 정책을 무시하고 update_reservation 호출.

📜 오래된 context에 의존

대화 초반의 잘못된 정보를 마지막에 행동으로 옮김.

🔧 부적절한 tool call

cancel_reservation을 잘못된 reservation_id로 호출.

🕰 Lost-in-the-middle

긴 대화 중간의 중요 정보를 모델이 무시.

기존 대응의 패턴: "모든 step을 똑같이 검사"

지금까지의 방법들은 모든 step을 동일한 강도로 검사했다:

① End-to-end prompting 강화

system prompt에 모든 제약을 길게 기록. context 폭증 + lost-in-the-middle 가속.

② Generic process reward

매 step마다 reward 계산. 99% 이상이 무관한 step인데 다 비용 들임.

③ 5-turn self-reflection

모든 답을 5번 revise. compounding error, 모든 step이 동등하게 다뤄짐.

④ Multi-agent debate (MoA)

여러 agent가 모든 step에 의견. cost 폭발 + 진짜 risky step 식별 못함.

가장 충격적인 발견: 모든 행동은 똑같이 위험하지 않다

저자들이 τ-Bench (항공 예약, 소매 환불)의 실패 trajectory를 분석한 결과:

▲ Mutating action 한 번 더 실수하면 91–96% 확률로 실패. 반면 non-mutating은 9–19%에 그친다.

핵심 통계

  • Mutating action 비율: 전체 step의 단 14–18%.
  • Mutating 실수 1회 → 실패 확률: 91–96% (모델·도메인 무관 일관).
  • Non-mutating 실수 1회 → 실패 확률: 9–19% (대부분 통계적으로 무의미).
  • 결론: 소수의 mutating step에 verification을 집중하면 실패의 거의 전부를 잡을 수 있다.
🏥 비유: 수술실의 "Surgical Time-Out" 프로토콜. 환자 입원 → 검사 → 마취 → 절개 → 봉합 → 회복까지 수십 단계가 있지만, 수술팀이 모두 멈춰 큰 소리로 "환자 이름, 수술 부위, 절차"를 확인하는 것은 메스를 들기 직전 단 한 번뿐이다. 매 단계마다 이 의식을 치르면 시간이 너무 걸리고 직원 피로가 누적된다. 돌이킬 수 없는 행동 직전에만 멈추는 것이 핵심.

배경 지식: 알아두면 좋은 6가지 개념

1) LLM Agent

LLM이 단순 텍스트 답변을 넘어 외부 도구(tool)를 호출하며 multi-turn으로 task를 완료하는 시스템. 항공 예약, 코드 작성, 파일 관리 등 실제 행동을 수행.

2) Tool Call

Agent가 외부 함수를 부르는 액션. get_reservation_details(...), cancel_reservation(...)처럼 함수명과 파라미터.

3) τ-Bench

conversational agent를 평가하는 표준 벤치마크 (Yao et al. 2024). 항공 예약 (Airline)과 소매 환불 (Retail) 도메인. simulated user가 agent와 multi-turn 대화.

4) Mutating vs Non-Mutating Action

본 논문의 핵심 구분. Mutating: 환경 상태를 바꾸는 행동 (cancel, update, send, delete). Non-Mutating: 정보만 조회 (get, search, lookup).

5) Lost-in-the-Middle

긴 context에서 LLM이 중간 부분의 정보를 무시하는 현상. 항공 정책 같은 중요 제약이 대화 중간에 있으면 mutating step에서 잊혀짐.

6) Context Poisoning

verification turn을 자주 추가하면 대화 history가 부풀어 진짜 중요한 정보가 묻히는 현상 (Breunig 2025).

SABER의 3가지 안전장치

SABER는 main model (행동 생성)과 auxiliary model (검증/요약/관리) 두 모델을 함께 쓴다.

▲ SABER는 mutating 분류 후 3가지 안전장치를 직렬 적용. 80% 이상의 step은 검증 없이 통과.

① Mutation-Gated User Verification

매 action마다가 아니라 mutating action을 실행하기 직전에만 user confirmation 요청. Verification 빈도가 평균 turn 6번에 1번으로 manageable.

🛒 비유: 온라인 쇼핑몰에서 상품 검색, 카트 담기, 옵션 변경은 그냥 진행되지만, 최종 결제 버튼 직전에는 "정말 결제하시겠습니까?" 확인창이 뜬다. 이게 mutation gate의 사용자 경험.

② Targeted Reflection

긴 대화 도중 mutating action 직전에 시스템 제약 핵심 요약을 <think> block(또는 ReAct format)에 inject. 모델이 직전 직접에 정책을 다시 떠올리게 만듦.

📋 비유: 비행기 조종사가 이륙 직전 체크리스트를 읽는다. 비행 내내 모든 정책을 머릿속에 두려고 하면 인지 부하가 폭증한다. 결정적 순간에만 핵심을 다시 본다.

③ Block-based Context Cleaning

대화 history를 semantic block으로 분할 → 각 블록의 embedding + 요약을 저장 → 현재 user query에 가장 관련 있는 top-N block만 retrieve해 context에 유지. verification turn이 context를 오염시키는 것을 방지.

🗂 비유: 책상에 쓸데없는 종이가 쌓이면 정작 필요한 서류를 못 찾는다. 주기적으로 관련된 폴더만 책상에 두고 나머지는 캐비닛으로 정리.

부산물: τ-Bench Verified 공개

저자들은 τ-Bench를 분석하다가 벤치마크 자체의 결함을 발견했다:

▲ τ-Bench의 ground truth 자체가 정책 위반. Airline 50개 중 31개, Retail 115개 중 53개 지시가 underspecified.

τ-Bench의 attainable performance가 Airline ~70%, Retail ~92%로 인위적 cap되어 있다. 저자들이 모든 task를 수동 검토하여 정정한 τ-Bench Verified를 공개. 새 벤치마크 표준이 될 수 있다.

Qwen3-Thinking-235B를 τ-Bench Verified Airline에서 측정:

핵심 발견: 두 컴포넌트가 거의 가산적이고 시너지. 각 컴포넌트 단독은 +10pp 정도지만 결합 시 +20.7pp.

  • Reflection: model의 인지 부담 줄임 (semantic). 정책을 다시 떠올림.
  • Verification: 사용자 감독 (oversight). 모델 실수를 외부에서 catch.
  • 둘이 다른 차원에서 작동하므로 함께 쓰면 보완적.

한계점

① Online safeguard지 training이 아니다

SABER는 runtime intervention. inference 시 항상 auxiliary model이 함께 돌아야. 향후 loss function으로 decisive deviation 인식을 학습하는 방향이 필요.

② User-in-the-loop에 의존

Mutation-Gated Verification은 user 또는 user simulator의 confirmation이 필요. 실제 production에서 user가 항상 즉시 응답한다는 보장 없음. SWE-Bench 같은 user 없는 환경에서는 reflection만 사용 가능해 효과 절반.

③ Mutating action 정의가 manual

$A^\text{mut}$를 도메인별로 사람이 정의. 새 도메인 (web browsing, embodied robotics)으로 transfer 시 자동 식별 메커니즘 부재.

④ Auxiliary model 비용 측정 불완전

"latency is marginal"이라 명시되지만 wall-clock 비교가 정량화되지 않음. block embedding + summarization + classification은 모두 추가 forward pass.

⑤ Anonymous submission

ICLR 2026 double-blind review 중. peer review 완료 전이므로 추가 검증이 필요.

활용 분야

▲ 되돌릴 수 없는 행동이 있는 모든 도메인에 적용 가능.

한 줄 결론

AI agent의 실패는 모든 step에 골고루 일어나지 않는다.
전체 14–18%의 mutating step에 91–96%의 실패가 집중되어 있다.
그 좁은 지점에만 verification + reflection을 거는 것이 가장 효율적인 안전장치다.

가져갈 인사이트 3가지

  • "Verification은 희소 자원": 사용자의 주의력과 시간은 한정. 모든 step에 verification을 거는 것이 아니라 실패를 flip할 가장 가능성 높은 지점에 집중. cost와 quality 동시 향상.
  • "Reliability research의 새 방향": agent failure를 decision step level에서 정량화하면 generic safeguard보다 targeted intervention이 훨씬 효과적. 이 발견은 다른 도메인(코딩, 의료)에도 transferable.
  • "Benchmark 자체의 결함을 추적해야 진짜 progress가 보인다": τ-Bench Verified가 보인 것 — annotation noise가 strong model 차이를 가리고 method benefit을 숨긴다. cleaner benchmark가 progress 측정의 prerequisite.
